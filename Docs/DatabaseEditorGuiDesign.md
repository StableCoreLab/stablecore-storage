# 数据库编辑工具 GUI 设计文档

本文定义一个图形界面的数据库编辑工具，用于在 SCStorage 之上定义、维护并检查算量软件所需的原始事实数据。

状态：

- 当前为设计文档。
- 本文不要求与实现一一等长，但要求为后续实现提供明确方向。

设计日期：

- 2026-03-25

## 1. 目标

工具目标不是做通用数据库管理器，而是做“面向 Storage 数据模型的研发与验证工具”。它应帮助开发、规则和诊断人员完成以下工作：

-- 建立原始数据库。
-- 定义表与列。
-- 编辑事实记录（列值）。
-- 查看关系与计算列结果。
- 在启动、恢复、迁移异常时提供诊断入口。

## 2. 目标用户

### 2.1 数据定义用户

关注表结构、列元数据、默认值、可编辑性和关系定义。

### 2.2 规则验证用户

关注计算列、规则结果、依赖关系和局部重算是否符合预期。

### 2.3 诊断用户

关注数据库启动、迁移、恢复、健康报告和问题定位效率。

## 3. 产品目标

该工具应实现以下产品目标：

- 降低 Storage 数据模型的验证成本。
- 让事实层与计算层的差异在界面上清晰可见。
- 让研发人员能快速构造测试数据库与业务样例。

## 4. 工具定位

工具定位为“研发工具 + 接入验证工具”，不直接等同于最终用户产品界面。因此它更强调：

- 结构清晰。
- 错误可见。
- 反馈直接。

而不是先追求最终产品级视觉包装。

## 5. 技术选型

### 5.1 推荐 GUI 技术栈

当前推荐使用 Qt Widgets：

- 与现有工程结构兼容。
- 对表格、树、对话框与停靠面板支持成熟。
- 适合快速构建研发工具。

### 5.2 应用形态

推荐形态为桌面单体应用：

- 一次打开一个数据库窗口。
- 重点优化本地研发调试场景。

## 6. 核心 UX 原则

建议遵循以下原则：

- 事实数据与计算结果在视觉上要明确区分。
- 所有危险操作都应有确认和明确反馈。
- 错误信息优先讲清“为什么失败”，而不是只给错误码。
- 用户切换表、切换记录时，检查器与摘要面板要同步更新。

## 7. 主窗口信息架构

推荐主窗口包含：

- 顶部菜单与工具栏。
- 左侧表列表或数据库浏览区。
- 中央数据网格。
- 右侧检查器与计算列面板。
- 底部诊断与状态栏。

## 8. 主要功能区域

### 8.1 文件区域

提供：

- 新建数据库。
- 打开数据库。
- 刷新当前视图。

### 8.2 数据库浏览树

展示数据库中的表，并支持：

- 切换当前表。
- 显示表的基础摘要。

### 8.3 Schema 编辑器

支持：

- 新增列。
- 查看列类型、显示名、是否可空、是否可编辑、是否索引、是否为关系列等信息。

### 8.4 数据网格

支持：

- 浏览记录。
- 编辑事实列。
- 按列排序。
- 文本过滤。

### 8.5 检查器面板

显示：

- 当前表 Schema 详情。
- 当前记录的列快照。

### 8.6 计算列面板

显示当前表挂载的会话级计算列，并支持：

- 新增。
- 编辑。
- 删除。

### 8.7 关系编辑器

对关系列提供专用选择器，避免直接让用户输入 recordId。

### 8.8 诊断视图

显示：

- 启动健康报告。
- 迁移与恢复摘要。
- 运行时错误与状态信息。

## 9. 核心用户流程

### 9.1 创建新的原始数据库

流程建议：

1. 新建数据库文件。
2. 创建表。
3. 定义列。
4. 新增记录并录入基础事实。

### 9.2 定义 Beam 原始数据

典型示例：

-- 创建 Beam 表。
-- 增加 Id、Code、Length、Width、FloorRef 等列。
- 录入若干样例记录。

### 9.3 预览计算量结果

流程建议：

1. 创建或编辑会话级计算列。
2. 在数据网格中查看计算列结果。
3. 结合检查器验证依赖关系。

### 9.4 诊断数据库启动问题

流程建议：

1. 打开数据库。
2. 查看底部诊断面板。
3. 根据错误与健康摘要定位问题。

## 10. 与存储层的交互模型

### 10.1 数据库会话

GUI 不直接分散调用底层接口，而是通过会话对象统一协调状态。

### 10.2 编辑模型

界面层负责收集输入，会话层负责把输入转换为 Storage 操作并返回错误信息。

### 10.3 变更跟踪

会话层应把当前表、当前记录、计算列列表和诊断摘要整理成可被 GUI 消费的状态。

## 11. ViewModel 结构建议

### 11.1 DatabaseSessionViewModel

负责数据库打开、当前表切换、摘要信息和全局状态。

### 11.2 SchemaEditorViewModel

负责列定义展示与新增列操作。

### 11.3 DataGridViewModel

负责网格数据、排序、过滤、单元格编辑与关系列写回。

### 11.4 DiagnosticsViewModel

负责健康报告、错误摘要和启动诊断信息。

## 12. 计算列处理方式

当前建议先支持会话级计算列：

- 不写入数据库。
- 用于研发验证。
- 由表视图统一管理展示结果。

## 13. 校验规则

### 13.1 Schema 校验

- 列名不能为空。
- 同表内列名不能重复。
-- 关系列需要合法目标表。

### 13.2 记录校验

- 输入值需满足列类型。
- 对不可编辑列禁止在网格中直接修改。

### 13.3 删除校验

- 删除记录前应有确认。
- 删除计算列前应有确认。

## 14. 诊断与日志

建议至少提供：

- 启动健康摘要。
- 当前表状态摘要。
- 最近一次失败操作的错误信息。

## 15. 持久化范围

当前建议：

- 事实数据、Schema、Journal 等由数据库持久化。
- 会话级计算列不持久化。
- GUI 的临时筛选与当前选中状态也不持久化。

## 16. 推荐界面集合

最小界面集合可以包括：

- 主窗口。
- 新增列对话框。
- 关系选择对话框。
- 计算列编辑对话框。

## 17. 安全与约束

当前阶段仍需注意：

- 工具不应默认承担生产环境数据维护职责。
- 需避免误删、误写和错误关系输入。
- 对危险操作应保留明确确认和错误提示。

## 18. 实施分期

### Phase 1

基础数据库浏览、表切换、网格展示与新增记录。

### Phase 2

列新增、关系编辑、过滤排序与检查器。

### Phase 3

会话级计算列支持。

### Phase 4

更完整的诊断、导入导出与产品化增强。

## 19. 待确认设计决策

仍需根据后续研发情况继续确认：

- 是否支持持久化计算列。
- 是否支持批量 Schema 变更。
- 是否把诊断视图扩展为完整工作台。

## 20. 当前推荐决策

当前推荐先坚持：

- 桌面单体应用。
- Qt Widgets。
- 会话级计算列优先。
- 数据与诊断分区清晰。

## 21. 设计结论

数据库编辑器应首先成为 Storage 仓库的“验证放大器”：

- 帮助研发团队快速验证事实模型。
- 帮助规则与计算团队观察结果。
- 帮助排查启动、恢复和迁移问题。

在这个前提下，后续再逐步增强其工程完整性与可用性。
