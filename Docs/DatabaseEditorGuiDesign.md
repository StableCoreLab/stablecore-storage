# 数据库编辑工具 GUI 设计文档

本文档定义一个图形界面的数据库编辑工具，用于在 `stablecore_storage` 之上定义和维护算量软件所需的原始事实数据。

状态：

- 仅设计
- 本文档不包含实现代码

设计日期：

- `2026-03-25`

## 1. 目标

这个编辑工具要解决的核心问题是：

- 让产品人员、业务人员、实施人员可以直接定义原始数据，而不需要写代码

主要编辑对象：

- 创建和打开 SQLite 存储数据库
- 定义表
- 定义字段
- 编辑事实记录
- 编辑关系字段
- 查看计算列
- 查看与 `ChangeSet` 相关的诊断信息

这个工具不打算在 `V1` 做成：

- 通用 SQL 浏览器
- 底层 SQLite 原始表编辑器
- 完整产品 UI
- 全功能公式设计器

## 2. 目标用户

### 2.1 数据定义用户

典型用户：

- 产品设计人员
- 业务工程师
- 实施工程师

主要任务：

- 定义 `Floor`、`Beam`、`Wall`、`Opening`、`Material` 等原始表
- 增加尺寸、分类、标签、来源值、引用关系等字段
- 录入基础样例数据

### 2.2 规则验证用户

典型用户：

- 开发人员
- 规则工程师

主要任务：

- 同时查看事实列和计算列
- 验证关系连线是否正确
- 验证原始数据是否足以支撑后续算量逻辑

### 2.3 诊断用户

典型用户：

- 开发人员
- 支持工程师

主要任务：

- 查看启动诊断
- 查看迁移状态
- 查看存储健康摘要

## 3. 产品目标

`V1` 的目标是：

- 做成一个可用的桌面 GUI 编辑器，直接操作 SQLite 存储数据库
- 优先解决原始数据定义问题，其次再提升业务易用性
- 保证行为明确、编辑过程可控、事务语义清晰

`V1` 非目标：

- 多人协同编辑
- 面向非开发人员的复杂公式平台
- 拖拽式结构建模
- 插件体系
- 报表设计器

## 4. 工具定位

GUI 编辑器位于当前存储运行时之上：

```text
数据库编辑工具 GUI
  -> 编辑器 ViewModel 层
    -> stablecore_storage 公共 API
      -> InMemory / SQLite 后端
```

编辑器必须直接使用当前公共运行时，而不是另外维护一套平行模型：

- `IDatabase`
- `ITable`
- `ISchema`
- `IRecord`
- `IComputedTableView`
- 诊断和迁移辅助接口

这样可以保证编辑器验证的是“真实产品存储语义”，而不是“假语义”。

## 5. 技术选型

### 5.1 推荐 GUI 技术栈

推荐：

- `Qt Widgets`

原因：

- Windows 桌面工具成熟
- 表格、树、属性面板、对话框等模式易于实现
- Unicode 支持成熟
- 适合做 schema 树、记录网格、属性检查器、关系选择弹窗
- 适合长期维护的内部工具

`V1` 不推荐：

- `Dear ImGui`
  原因：
  更适合调试工具，不适合长流程数据录入
- `WPF`
  原因：
  当前仓库是 C++ 内核，额外引入 C#/CLI 交互层复杂度不划算
- 浏览器 / Electron
  原因：
  打包、运行时、架构开销过大，不适合第一版内部工具

### 5.2 应用形态

推荐：

- 单进程桌面应用
- 直接打开 / 创建 SQLite 文件
- `V1` 不做前后端拆分

## 6. 核心 UX 原则

- schema 编辑和数据编辑必须明确区分
- 关系字段编辑必须显式，不允许默认只暴露原始整数 id
- 每次写入都应映射到清晰的存储事务
- 计算列必须在视觉上和事实列区分
- 破坏性操作必须要求确认
- 诊断信息要可见，但不能干扰主要录入流程

## 7. 主窗口信息架构

推荐布局：

```text
+--------------------------------------------------------------+
| 菜单 / 工具栏                                                |
+----------------------+----------------------+-----------------+
| 数据库浏览树         | 数据网格             | 检查器面板      |
| - Tables             | - 事实列             | - schema 详情   |
| - Views              | - 计算列             | - record 详情   |
| - Diagnostics        | - 过滤               | - 关系信息      |
+----------------------+----------------------+-----------------+
| 状态栏：文件 / 版本 / 编辑状态 / 诊断数量                    |
+--------------------------------------------------------------+
```

## 8. 主要功能区域

### 8.1 文件区域

功能：

- 新建数据库
- 打开数据库
- 最近使用数据库
- “保存”由底层存储提交隐式完成
- 后续版本可增加 JSON / CSV 导出

打开时应做的事：

- 调用存储层打开数据库
- 显示迁移结果
- 显示启动诊断
- 如存在 integrity check 告警，明确提示

### 8.2 数据库浏览树

树节点建议：

- database
- tables
- 每张表的数据视图
- 每张表的 schema 视图
- diagnostics
- startup history

每张表的操作：

- 打开数据网格
- 打开 schema 编辑器
- 后续版本可增加复制记录

### 8.3 Schema 编辑器

主要职责：

- 创建表
- 添加事实列
- 添加关系列
- 标记索引列
- 标记 participates-in-calc 列

每个 schema 字段应显示：

- `name`
- `displayName`
- `valueKind`
- `columnKind`
- `nullable`
- `editable`
- `userDefined`
- `indexed`
- `participatesInCalc`
- `unit`
- `referenceTable`
- `defaultValue`

校验要求：

- 关系列必须有 `referenceTable`
- 事实列不能误带 `referenceTable`
- 默认值必须和类型一致
- 非空列必须有非空默认值

`V1` 范围：

- 只支持加列
- 不支持删列
- 不支持重命名列

原因：

- 当前存储核心更适合 append-only schema 变更

### 8.4 数据网格

主要职责：

- 浏览记录
- 新增记录
- 删除记录
- 编辑事实单元格
- 通过人类可读选择器编辑关系字段
- 展示计算列

列展示建议：

- 事实列：普通背景
- 关系列：带关系标记或链接图标
- 计算列：只读样式，并分组显示

编辑模式建议：

- 行级即时编辑
- 提供显式 `Apply` / `Commit`
- 支持一个事务内连续编辑多个单元格

推荐工具栏操作：

- 添加行
- 删除行
- 提交修改
- 回滚修改
- 刷新
- 过滤

并要求：

- 内联显示校验错误
- 当前记录 id 始终可见

### 8.5 检查器面板

用于显示当前选中记录的详情：

- record id
- deleted 状态
- last modified version
- 字段值列表
- 关系目标预览
- 未来版本可增加反查引用列表

### 8.6 计算列面板

这个面板是必须的，因为算量软件开发需要同时看原始值和派生值。

`V1` 能力：

- 为当前表创建一个 computed table view
- 添加计算列定义
- 支持：
  - expression 列
  - `ruleId` 列
  - aggregate 列
- 在数据网格中实时预览结果

重要范围决定：

- `V1` 不要求编辑器自身持久化计算列定义，除非产品明确需要

当前推荐策略：

- 持久化原始数据
- 计算列定义在 `V1` 先做会话级
- 后续再支持 sidecar 配置或产品级布局保存

### 8.7 关系编辑器

关系字段默认不能只暴露原始整数 id。

推荐行为：

- 点击关系单元格时打开关系选择对话框
- 对话框列出目标表候选记录
- 候选项显示：
  - record id
  - `Name` / `Title` / `Code` 等人类可读字段

附加能力：

- 若字段允许为空，可清空关系
- 支持跳转到目标记录

### 8.8 诊断视图

显示内容：

- 存储后端类型
- 当前版本
- schema version
- startup diagnostics 列表
- dirty startup 警告
- migration 信息
- 健康摘要

这个视图应直接使用：

- `BuildStorageHealthReport(...)`
- 数据库诊断提供者输出

## 9. 核心用户流程

### 9.1 创建新的原始数据库

1. 用户点击 `New Database`
2. 选择文件路径
3. 工具创建 SQLite 数据库
4. 打开空白数据库浏览树
5. 用户创建 `Floor`、`Beam`、`Wall` 等表
6. 用户定义字段

### 9.2 定义 Beam 原始数据

1. 用户打开 `Beam` schema 编辑器
2. 添加 `Length`、`Width`、`Height`
3. 添加指向 `Floor` 的 `FloorRef`
4. 保存 schema 变更
5. 打开 `Beam` 数据网格
6. 插入记录
7. 通过关系选择器指定楼层

### 9.3 预览计算量结果

1. 用户打开 `Beam` 数据视图
2. 添加临时计算列 `Volume = Length * Width * Height`
3. 网格展示计算结果
4. 用户确认原始输入是否满足下游算量需求

### 9.4 诊断数据库启动问题

1. 用户打开数据库
2. 工具发现 startup warning
3. 诊断面板显示：
   - 非正常关闭
   - 执行过的迁移
   - integrity check 结果

## 10. 与存储层的交互模型

### 10.1 数据库会话

一个已打开 SQLite 文件对应一个运行时 `IDatabase`。

GUI 每打开一个数据库窗口，就持有一个数据库运行时实例。

### 10.2 编辑模型

`V1` 采用“单次编辑即单次提交”的事务策略。

含义：

- 用户完成一次编辑动作，就形成一次独立事务
- 该事务提交后，立即产生一条可撤消日志
- 不采用“先开启长事务，再由用户统一提交一批修改”的交互模式

典型流程：

1. 用户编辑一个单元格、修改一个关系、添加一条记录或执行一次删除
2. 工具立即调用 `BeginEdit(...)`
3. 工具执行该次修改
4. 工具立即调用 `Commit(...)`
5. 系统生成一条对应本次操作的 `ChangeSet`
6. 系统形成一条对应本次操作的 Undo 日志

这意味着：

- 每次用户完成一个明确动作，就能立即撤消
- Undo/Redo 粒度与用户操作粒度一致
- GUI 不需要暴露长事务管理界面

适用场景：

- 原始数据录入
- 逐条修正字段
- 关系选择与切换
- 小粒度结构维护

不适用场景：

- 希望把大量多步修改强制合并成一次 Undo 的复杂编辑流程

这类需求可以在后续版本通过“批量编辑模式”单独设计，不作为 `V1` 默认行为。

### 10.3 变更跟踪

GUI 应通过 `IDatabaseObserver` 监听变更。

用途：

- 刷新可见记录
- 失效计算列缓存
- 更新状态栏版本号
- 向诊断控制台追加日志

## 11. ViewModel 结构建议

推荐分层：

```text
EditorApp
  -> DatabaseSessionViewModel
     -> ExplorerViewModel
     -> SchemaEditorViewModel
     -> DataGridViewModel
     -> DiagnosticsViewModel
```

### 11.1 DatabaseSessionViewModel

职责：

- 持有 `DbPtr`
- 创建 / 打开数据库
- 提供表列表
- 维护当前版本号
- 转发变更通知

### 11.2 SchemaEditorViewModel

职责：

- 加载 `ISchema`
- 暴露可编辑字段定义
- 做输入校验
- 提交 schema 新增

### 11.3 DataGridViewModel

职责：

- 暴露表记录
- 暴露事实列
- 通过 `IComputedTableView` 暴露计算列
- 把界面修改映射为存储写操作

### 11.4 DiagnosticsViewModel

职责：

- 生成健康报告
- 暴露 startup diagnostics
- 暴露变更日志流

## 12. 计算列处理方式

GUI 应以 `IComputedTableView` 作为事实列和计算列统一展示的主抽象。

`V1` 推荐支持：

- 增加会话级计算列
- expression 文本输入
- `ruleId` 文本输入
- aggregate relation 与 aggregate field 选择
- 在网格中即时预览

`V1` 限制建议：

- 不做计算列布局持久化
- 不做列顺序持久化
- 第一版不做表达式自动补全

## 13. 校验规则

### 13.1 Schema 校验

GUI 必须在提交前镜像存储层校验：

- 重复列名拒绝
- 默认值类型错误拒绝
- 非法关系列定义拒绝

### 13.2 记录校验

GUI 必须校验：

- 值类型转换
- 非空字段
- 关系目标存在性
- 引用已删除目标时报错

### 13.3 删除校验

GUI 必须明确显示删除约束失败：

- 删除被引用目标时显示阻塞提示
- 后续版本尽量提示来源表和字段

## 14. 诊断与日志

GUI 应内置一个轻量事件控制台，显示：

- 事务提交名称
- 回滚事件
- undo / redo 事件
- 启动诊断
- migration 动作

日志来源：

- `ChangeSet`
- startup health report
- 显式 GUI 操作

## 15. 持久化范围

`V1` 持久化内容：

- SQLite 数据库文件
- 原始 schema
- 原始记录
- 运行时已写入的诊断元数据

`V1` 默认不持久化的编辑器工件：

- 列宽
- 网格布局
- 临时计算列
- 过滤器

如果后续要持久化，建议放在：

- 数据库旁边的 sidecar 配置文件
- 或编辑器专属元数据命名空间，不能污染事实数据

## 16. 推荐界面集合

`V1` 推荐界面：

- Welcome / Open Database
- 主编辑窗口
- New Table 对话框
- Add Column 对话框
- Relation Picker 对话框
- Computed Column 对话框
- Diagnostics 面板

## 17. 安全与约束

- `V1` 不开放原始 SQL 输入
- 不支持任意脚本执行
- 所有写入必须走存储 API
- 破坏性操作必须要求确认
- schema 变更必须提示这是持久化结构修改

## 19. 待确认设计决策

开始编码前需要确认：

1. `V1` 计算列定义是否只做会话级，不持久化？
2. 编辑器是“一库一窗口”，还是“多库标签页”？
3. 编辑工具是否采用“单次编辑即单次提交，并形成一条撤消日志”的模式？
4. 关系选择器默认显示哪些字段作为标签？
5. GUI 技术栈是否确定为 `Qt Widgets`？

## 20. 当前推荐决策

当前推荐方案：

1. 使用 `Qt Widgets`
2. `V1` 一库一窗口
3. 采用“单次编辑即单次提交”，每次编辑形成一条 Undo 日志
4. `V1` 计算列定义默认只做会话级
5. 关系选择器默认显示字段优先级：
   - `Name`
   - `Title`
   - `Code`
   - 最后退化为 `displayName`

## 21. 设计结论

这个工具应当是一个面向算量软件原始数据定义的桌面 GUI 编辑器。

`V1` 最重要的几个特征是：

- schema 编辑器
- 原始数据网格
- 关系感知编辑
- 事实列与计算列统一视图
- 与真实存储一致的事务语义
- 与用户单次操作一致的 Undo/Redo 粒度
- 启动诊断可见
