# StableCore Storage V1 基线决策

本文用于固化 V1 阶段最容易反复修改的接口边界与核心语义。目标不是覆盖全部设计，而是尽快统一后续实现的判断标准。

## 1. Schema 基线

### 1.1 设计结论

V1 采用受控动态 Schema：

-- 表有正式 Schema。
-- 列必须注册后才能被可靠访问。
-- 用户可新增事实列，但仍纳入 Schema 管理。

### 1.2 列分类

建议区分：

- 事实列：直接进入存储层。
- 计算列：作为视图层结果，不直接等同于事实存储。

### 1.3 Column 元信息

列元信息至少包括：

- name
- displayName
- valueKind
- columnKind
- nullable
- editable
- indexed
- referenceTable

### 1.4 V1 约束

- 不做完全自由的无 Schema 键值存储。
- 不把计算结果直接混入事实层核心模型。

### 1.5 推荐接口

V1 通过 Schema、ColumnDef 等结构对外表达列定义，不鼓励业务层直接绕过定义写入未知列。

### 1.6 原因

该方案能同时兼顾：

- 研发期可扩展性。
- 运行期数据一致性。
- 后续持久化与工具化能力。

## 2. SCValue 基线

### 2.1 设计结论

SCValue 是 V1 统一值容器，用于承载事实列与部分关系列值。

### 2.2 推荐值对象

V1 建议至少支持：

- Null
- Int64
- Double
- Bool
- String
- RecordId
- Enum

### 2.3 类型规则

- 不用字符串伪装布尔、关系或枚举。
- 类型不匹配要返回明确错误。

### 2.4 字符串规范

- 对外同时提供“直接访问”和“复制访问”两类语义。
- 文档需明确所有权和生命周期。

### 2.5 推荐接口

围绕 GetKind、AsXxx、FromXxx 建立统一模式。

### 2.6 原因

该方案能降低跨 DLL、跨层调用时的类型歧义与所有权问题。

## 3. Record Delete 基线

### 3.1 设计结论

Record 删除后不应立即变成野指针，而应进入可识别的删除状态。

### 3.2 状态模型

建议至少区分：

- Alive
- Deleted

### 3.3 删除规则

- 删除后 recordId 不复用。
- 已删除记录仍可通过旧句柄识别到删除状态。

### 3.4 Undo/Redo 规则

- Undo 恢复记录时保留原 recordId。
- Redo 再删除时继续沿用同一逻辑身份。

### 3.5 推荐接口

Record 需暴露 GetId、IsDeleted、GetLastModifiedVersion 等能力。

### 3.6 V1 删除策略

V1 采用保守删除模型，优先保证一致性和可恢复性，而不是追求最激进的回收策略。

### 3.7 原因

这样更适合支撑 UI 选中状态、关系引用和缓存失效。

## 4. ChangeSet 基线

### 4.1 设计结论

ChangeSet 以事务为单位表达一次提交后的变化集合。

### 4.2 事务级别语义

- 一个用户动作通常对应一个事务。
- 一个事务提交后生成一个 ChangeSet。

### 4.3 变更来源

建议区分：

- UserEdit
- Undo
- Redo
- Import
- RuleWriteback

### 4.4 变更类型

建议至少区分：

- FieldUpdated
- RecordCreated
- RecordDeleted
- RelationUpdated

### 4.5 推荐结构

ChangeSet 应至少包含：

- version
- source
- changes

### 4.6 聚合规则

同一事务内允许对重复写入进行合理聚合，但不能破坏 Undo/Redo 正确性。

### 4.7 Observer 语义

观察者应在事务提交完成后接收 ChangeSet，而不是在每次列写入时立即收到零散事件。

### 4.8 原因

这样更适合 UI 局部刷新、缓存失效与增量重算。

## 5. 推荐的下一步

在 ChangeSet 基线收紧之后，应优先补测试和持久化对齐，而不是继续扩展事件类型数量。

## 5. Error Code 基线

### 5.1 设计结论

错误码应作为跨层、跨模块统一返回值使用。

### 5.2 推荐原则

- 同一类问题返回同一类错误码。
- 不通过模糊布尔值隐藏失败原因。

### 5.3 推荐错误码集合

V1 至少要覆盖：

- 成功
- 指针为空
- 参数非法
- 类型不匹配
- 值为空
- 未找到
- 已删除
- 未实现

### 5.4 语义说明

所有公共接口都应遵守同一套错误语义，避免后端间行为差异过大。

### 5.5 原因

统一错误码是测试、诊断和跨层调用可控的前提。

## 6. EditSession 基线

### 6.1 核心问题

写入是否必须显式带 edit 对象，以及事务边界如何管理，是 V1 需要尽早统一的问题。

### 6.2 设计结论

V1 采用数据库拥有单个活动事务的模型。

### 6.3 推荐约束

- 所有写操作必须处于活动事务中。
- Database 负责 BeginEdit、Commit、Rollback。

### 6.4 推荐状态

EditSession 至少应表达：

- Active
- Committed
- RolledBack

### 6.5 推荐接口

EditSession 暴露名称、打开版本和状态即可，不必承载过多业务逻辑。

### 6.6 为什么不让写接口显式带 edit

因为 V1 更适合用数据库统一管理活动事务，减少调用层复杂度和误用概率。

## 7. Table Query 基线

### 7.1 设计结论

V1 提供最小查询能力，但不提前设计成完整查询语言。

### 7.2 V1 最小查询能力

- 枚举记录。
- 按单列等值条件查找。
- 通过 recordId 直接读取记录。

### 7.3 推荐轻量接口

使用 RecordCursor、QueryCondition 等轻量对象即可。

### 7.4 V1 查询约束

- 不追求复杂组合条件。
- 不要求分页、排序和分析型查询。

### 7.5 后续可扩展方向

- 更复杂过滤。
- 排序分页。
- 关系反查。

### 7.6 原因

先把最小主路径做稳，再扩展查询表达能力更合适。

## 8. JournalEntry 基线

### 8.1 设计结论

JournalEntry 用于记录事务中的基础操作，以支撑 Undo/Redo 与持久化回放。

### 8.2 操作类型

建议至少包括：

- SetValue
- CreateRecord
- DeleteRecord
- SetRelation

### 8.3 推荐结构

每条 JournalEntry 应能定位表、记录、列以及前后值或必要上下文。

### 8.4 语义说明

Journal 面向回放正确性，ChangeSet 面向通知与摘要；两者职责不同但可以相互转换。

### 8.5 V1 聚合策略

允许在单事务内做合理聚合，但不能牺牲回放完整性。

### 8.6 推荐事务结构

事务是 JournalEntry 的容器，并带有 actionName、openedVersion 等上下文信息。

### 8.7 Journal 与 ChangeSet 的转换原则

转换时应优先保留“事务级结果”，而不是机械逐条透传内部操作。

### 8.8 原因

这样既能支撑回放，又能给 UI 和日志提供可读摘要。

## 9. 下一步

Journal 与 ChangeSet 基线确定后，应继续补 SQLite 持久化对齐与异常恢复测试。

## 10. Computed Field 基线

### 10.1 设计结论

计算列属于面向产品接入的重要能力，但在 V1 只做最小可用范围。

### 10.2 计算列的表达目标

V1 重点服务：

- 研发验证。
- 简单表达式。
- 规则型扩展。
- 聚合结果展示。

### 10.3 计算列分类

- Expression
- Rule
- Aggregate

### 10.4 推荐列定义

计算列定义至少应包括：

- name
- displayName
- valueKind
- kind
- expression 或 ruleId 或 aggregate 配置
- dependencies

### 10.5 依赖声明

依赖应显式记录，便于缓存失效和增量重算。

### 10.6 依赖示例

例如面积可依赖 Length 与 Width，楼层汇总可依赖关系列与目标表列。

### 10.7 推荐统一上下文

计算运行时通过统一上下文读取列值、关系值和相关记录。

### 10.8 推荐求值接口

由求值器、上下文和规则注册表共同完成计算。

### 10.9 V1 表达式能力边界

仅支持最小表达式能力，不引入完整脚本语言。

### 10.10 规则型列建议

通过 ruleId 注册机制与存储层解耦。

### 10.11 聚合型列建议

通过关系与聚合种类描述最常见统计场景。

### 10.12 缓存策略

以依赖失效为主，不追求过度复杂的缓存系统。

### 10.13 与数据表编辑工具的关系

数据库编辑器优先支持会话级计算列，作为研发验证工具。

### 10.14 V1 推荐结论

先把最小表达式、规则注册和缓存失效做稳，再考虑更丰富的产品级能力。

## 11. Floor Relation 基线

### 11.1 设计结论

楼层关系应优先建模为明确关系，而不是仅靠字符串列表达。

### 11.2 推荐表结构

建议用独立 Floor 表与引用列表达归属关系。

### 11.3 为什么不直接用 FloorName

因为字符串无法稳定承载引用、一致性检查和后续反查。

### 11.4 推荐读取方式

通过 RecordId 引用与查询接口读取楼层关系。

### 11.5 推荐楼层列

可包括楼层名称、标高、排序号等基础属性。

### 11.6 单归属与多归属

V1 先以单归属为主，多归属作为后续业务扩展。

### 11.7 跨层构件建议

跨层构件的业务解释可放在上层领域模型中，不强行在 V1 底层一次性做完。

### 11.8 业务层表达建议

业务层可在 Storage 之上继续封装 Beam、Wall、Floor 等领域对象。

### 11.9 ChangeSet 语义

楼层引用变化应作为关系变更进入 ChangeSet。

### 11.10 V1 推荐结论

关系优先于字符串列，是更适合后续产品演化的基线。

## 12. SQLite Engine 基线

### 12.1 设计结论

V1 直接基于 sqlite3 构建轻量持久化引擎。

### 12.2 为什么直接用 sqlite3

- 依赖少。
- 可控性高。
- 适合当前仓库体量。

### 12.3 为什么不引入 ORM

因为当前更需要的是精确控制存储布局、事务、Journal 和恢复，而不是对象映射便利性。

### 12.4 适配层原则

SQLite 层负责持久化，不负责上层业务规则。

## 13. SQLite Persistence Boundary

### 13.1 分层结论

Storage 核心负责抽象语义，SQLite 层负责具体持久化实现。

### 13.2 SQLite 持久化层职责

- 数据落盘。
- 元数据加载。
- 查询与索引。
- Journal 持久化与恢复支撑。

### 13.3 SQLite 持久化层不负责

- 业务领域规则。
- 高层对象模型。
- 最终产品 UI 逻辑。

### 13.4 推荐适配对象

建议围绕 Database、Table、Record、Schema 等核心抽象做适配。

### 13.5 推荐接口风格

保持与内存实现一致的公共接口，不向上暴露 SQLite 特有细节。

### 13.6 推荐数据映射策略

优先保证一致性与可维护性，再逐步优化存储布局与性能。

### 13.7 事务边界建议

SQLite 层事务应与上层 EditSession 语义保持一致。

### 13.8 Journal 持久化建议

优先支撑正确恢复与 Undo/Redo，再考虑更激进的压缩和优化。

### 13.9 错误映射建议

SQLite 错误要转换到统一 ErrorCode，不让上层直接依赖底层错误码。

### 13.10 V1 推荐结论

SQLite 持久化边界应保持清晰、轻量、可测试，避免提早与业务层耦合。
