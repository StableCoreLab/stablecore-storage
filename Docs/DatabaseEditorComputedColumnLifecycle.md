# 数据库编辑器会话级计算列生命周期设计

本文定义 Tools/DatabaseEditor 中“会话级计算列”的最小可用生命周期规则，用于支撑新增、编辑、删除功能，并保持实现与交互一致。

设计日期：

- 2026-03-25

## 1. 范围

本文只讨论数据库编辑器中的“会话级计算列”：

- 仅在当前编辑器进程内有效。
- 不持久化到 SQLite 数据库。
- 通过 ComputedTableView 挂载到当前表视图。

本文不讨论：

- 事实 Schema 列。
- 持久化计算列。
- 通用规则注册系统的完整设计。

## 2. 基本结论

当前版本采用以下规则：

1. 计算列以“当前表内唯一的 name”作为标识。
2. 编辑计算列按“整条定义替换”处理，不做局部 patch。
3. 删除或编辑后统一重建当前表的 ComputedTableView。
4. 计算列删改不进入数据库 Undo/Redo。
5. 计算列的会话状态由 SCDatabaseSession 的内存结构负责维护。

## 3. 标识规则

### 3.1 唯一性

会话级计算列的最小唯一键可视为：

```text
tableName + computedColumn.name
```

约束如下：

- 同一张表内，计算列 name 必须唯一。
- 不同表可以使用相同的计算列 name。

采用该方案的原因：

- 当前实现已经按表维护 sessionComputedColumnsByTable_。
- 不需要额外引入内部 id。
- 便于 UI 与会话层之间直接传递列名。

### 3.2 改名语义

如果编辑时修改了 name，本质语义仍然是：

- 用旧 name 定位原列。
- 用新定义整体替换原列。

替换后仍需保证目标表内 name 唯一。

## 4. 存储规则

计算列定义保存在 SCDatabaseSession 的内存结构中：

```text
QHash<QString, QVector<SCComputedColumnDef>> sessionComputedColumnsByTable_
```

含义如下：

- Key 为 tableName。
- Value 为该表当前会话下挂载的全部计算列定义。

这个结构是当前会话的单一事实来源。ComputedTableView 不是事实来源，而是根据该结构重建出来的运行时视图。

## 5. 新增、编辑、删除语义

### 5.1 新增

新增流程：

1. 校验 name 非空。
2. 校验当前表内不存在同名计算列。
3. 追加到当前表的会话计算列列表。
4. 重建当前 ComputedTableView。
5. 刷新表格与计算列面板。

### 5.2 编辑

编辑流程：

1. 用原 name 定位旧定义。
2. 校验新定义合法。
3. 若新 name 与其他列冲突，则拒绝提交。
4. 用新定义整体替换旧定义。
5. 重建当前 ComputedTableView。
6. 刷新表格与计算列面板。

当前不做：

- 按列差异做局部更新。
- 保留旧列实例并原地修改底层对象。

原因是当前统一重建更简单，行为也更明确。

### 5.3 删除

删除流程：

1. 按 name 定位目标计算列。
2. 从当前表的会话计算列列表中移除。
3. 重建当前 ComputedTableView。
4. 刷新表格与计算列面板。

## 6. 视图重建规则

以下操作需要重建当前 ComputedTableView：

- 新增会话级计算列。
- 编辑会话级计算列。
- 删除会话级计算列。
- 切换当前表。

统一重建的好处：

- 避免局部更新状态不一致。
- 让 UI 刷新规则更容易理解。
- 降低运行时对象复杂度。

## 7. Undo/Redo 规则

数据库级 Undo/Redo 只作用于事实数据，不作用于会话级计算列定义。

因此：

- 编辑事实记录后可 Undo/Redo。
- 新增、编辑、删除会话计算列后，不进入数据库事务历史。

这样做的原因是会话计算列属于编辑器级临时视图状态，而不是数据库中的持久事实。

## 8. UI 规则

### 8.1 入口

当前 UI 至少应提供以下入口：

- 新增计算列。
- 编辑当前选中的计算列。
- 删除当前选中的计算列。

### 8.2 选中规则

- 编辑完成后，若列仍存在，应自动选中新列。
- 删除完成后，应尽量保留相邻项作为新的选中项。
- 若列表为空，则清除选中状态。

### 8.3 反馈规则

- 新增成功后给出明确状态提示。
- 编辑失败时给出具体错误原因。
- 删除前建议做确认。

## 9. 约束

当前版本约束如下：

- 仅支持当前表范围内的会话级定义管理。
- 不支持导入、导出会话级计算列配置。
- 不支持跨会话保留。
- 不支持更细粒度的依赖图可视化。

## 10. 后续可扩展点

后续可以考虑：

- 会话级计算列模板导入导出。
- 会话计算列分组与排序。
- 更丰富的依赖检查与循环依赖提示。
- 允许用户选择“会话级”与“持久化级”两种计算列模式。
