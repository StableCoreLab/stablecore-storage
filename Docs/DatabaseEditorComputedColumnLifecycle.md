# DatabaseEditor 会话级计算列生命周期设计

本文定义 `Tools/DatabaseEditor` 中“会话级计算列”的最小可用生命周期规则，用于支撑新增、编辑、删除，以及与普通 Schema 字段之间的显式互转。

设计日期：
- 2026-03-25

## 1. 范围

本文只讨论数据库编辑器中的“会话级计算列”：

- 仅在当前编辑器会话内有效。
- 不持久化到 SQLite。
- 通过 `ComputedTableView` 挂载到当前表视图。
- 支持与普通 Schema 字段之间的显式互转。

本文不讨论：

- 实体 Schema 列的完整通用编辑模型。
- 持久化计算列。
- 通用规则注册系统的完整设计。

## 2. 核心结论

会话级计算列遵循以下原则：

1. 计算列的唯一标识按“当前表内 `name` 唯一”处理。
2. 计算列编辑按“整条定义替换”处理，不做局部 patch。
3. 新增、编辑、删除后统一重建当前表的 `ComputedTableView`。
4. 计算列的增删改不进入数据库 Undo/Redo 历史。
5. 计算列状态由 `SCDatabaseSession` 的会话内结构维护。
6. 普通字段与计算列之间的互转必须显式触发，不能隐藏在普通编辑里。

## 3. 标识规则

会话级计算列的最小唯一键可视为：

```text
tableName + computedColumn.name
```

约束如下：

- 同一张表内，计算列 `name` 必须唯一。
- 不同表可以使用相同的计算列 `name`。
- UI 与会话层都按“表 + 名称”定位会话计算列。

如果编辑时修改了 `name`，其语义仍然是：

- 用旧 `name` 定位原列；
- 用新定义整体替换原列；
- 替换后仍必须保证目标表内唯一。

## 4. 存储模型

计算列定义保存在 `SCDatabaseSession` 的会话内结构中：

```text
QHash<QString, QVector<SCComputedColumnDef>> sessionComputedColumnsByTable_
```

含义如下：

- Key：表名。
- Value：该表当前会话中挂载的所有计算列定义。

`ComputedTableView` 不是事实源，而是根据该结构重建出来的运行时视图。

## 5. 生命周期语义

### 5.1 新增

新增流程：

1. 校验 `name` 非空。
2. 校验当前表内不存在同名计算列。
3. 校验定义本身合法。
4. 将定义加入当前表的会话计算列列表。
5. 重建当前 `ComputedTableView`。
6. 刷新 UI。

### 5.2 编辑

编辑流程：

1. 用旧 `name` 定位目标计算列。
2. 校验新定义合法。
3. 若新 `name` 与其他列冲突，则拒绝提交。
4. 用新定义整体替换旧定义。
5. 重建当前 `ComputedTableView`。
6. 刷新 UI。

当前不做：

- 按字段差异做局部更新。
- 保留旧列实例并原地修改底层对象。

### 5.3 删除

删除流程：

1. 按 `name` 定位目标计算列。
2. 从当前表的会话计算列列表中移除。
3. 重建当前 `ComputedTableView`。
4. 刷新 UI。

## 6. 互转语义

### 6.1 普通字段 -> 计算列

该转换是一个显式动作，不应被普通 `Edit Column...` 隐式触发。

转换规则：

1. 仅允许同名转换。
2. 转换前必须由后端结构化清空该普通字段已有数据。
3. 移除 Schema 中的普通字段定义。
4. 添加同名会话级计算列定义。
5. 重建当前 `ComputedTableView`。
6. 刷新 UI。

清空数据的原因：

- 普通字段与计算列属于不同语义层。
- 避免旧持久数据继续以普通字段语义残留。
- 让转换行为保持显式且可预测。

### 6.2 计算列 -> 普通字段

该转换同样是显式动作，不应被普通编辑入口隐式触发。

转换规则：

1. 仅允许同名转换。
2. 先从会话计算列列表中移除目标计算列。
3. 添加同名普通 Schema 字段。
4. 重建当前 `ComputedTableView`。
5. 刷新 UI。

注意：

- 该方向转换不会生成新的持久计算列数据。
- 如果同名普通字段已存在，必须拒绝。

## 7. 视图重建规则

以下操作必须重建当前 `ComputedTableView`：

- 新增会话级计算列。
- 编辑会话级计算列。
- 删除会话级计算列。
- 普通字段与会话级计算列之间的互转。
- 切换当前表。

统一重建的好处：

- 避免局部更新状态不一致。
- 让 UI 刷新规则更容易理解。
- 降低运行时对象复杂度。

## 8. Undo/Redo 规则

数据库级 Undo/Redo 只作用于事实数据，不作用于会话级计算列定义。

因此：

- 事实记录编辑后可 Undo/Redo。
- 新增、编辑、删除会话计算列后，不进入数据库事务历史。
- 普通字段与计算列互转时，只对持久数据部分做必要的显式事务处理。

会话计算列属于编辑器级临时视图状态，而不是数据库中的持久事实。

## 9. UI 规则

### 9.1 入口

当前 UI 至少应提供以下入口：

- 新增计算列。
- 编辑当前选中的计算列。
- 删除当前选中的计算列。
- 将选中的普通字段转换为会话级计算列。
- 将选中的会话级计算列转换为普通字段。

### 9.2 选中规则

- 编辑完成后，若列仍存在，应自动选中新列。
- 删除完成后，应尽量保留相邻项作为新的选中项。
- 若列表为空，则清除选中状态。

### 9.3 反馈规则

- 新增成功后给出明确状态提示。
- 编辑失败时给出具体错误原因。
- 删除前建议做确认。
- 互转失败时必须说明失败原因，尤其是“同名冲突”“定义非法”“数据清空失败”“视图重建失败”。

## 10. 当前约束

当前版本约束如下：

- 只支持当前表范围内的会话级定义管理。
- 不支持导入、导出会话级计算列配置。
- 不支持跨会话持久化。
- 不支持更细粒度的依赖图可视化。
- 转换入口必须保持显式，不允许隐藏在普通编辑对话框中。

## 11. 后续可扩展点

后续可以考虑：

- 会话级计算列模板导入/导出。
- 会话级计算列分组与排序。
- 更丰富的依赖检查与循环依赖提示。
- 允许用户显式选择“会话级”与“持久化”两种计算列模式。
