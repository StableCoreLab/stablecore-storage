# 数据库编辑器会话级计算列生命周期设计

本文档定义 `Tools/DatabaseEditor` 中“会话级计算列”的最小可用生命周期规则，用于支撑新增、编辑、删除功能，并保持实现与交互一致。

设计日期：

- `2026-03-25`

## 1. 范围

本文档只讨论数据库编辑器中的“会话级计算列”：

- 仅在当前编辑器进程内有效
- 不持久化到 SQLite 数据库
- 通过 `ComputedTableView` 挂载到当前表格视图

不讨论：

- 事实列 Schema
- 持久化计算列
- 通用规则注册系统

## 2. 基本结论

当前版本采用以下规则：

1. 计算列以“当前表内唯一 `name`”作为标识
2. 编辑计算列按“整条定义替换”处理，不做局部 patch
3. 删除或编辑后统一重建当前表的 `ComputedTableView`
4. 计算列删改不进入数据库 `Undo/Redo`
5. 计算列的会话状态由 `DatabaseSession` 内存结构负责维护

## 3. 标识规则

### 3.1 唯一键

会话级计算列的最小唯一键定义为：

```text
tableName + computedColumn.name
```

约束：

- 同一张表内，计算列 `name` 必须唯一
- 不同表可以使用相同的计算列 `name`

选择该方案的原因：

- 当前实现已经按表维护 `sessionComputedColumnsByTable_`
- 不需要额外引入内部 id
- 便于在 UI 和会话层之间直接传递列名

### 3.2 改名语义

如果编辑时修改了 `name`，本质语义仍然是：

- 用旧 `name` 定位原列
- 以新定义整体替换

替换后仍需保证目标表内 `name` 唯一。

## 4. 存储规则

计算列定义保存在 `DatabaseSession` 的内存结构中：

```text
QHash<QString, QVector<ComputedColumnDef>> sessionComputedColumnsByTable_
```

含义：

- Key 为 `tableName`
- Value 为该表当前会话下挂载的全部计算列定义

此结构是当前会话的单一事实来源。

`ComputedTableView` 不是事实来源，而是根据该结构重建出的运行时视图。

## 5. 新增、编辑、删除语义

### 5.1 新增

新增流程：

1. 校验 `name` 非空
2. 校验当前表内不存在同名计算列
3. 追加到当前表的会话计算列列表
4. 重建当前 `ComputedTableView`
5. 刷新表格和计算列面板

### 5.2 编辑

编辑流程：

1. 用原 `name` 定位旧定义
2. 校验新定义合法
3. 若新 `name` 与其他列冲突，则拒绝提交
4. 用新定义整体替换旧定义
5. 重建当前 `ComputedTableView`
6. 刷新表格和计算列面板

当前不做：

- 按字段差异局部更新
- 保留旧列实例并原地修改底层对象

原因：

- 当前 `ComputedTableView` 只有新增接口，缺少稳定的原地更新/移除接口
- 统一重建更简单，行为也更明确

### 5.3 删除

删除流程：

1. 用 `name` 定位目标计算列
2. 从当前表会话计算列列表中移除
3. 重建当前 `ComputedTableView`
4. 刷新表格和计算列面板

## 6. 视图重建规则

删改计算列后，统一调用当前表视图重建流程：

1. 清空现有 `currentTableView_`
2. 基于当前表重新创建新的 `ComputedTableView`
3. 重新挂载该表在 `sessionComputedColumnsByTable_` 中保存的全部计算列

这样可以避免以下复杂问题：

- 底层视图不支持移除单个计算列
- 编辑后旧定义残留
- UI 状态与运行时列集合不一致

## 7. Undo/Redo 规则

当前版本规定：

- 会话级计算列的新增、编辑、删除 **不进入数据库 `Undo/Redo`**

原因：

- 数据库 `Undo/Redo` 对应的是持久化事务
- 会话级计算列不写入数据库，也不属于存储层事实数据
- 如果强行混入数据库事务模型，会模糊“事实数据变更”和“编辑器会话配置变更”的边界

后续若需要支持撤销，建议单独设计“编辑器级命令栈”，而不是复用数据库事务日志。

## 8. UI 规则

### 8.1 入口

当前 UI 需要提供：

- `Add Session Computed Column...`
- `Edit Selected Computed Column...`
- `Delete Selected Computed Column`

### 8.2 选中规则

计算列面板中的顶层项对应一个计算列定义。

UI 定位规则：

- 顶层项保存该计算列的 `name`
- 若用户选中子项，则向上找到顶层项
- 编辑和删除都以顶层项保存的 `name` 为准

### 8.3 反馈规则

删除前需要二次确认。

编辑或删除成功后应：

- 刷新主表格
- 刷新计算列面板
- 刷新状态栏消息

## 9. 约束

当前约束保持如下：

- 计算列仍然是只读列，不可直接在表格里编辑值
- 计算列仍然不持久化
- `Rule` 类型仍依赖底层 `ruleId` 注册表
- 同名冲突直接拒绝，不做自动改名

## 10. 后续可扩展点

如果未来能力继续增强，可考虑：

1. 为计算列引入稳定内部 id，而不只依赖 `name`
2. 增加复制计算列
3. 引入编辑器级 Undo/Redo
4. 支持导入导出会话计算列配置
5. 支持 sidecar 持久化
