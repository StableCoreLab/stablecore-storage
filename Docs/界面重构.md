# DatabaseEditor 界面重构蓝图

## 1. 目标

这次重构的目标不是单独美化 `SCSchemaTableDialog`，而是把整个 `DatabaseEditor` 调整成更接近数据库工具的连续工作流：

```text
对象浏览 -> 表/数据编辑 -> 结构设计 -> 变更/诊断查看
```

重构后，用户应该围绕“当前表”持续工作，而不是在菜单、对话框、右侧面板和底部标签之间来回寻找入口。

本次方案只涉及 `Tools/DatabaseEditor` 的工具层界面组织，不改变存储语义，不新增绕过 `SCDatabaseSession` 的访问路径，不引入新的持久化状态来源。

## 2. 现状判断

当前实现已经具备以下基础：

- 主窗口已有左侧对象树、右侧 inspector、底部 activity tabs。
- 中央区域已有数据网格、过滤、记录操作、列操作。
- schema 导出已由 `SCSchemaTableDialog` 提供。
- schema 导入已由 `SCSchemaTableImportDialog` 提供。
- 约束和索引已有会话层接口与编辑对话框支持。

当前核心问题不是“没有能力”，而是“能力分散且工作流不连续”：

- 中央主工作区只有数据页，没有结构设计和 schema text 的常驻工作区。
- 结构设计入口隐藏在菜单和弹窗之后。
- `SCSchemaTableDialog` 更像代码导出器，不像表设计器。
- schema 导出和 schema 导入是两个独立心智模型。
- inspector 和底部 tabs 信息很多，但高频设计动作不够集中。
- 文案和编码状态不统一，削弱工具感。

## 3. 设计原则

### 3.1 工作流优先

主窗口应围绕“当前表”构建，而不是围绕“功能菜单”构建。

### 3.2 中央工作区优先

高频任务必须进入中央工作区，不能继续主要依赖弹窗。

### 3.3 对象导航只负责导航

左侧 `Object Explorer` 负责切换工作上下文，不负责承载完整编辑行为。

### 3.4 Session 仍是唯一真实入口

所有数据、schema、约束、索引的读取和写入仍通过 `SCDatabaseSession` 完成。  
UI 不缓存第二份“可提交 schema 真相”，只维护当前选择状态和当前视图状态。

### 3.5 先重组信息架构，再做视觉 polish

优先处理入口可见性、工作区结构、文案一致性，再处理按钮层级、空状态和视觉细节。

## 4. 目标布局

目标主窗口布局如下：

```text
+--------------------------------------------------------------------------------------------------+
| Menu Bar                                                                                         |
+--------------------------------------------------------------------------------------------------+
| Main Toolbar                                                                                     |
+--------------------------------------------------------------------------------------------------+
| Object Explorer      | Workspace Header + Mode Switcher + Primary Actions | Inspector            |
|                      |-----------------------------------------------------|                      |
| Database             | Data Page                                           | Selection Details    |
| Tables               | Design Page                                         | Record Details       |
| Computed             | Schema Text Page                                    | Relation Details     |
| Activity             |                                                     | Computed Details     |
| Diagnostics          |                                                     |                      |
+----------------------+-----------------------------------------------------+----------------------+
| Activity / SQL / Diagnostics                                                                      |
+--------------------------------------------------------------------------------------------------+
| Status Bar                                                                                       |
+--------------------------------------------------------------------------------------------------+
```

### 4.1 四区职责

- 左侧：`Object Explorer`
- 中央：`Workspace`
- 右侧：`Inspector`
- 底部：`Activity`

其中中央工作区是本次重构的核心。

## 5. Phase 1：主窗口重组

Phase 1 的目标是先把主窗口重排成标准数据库工具结构，不要求一次完成所有局部 widget 拆分。

### 5.1 中央工作区改造

当前 [SCDatabaseEditorMainWindow.cpp](</d:/code/StableCore/Storage/Tools/DatabaseEditor/SCDatabaseEditorMainWindow.cpp>) 的中央区只有 `tablePage_`。  
建议改成：

- `workspaceHeader_`
- `workspaceModeTabs_` 或 `workspaceModeBar_`
- `workspaceStack_`

建议成员：

```cpp
QWidget* workspaceHeader_{nullptr};
QLabel* workspaceTitleLabel_{nullptr};
QLabel* workspaceStatsLabel_{nullptr};
QLabel* workspaceModeStateLabel_{nullptr};

QTabBar* workspaceModeBar_{nullptr};
QStackedWidget* workspaceStack_{nullptr};

QWidget* dataPage_{nullptr};
QWidget* designPage_{nullptr};
QWidget* schemaTextPage_{nullptr};
```

建议模式枚举：

```cpp
enum class WorkspaceMode
{
    Data,
    Design,
    SchemaText,
};
```

### 5.2 Workspace Header

中央区顶部应固定显示：

- 当前表名
- 当前模式
- 行数 / 列数 / dirty 状态
- 当前模式下的主操作按钮

建议布局：

```text
[Table: Beam] [Rows: 128 | Columns: 12 | Pending edits]
                     [Data] [Design] [Schema Text]
 [Primary Actions...]
```

建议动作分组：

- `Data`
  - `Add Record`
  - `Delete Record`
  - `Import CSV`
  - `Export CSV`

- `Design`
  - `Add Column`
  - `Edit Column`
  - `Add Constraint`
  - `Add Index`

- `Schema Text`
  - `Refresh`
  - `Copy`
  - `Validate`
  - `Import as New Table`

第一阶段不要求把所有按钮都迁移到 header，也可以先保留现有 `tableToolBar_`，但逻辑上要开始按模式分组。

### 5.3 Data Page

`DataPage` 复用现有中央数据编辑能力：

- 过滤框
- 清空过滤
- 数据网格
- 行级右键菜单

推荐布局：

```text
[Quick Filter: ____________] [Clear]
------------------------------------------------
| Grid                                          |
------------------------------------------------
```

现有 `tablePage_` 可以直接重命名并演进为 `dataPage_`，尽量减少一次性搬迁量。

### 5.4 Design Page

`DesignPage` 在 Phase 1 先占位，Phase 2 再做完整设计器。

第一阶段可接受的过渡方案：

- 中央 `DesignPage` 放一个嵌入式占位面板
- 顶部显示当前表 overview
- 中部提供：
  - `Open Full Design`
  - `Manage Constraints`
  - `Manage Indexes`
  - `Generate Schema Text`

如果希望 Phase 1 直接可用，也可以把 `SCSchemaTableDialog` 的核心内容以 widget 形式内嵌，但不建议把整个 dialog 逻辑硬塞进主窗口。

### 5.5 Schema Text Page

`SchemaTextPage` 在 Phase 1 也可以先占位，但必须成为主工作区的正式模式，而不是继续只挂在菜单里。

建议最小内容：

- 左侧：当前表 schema 预览
- 右侧：导入文本区
- 顶部：导出 / 刷新 / 复制 / 验证 / 导入按钮
- 底部：warnings / parse message

### 5.6 左侧 Object Explorer 重组

当前左侧树结构只有表和系统对象，建议调整为更贴近数据库工具语义的导航结构。

目标结构：

```text
Database
├─ Tables
│  ├─ Beam
│  │  ├─ Columns
│  │  ├─ Constraints
│  │  ├─ Indexes
│  │  └─ Records
│  ├─ Column
│  └─ Wall
├─ Computed
├─ Activity
│  ├─ Edit Log
│  └─ Snapshots
└─ Diagnostics
```

这里的子节点主要是导航语义，不意味着要新增持久模型或独立数据查询通道。

建议扩展 `ExplorerNodeType`：

```cpp
enum class ExplorerNodeType
{
    Database,
    TablesRoot,
    Table,
    TableColumns,
    TableConstraints,
    TableIndexes,
    TableRecords,
    ComputedRoot,
    ComputedColumn,
    ActivityRoot,
    EditLog,
    Journal,
    Snapshots,
    Diagnostics,
};
```

### 5.7 对象树行为表

对象树点击后的中央区和右侧区联动建议如下：

| 选择对象 | 中央模式 | 右侧 Inspector | 底部 Activity |
| --- | --- | --- | --- |
| `Table` | `Data` | Schema / Selection | 保持当前 |
| `Columns` | `Design` | Column details | 保持当前 |
| `Constraints` | `Design` | Constraint details | 保持当前 |
| `Indexes` | `Design` | Index details | 保持当前 |
| `Records` | `Data` | Record details | 保持当前 |
| `Computed` | 当前或 `Data` | Computed details | 保持当前 |
| `Edit Log` | 保持当前 | 保持当前 | 切到 `Edit Log` |
| `Snapshots` | 保持当前 | 保持当前 | 切到 `Health` 或 Snapshot view |
| `Diagnostics` | 保持当前 | 保持当前 | 切到 `Diagnostics` |

建议原则：

- 系统对象只切换底部 activity，不抢占中央表工作区。
- 表相关节点始终围绕当前表切换 `Data` / `Design`。

### 5.8 Inspector 重命名和职责收敛

当前 inspector 由固定 tabs 构成：

- schema
- record
- computed
- relation

第一阶段不需要完全重写，但建议做两件事：

- 统一英文文案
- 让 tab 切换更明显地服务于当前选择对象

建议 tab：

- `Selection`
- `Record`
- `Computed`
- `Relations`

其中 `Selection` tab 在 `Data` 模式显示当前列/字段信息，在 `Design` 模式显示当前列、约束或索引信息。

### 5.9 底部 Activity 重新定位

当前底部 tabs 能力保留，但默认视觉权重下调。

建议 tabs：

- `Diagnostics`
- `Edit Log`
- `Health`
- `SQL Preview`
- `Debug Package`

建议改动：

- 默认高度减小
- 默认停留在更少打扰的 tab
- 从“主工作内容”退到“辅助查看内容”

### 5.10 旧入口迁移表

| 当前入口 | 问题 | 新位置 |
| --- | --- | --- |
| `Table -> 表结构...` | 隐蔽，依赖弹窗 | `Design` 模式 |
| `Table -> 从模式创建表...` | 与导出流程割裂 | `Schema Text` 模式 |
| `SCSchemaTableDialog` | 像导出器，不像设计器 | 拆入 `Design` 和 `Schema Text` |
| `SCSchemaTableImportDialog` | 单点导入，不成工作区 | 合并到 `Schema Text` |
| 右侧 schema tree | 信息多，动作弱 | 保留为 inspector，不再承担主设计入口 |

### 5.11 Phase 1 建议新增槽函数

建议新增：

```cpp
void SetWorkspaceMode(WorkspaceMode mode);
void RefreshWorkspaceHeader();
void RefreshWorkspacePages();
void RefreshDesignPage();
void RefreshSchemaTextPage();
void OnWorkspaceModeChanged(int index);
void OpenCurrentTableDesign();
void OpenCurrentTableSchemaText();
```

建议保留但逐步弱化：

- `OpenSchemaTableConverter()`
- `CreateTableFromSchemaDescription()`

它们在 Phase 1 可以继续工作，但应改为跳转或桥接到新工作区，而不是长期保留为主入口。

## 6. Phase 2：表设计器重做

Phase 2 的目标是把 `SCSchemaTableDialog` 从“导出弹窗”升级为真正的 `Design` 工作区。

建议新增局部 widget：

- `SCTableDesignPane`
- `SCConstraintListPane`
- `SCIndexListPane`

### 6.1 `SCTableDesignPane` 结构

建议结构：

```text
Table Overview
------------------------------------------------
Table Name
Description
Primary Key
Include Legacy Index Hints
Columns / Constraints / Indexes summary

Columns                           Structure
---------------------------      ----------------------------
[column list]                    [Constraints] [Indexes]
Add / Edit / Delete              list + actions

Generated Schema
------------------------------------------------
[preview text]
[Copy] [Refresh]
```

### 6.2 Overview 区

建议字段：

- `Table Name`
- `Description`
- `Primary Key`
- `Include Legacy Index Hints`
- `Stats`

建议说明：

- 主键输入框保留为快捷编辑入口
- 但主键也必须出现在 `Constraints` 视图中，避免用户误解“主键不属于约束”

### 6.3 Columns 区

列区是设计器左侧主列表，建议使用表格或多列 list，而不是单纯字符串列表。

建议列信息：

- `Name`
- `Display Name`
- `Type`
- `Nullable`
- `Default`
- `Kind`

建议动作：

- `Add Column`
- `Edit Column`
- `Delete Column`
- 后续可预留 `Move Up / Move Down`

### 6.4 Constraints 区

约束必须升级为一级概念，而不是主键输入框旁边的附属能力。

建议顶部 summary：

- `Primary Key: 1`
- `Unique: n`
- `Foreign Key: n`
- `Check: n`

建议列表列：

- `Name`
- `Type`
- `Columns`
- `Target`
- `Actions`
- `Source`

说明：

- `PrimaryKey` 出现在列表中
- 但编辑主键时跳转到 overview 的快捷字段
- 其余约束通过 `SCConstraintEditorDialog` 或后续内嵌编辑器维护

### 6.5 Indexes 区

索引区应提升成真正的索引管理器，不再只展示字符串拼接结果。

建议列表列：

- `Name`
- `Columns`
- `Descending`
- `Source Kind`
- `Badge`

说明：

- `Descending` 用独立列或 badge，不仅仅拼在文本里
- `Source Kind` 明确区分 `LegacyHint` 和 `Explicit`
- `LegacyHint` 应弱化展示，必要时只读

### 6.6 Generated Schema 区

虽然设计器重点不再是“代码导出”，但 schema preview 仍然是有价值的辅助信息。

建议保留：

- `Generated Schema`
- `Copy`
- `Refresh`
- `Show metadata comments`

但它应位于设计器底部，作为辅助结果，而不是整个界面的主角。

## 7. Phase 3：Schema Text 工作区整合

Phase 3 的目标是把导入和导出统一成单一心智模型。

建议新增：

- `SCSchemaTextPane`

### 7.1 结构

```text
Schema Text Workspace
------------------------------------------------
Export Current Table         Import Schema as New Table
[generated preview]          [editable text]

Actions
Refresh / Copy               Validate / Import

Messages
[warnings / parse messages / import results]
```

### 7.2 页面语义

必须明确区分：

- `Export current table schema`
- `Import schema as new table`

如果暂时不支持 patch apply，应明确显示：

- `Apply to current table`
- disabled
- `Not implemented yet`

避免用户误解为双向在线 schema patch 工具。

### 7.3 与旧对话框关系

- `SCSchemaTableDialog`
  - 导出逻辑保留
  - UI 逐步下沉到 `SCTableDesignPane` 和 `SCSchemaTextPane`

- `SCSchemaTableImportDialog`
  - 解析与导入逻辑保留
  - UI 逐步下沉到 `SCSchemaTextPane`

## 8. 文案与命名统一

文案和编码异常建议作为独立子任务推进，但命名方向应先固定。

建议统一英文命名：

- `Object Explorer`
- `Data`
- `Design`
- `Schema Text`
- `Inspector`
- `Selection`
- `Constraints`
- `Indexes`
- `Activity`
- `Edit Log`
- `Diagnostics`
- `Health`
- `SQL Preview`
- `Debug Package`

建议统一状态文案：

- `No database open`
- `No table selected`
- `Read-only`
- `Pending edits`
- `Schema changed`
- `Legacy index hints detected`
- `Ready`

## 9. 类与职责建议

### 9.1 保持主窗口只做协调

`SCDatabaseEditorMainWindow` 负责：

- 布局
- 导航切换
- 模式切换
- pane 之间协调
- 状态刷新触发

不建议把复杂的表设计细节继续堆回主窗口。

### 9.2 建议新增局部 widget

- `SCTableDesignPane`
  - 当前表 overview
  - columns / constraints / indexes 编辑

- `SCConstraintListPane`
  - 约束列表和摘要

- `SCIndexListPane`
  - 索引列表和摘要

- `SCSchemaTextPane`
  - schema 导出预览
  - schema 导入文本
  - parse / import 消息

### 9.3 数据边界

这些 widget 只调用 `SCDatabaseSession` 提供的接口，不直接碰 SQLite，也不在 UI 中复制持久语义。

## 10. 实施顺序建议

### 10.1 Phase 1

主窗口重组：

- 增加 `Data / Design / Schema Text` 模式切换
- 重组 object explorer
- 重命名并弱化底部 activity
- 让旧入口桥接到新模式

### 10.2 Phase 2

重做表设计器：

- 拆分 `SCSchemaTableDialog`
- 引入 `SCTableDesignPane`
- 让约束和索引成为一级对象

### 10.3 Phase 3

整合 schema text：

- 引入 `SCSchemaTextPane`
- 合并导入/导出心智模型
- 明确 disabled 的 future action

### 10.4 Phase 4

文案和视觉 polish：

- 清理乱码和编码异常
- 统一命名
- 调整层级、空状态、摘要和按钮优先级

## 11. 预期收益

完成后，工具会从“调试面板拼装体”更接近“围绕当前表工作的数据库工具”：

- 结构设计入口显著提升
- schema 导入导出心智模型更清晰
- 约束和索引不再是隐藏能力
- inspector 与 activity 退回辅助角色
- 主工作流围绕当前表连续展开

## 12. 风险与控制

### 12.1 最大风险

最大的风险不是单个控件怎么排，而是一次性改得过大，导致工具更乱。

### 12.2 控制策略

- Phase 1 只重组主窗口信息架构
- Phase 2 再重做设计器
- Phase 3 最后统一 schema text
- 每一阶段都保持原有 session 语义和编辑边界不变

### 12.3 明确不做的事

本方案不包括：

- 改存储语义
- 改 SQLite backend
- 引入 UI 本地 schema patch 引擎
- 一次性推翻全部旧对话框逻辑
