# 当前实现状态

本文档用于记录“文档承诺 vs 代码事实”的对照结果。

处理规则：
- 文档描述与代码一致，标记为“同步”。
- 文档描述比代码更乐观，标记为“文档需更新”。
- 文档设计正确，但代码尚未实现，标记为“代码需补”。

## 总体判断

- 核心语义层已经落地：`Database / Table / Record / Schema / Transaction / ChangeSet` 主线是通的。
- 内存后端与 SQLite 后端都可工作，但 SQLite 主要承担持久化镜像与事务日志职责，不是完整的查询执行引擎。
- 计算列、批量导入、诊断、迁移、编辑器会话都已经有可运行原型或部分正式实现。
- 当前最大的偏差集中在查询/索引、启动恢复、引用索引边界、以及部分工程化收口。

## 代码需补但不该降级文档的项

这些项的设计方向本身是合理的，问题在代码实现，还不应靠改文档“解释掉”：

- 启动完整性检查未接到主启动路径。
- 索引物化入口未真正落线。
- SQLite 查询未下推到数据库引擎。
- 引用索引仍偏扫描和重建，缺少增量维护。
- 批处理和导入 finalize 的原子边界还不够强。
- 观察者与会话生命周期的工程化防护还需要加强。

## 备份与基线

- `CreateBackupCopy` 已按 Level 2 语义落地，SQLite 支持文件型一致性副本生成，Memory 后端明确返回 `SC_E_NOTIMPL`
- `ResetHistoryBaseline` 已与 `baselineVersion` 的持久化语义对齐，不再把当前版本冒充基线版本
- `ResetHistoryBaseline(...)` 现在会在 SQLite `COMMIT` 成功后再发布 `baselineVersion_ / undoStack_ / redoStack_`，失败时不再提前清空会话内历史视图
- 备份与基线相关的长期说明已经收敛到系统文档 `[Storage 备份与基线](StorageBackupAndBaseline.md)`
- 原 `CreateBackupCopy Level 2 完整开发方案.md` 已从专题材料转为一次性归档材料

## 结论

- 如果以“文档是否准确反映当前实现”为标准，当前需要更新的是查询/索引、启动恢复、基线定义和能力评估类文档。
- 如果以“设计是否正确”为标准，查询下推、引用索引增量维护、启动检查接入这些方向不是改文档的问题，而是继续补代码的问题。
## DatabaseEditor 补充

- `Tools/DatabaseEditor` 已补充选中 Schema 字段的原地编辑入口，并可直接调用 `ISCDatabase::CreateBackupCopy` 生成备份副本。
- 同时补充了普通字段与会话级计算字段之间的显式互转入口；普通字段转换为计算字段时会先清理该列现有数据。
- DatabaseEditor 的列编辑语义正在向统一事务状态机收口，后续不应继续依赖分散的事后补救路径。
- 表格右键已补充新增/删除行入口，Schema 右键已补充新增/编辑/删除列入口。
## DatabaseEditor UpdateColumn

- `Edit Column...` now migrates compatible existing values when `valueKind` changes and rejects unsupported conversions.
- The editor now wraps column edits in an explicit begin/commit boundary instead of relying on a post-failure manual repair path.
- Schema add/update/remove now enter journal history as `journal_schema_entries`, so Undo / Redo and rollback can restore the exact column definition and preserve column order when a removed column is restored.
- DatabaseEditor object explorer now exposes right-click actions for table selection and table creation, and schema edits first try to reuse the currently selected table before failing.
- SQLite now journals migrated column values during `UpdateColumn(...)`, so a failed view rebuild or rollback restores both schema and record values.
- `Commit` / `Undo` / `Redo` persist the post-transaction version number to metadata before the transaction closes, so reopened databases report the same `currentVersion`.
- `Add Column` / `Edit Column` / computed-column conversions now share the same `ApplyColumnMutation(...)` transaction helper and rebuild the current table view from the post-mutation state.
- The final closure contract is documented in `Docs/DatabaseEditorColumnConversionTask.md`: pre-image recovery for `RemoveColumn(...)` and unified rollback reporting across session helpers.
- `PersistRemovedColumn(...)` now restores the in-memory pre-image on failure instead of relying on `field_values` reload as the primary recovery path.
- `PersistUpdatedColumn(...)` now checkpoints `activeJournal_` and uses one cleanup path for schema, migrated values, and journal state, so a failed SQLite value rewrite does not leak a half-applied in-memory schema edit.
- `Tests/M2SqliteTests.cpp` now includes SQLite-level regressions for failed `UpdateColumn(...)` value rewrite and failed `ResetHistoryBaseline(...)` commit publication.
