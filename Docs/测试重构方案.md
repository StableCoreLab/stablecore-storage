### 2. 再按“测试语义”拆文件，不按历史模块名堆

当前最大的问题不是文件多，而是主题不纯。  
比如 `QuerySqliteExecutorTests.cpp` 里面同时混着：

- `requireIndex`
- 复合索引精确命中
- 范围查询
- 排序覆盖
- null/default
- tie-break
- reopen / undo / redo

这些应该拆成主题文件，而不是继续放一个总包里。

`QuerySqliteExecutorTests.cpp` 拆为：

### 3. 引入夹具，不再每个测试手搓环境

建议至少引入 3 个夹具：

**`FileDbTest`**
- 持有 `dbPath`
- 提供 `CreateDb()` / `ReopenDb()`

**`BeamTableTest`**
- 基于 `FileDbTest`
- 自动创建标准 `Beam` 表
- 暴露 `db`, `table`, `schema`

**`CompositeBeamQueryTest`**
- 基于 `BeamTableTest`
- 提供复合索引版本的 Beam schema
- 提供 query helper / seed helper

这样大部分测试会从：

```cpp
const fs::path dbPath = MakeTempDbPath(...);
sc::SCDbPtr db;
EXPECT_EQ(CreateFileDb(...), SC_OK);
sc::SCTablePtr table = CreateBeamTable(db);
...
EXPECT_EQ(db->BeginEdit(...), SC_OK);
...
```

变成：

```cpp
class CompositeBeamQueryTest : public BeamTableTest { ... };

TEST_F(CompositeBeamQueryTest, RequireIndexAcceptsExplicitCompositeSchemaIndex)
{
    AddWidthNameCompositeIndex();
    SeedWidthNameRows(...);
    auto result = ExecuteBeamQuery(...);
    ...
}
```

这个改动对“减少重复代码”的帮助非常直接。

### 4. 给 helper 设边界，避免 Support 反向变成垃圾堆

Support 层要控住，只允许放这三类东西：

- 环境搭建
- 数据 seed
- 断言辅助

不要把“测试业务语义”塞进 helper。  
例如：

- `SeedTripleCompositeRows(...)` 可以放 helper
- `ExpectTailRangeUsesPartialIndex(...)` 不建议做成太高层 helper

否则文件虽然拆了，语义会被隐藏，读测试反而更累。

### 5. 文件命名改成“行为导向”，不要再用 `M1/M2/M3`

`M1Tests.cpp` / `M2SqliteTests.cpp` / `M3Tests.cpp` 这种名字对新读者几乎没有信息量。  
建议：

- 保留兼容性入口时，改成 `LegacyM1CompatibilityTests.cpp`
- 真正的语义测试迁到主题文件
- 最后让 `M1/M2/M3` 只承担“阶段性兼容/历史回归”角色，甚至逐步清空

**建议的验收标准**

重构不是只看文件数量，建议用下面的标准判断是否成功：

- 单个测试文件尽量控制在 `300-800` 行
- 单个测试文件只表达一个主题
- `Tests/Support/` 中不出现业务语义断言堆积
- 新增一个复合索引测试时，不需要再复制建库/建表/seed/query glue
- 搜 `MakeTempDbPath` / `CreateBeamTable` / `ExecSqliteScript` 时，测试目录里只剩 helper 定义，不再到处重复

**Test Coverage**

这次是重构设计，不是代码修改。  
不过从测试覆盖角度，拆分后会更容易维护这些高优先级语义回归：

- rollback recovery
- deleted-record access
- undo/redo consistency
- failed upgrade recovery
- readonly open
- query-index consistency
- composite index range execution
- require-index rejection semantics

**Risk Assessment**

主要风险有 4 个：

- 抽 helper 时把测试语义藏太深，导致可读性下降
- 大文件拆分后，原有静态局部 helper / 命名空间函数链接关系出问题
- 夹具过重，导致测试初始化路径不透明
- 拆文件时改动 include / CMake 列表过大，容易引入无关噪音

所以最稳的策略是：

- 先抽 helper
- 再拆最大文件
- 每次只动一个主题域
- 不在同一轮里同时改 helper 命名、测试命名、文件命名、断言风格
