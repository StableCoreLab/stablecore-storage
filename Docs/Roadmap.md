# StableCore Storage Roadmap

这份文档用于把当前设计和代码骨架收敛成一份可执行开发清单。

目标不是列“理想功能”，而是回答：

**这个库从现在的骨架状态，到真正可用于算量产品开发，还要做什么。**

## 1. 当前状态

当前仓库已经具备：

- 公共头文件骨架
- `RefPtr` / 引用计数基础设施
- 核心类型定义：`Value`、`ColumnDef`、`ChangeSet`、`JournalTransaction`
- 公共接口：`IDatabase`、`ITable`、`IRecord`、`ISchema`
- 内存版最小实现
- SQLite 工厂占位

当前仓库还不具备：

- 真正可交付的 SQLite 持久化
- 单元测试
- 可验证的 Undo/Redo 可靠性
- 计算字段运行时
- 产品接入所需的工程化和迁移能力

## 2. 里程碑

建议按 3 个里程碑推进：

1. `M1` 可开发
2. `M2` 可持久化
3. `M3` 可接产品

---

## 3. M1 可开发

目标：

- 上层可以基于稳定 API 继续开发
- 内存实现可以作为语义参考和测试基线
- 核心设计不再频繁返工

### 3.1 公共 API 收紧

- 明确 V1 正式公开接口，清理仅为骨架存在的临时设计
- 给 `Value` 增加稳定访问函数，而不仅仅是 `TryGet`
- 明确字符串生命周期语义
- 给接口补充必要注释和错误码说明
- 明确 deleted record 的读取边界

### 3.2 内存实现补齐

- 修正 `CreateRecord` 的事务前置校验
- 完善 `Rollback`、`Undo`、`Redo` 的边界行为
- 补齐字段默认值和 `Null` 语义
- 完善关系字段校验
- 补齐 Journal 聚合逻辑
- 明确 observer 回调重入策略

### 3.3 单元测试

必须覆盖：

- 创建表、添加字段、读取 schema
- 事务提交
- 事务回滚
- 字段修改
- 关系字段修改
- 删除记录后的失效语义
- `Undo/Redo` 对字段更新、创建、删除的行为
- `ChangeSet` 内容正确性
- 查询接口最小行为

### 3.4 文档和示例

- 给出最小示例代码
- 同步 README 和基线文档到实际接口
- 明确哪些功能属于 V1，哪些只是预留

### 3.5 M1 完成标准

满足以下条件可视为进入 `M1` 完成：

- API 基本稳定
- 内存实现能完整跑通核心语义
- 单元测试能作为回归基线
- 上层可以基于内存版并行开发 Model 和表格工具

---

## 4. M2 可持久化

目标：

- SQLite 成为真实底层存储引擎
- 事务、查询、Undo/Redo、Schema 能持久化
- 重启后数据可恢复

### 4.1 SQLite 适配层实现

- `SqliteDb`
- `SqliteStmt`
- `SqliteTxn`
- SQLite 错误码到 `SC_E_*` 的映射
- 打开/关闭数据库
- 预编译语句和参数绑定

### 4.2 SQLite 物理表结构

建议至少落地：

- `tables`
- `schema_columns`
- `records`
- `field_values`
- `journal_transactions`
- `journal_entries`
- `metadata`

并补必要索引。

### 4.3 存储核心与 SQLite 接轨

- `CreateSqliteDatabase(...)` 返回真实实例
- `BeginEdit/Commit/Rollback` 对齐 SQLite 事务
- 记录创建、删除、字段写入落库
- `GetTable/GetRecord/FindRecords` 走 SQLite
- 启动时加载 schema、版本和元数据

### 4.4 Journal 与 Undo/Redo 持久化

- Journal transaction 持久化
- Undo 栈 / Redo 栈状态持久化
- `Undo/Redo` 跨进程重启后的恢复策略
- `Undo/Redo` 触发后 `ChangeSet` 保持正确

### 4.5 查询能力补齐

- 单字段等值查询走 SQLite
- 关系字段等值查询
- 全表遍历
- deleted record 查询策略
- 基本索引优化

### 4.6 Schema 管理补齐

- 创建字段持久化
- 加载字段定义
- 用户定义字段持久化
- 关系字段目标表约束
- 字段默认值持久化

### 4.7 M2 完成标准

满足以下条件可视为进入 `M2` 完成：

- SQLite 成为默认可用存储后端
- 数据重启后可恢复
- 事务、查询、Undo/Redo、Schema 均可持久化
- 内存版与 SQLite 版语义一致

---

## 5. M3 可接产品

目标：

- 算量产品和数据表工具可以真正接入
- 性能、迁移、缓存和计算列达到可用水平

### 5.1 典型业务关系跑通

- `Floor` 表与 `FloorRef`
- 关系字段的引用校验
- `RelationUpdated` 的 `ChangeSet`
- 按楼层查询构件
- 删除被引用对象的行为约束

### 5.2 计算字段体系

- `ComputedColumnDef`
- 依赖声明模型
- 简单表达式求值器
- `ruleId` 注册机制
- `ChangeSet + Version` 驱动失效和重算
- 表格工具中的事实列 / 计算列分层

### 5.3 批量和性能能力

- 批量写入优化
- 批量导入事务优化
- 大数据量查询基线
- 关系查询性能验证
- 索引策略复查

### 5.4 工程化能力

- 数据库升级/迁移方案
- 崩溃恢复策略
- 损坏处理策略
- 调试工具和诊断日志
- 示例工程

### 5.5 产品接入前检查项

- API 是否还存在频繁变更点
- SQLite 表结构是否支持升级
- `ChangeSet` 是否足够支撑 UI 和增量重算
- 计算字段缓存是否可失效
- Undo/Redo 是否能支撑高频编辑场景

### 5.6 M3 完成标准

满足以下条件可视为进入 `M3` 完成：

- 产品可以直接接入 SQLite 版存储
- 数据表工具可同时显示事实列和计算列
- 典型构件关系和楼层关系可以稳定运行
- 性能和恢复能力达到实际开发可接受水平

---

## 6. 推荐执行顺序

建议严格按这个顺序推进：

1. 完成 `M1`，先把语义和测试做稳
2. 再做 `M2`，把 SQLite 变成真实后端
3. 最后做 `M3`，补业务接入、计算字段和性能

不建议跳过 `M1` 直接冲 SQLite 实现，否则很容易在持久化阶段一边写库一边改语义，返工会很大。

## 7. 下一步建议

当前最合理的下一步是：

1. 先补单元测试工程
2. 用测试把内存实现的核心语义钉住
3. 然后开始写 SQLite 适配层和物理表结构

这会比先冲 SQLite 更稳。
