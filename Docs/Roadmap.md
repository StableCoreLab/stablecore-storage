# StableCore Storage 路线图

本文用于把当前设计和代码骨架整理成一份可执行的开发路线图。

目标不是罗列理想功能，而是回答：

当前这个仓库距离“可被算量产品稳定接入”还需要完成哪些阶段性工作。

## 1. 当前状态

当前仓库已经具备：

- 公共 API 与基础类型体系。
- 内存存储实现。
- SQLite 持久化实现。
- 测试、示例和基础设计文档。
- 计算列、批量导入、恢复诊断和数据库编辑器的初步能力。

因此，路线图的重点不再是“从零开始”，而是持续补齐可产品化的关键缺口。

## 2. 里程碑

建议继续沿用三阶段里程碑：

- M1：可开发。
- M2：可持久化。
- M3：可接产品。

## 3. M1 可开发

### 3.1 公共 API 收紧

目标：

- 让核心接口边界稳定。
- 明确 Record、Table、Schema、EditSession、ChangeSet 的职责。

### 3.2 内存实现补齐

目标：

- 在不依赖持久化的情况下跑通事务、删除、Undo/Redo 和查询主路径。

### 3.3 单元测试

目标：

- 为核心语义建立最小防回归护栏。

### 3.4 文档和示例

目标：

- 保证接入者能快速理解基本用法和设计边界。

### 3.5 M1 完成标准

判断标准：

- API 基线清晰。
- 内存后端可稳定跑通主路径。
- 核心测试与示例可用。

## 4. M2 可持久化

### 4.1 SQLite 适配层实现

目标：

- 建立与存储核心对齐的 SQLite 后端。

### 4.2 SQLite 物理表结构

目标：

- 明确元数据、记录、Journal、索引等基础结构。

### 4.3 存储核心与 SQLite 接轨

目标：

- 保证同一套接口既能驱动内存后端，也能驱动 SQLite 后端。

### 4.4 Journal 与 Undo/Redo 持久化

目标：

- 把事务历史与回放能力带到持久化后端。

### 4.5 查询能力补齐

目标：

- 支撑等值查询、关系查询和基础索引路径。

### 4.6 Schema 管理补齐

目标：

- 支撑持久化表与列定义管理。

### 4.7 M2 完成标准

判断标准：

- SQLite 后端可稳定读写。
- 启动、恢复和基础诊断可用。
- SQLite 集成测试已建立。

## 5. M3 可接产品

### 5.1 典型业务关系跑通

目标：

- 覆盖楼层关系、构件关系等常见业务形态。

### 5.2 计算列体系

目标：

- 建立表达式、规则型与聚合型计算列的基础能力。

### 5.3 批量和性能能力

目标：

- 提供批量导入、性能冒烟和索引优化起点。

### 5.4 工程化能力

目标：

- 让示例、测试、诊断和工具达到持续研发所需水平。

### 5.5 产品接入前检查项

建议重点检查：

- 数据恢复链路是否清晰。
- 典型业务关系是否稳定。
- 计算列是否能满足研发验证。
- 文档是否足够支撑接入团队。

### 5.6 M3 完成标准

判断标准：

- 上层产品可直接用 SQLite 后端开展研发接入。
- 数据表工具能支撑事实列与计算列验证。
- 性能与恢复达到可研发水平。

## 6. 推荐执行顺序

建议顺序：

1. 先收紧基线与测试。
2. 再持续完善持久化与恢复。
3. 最后集中补业务接入与工程化体验。

这样可以避免在基础不稳时过早叠加业务复杂度。

## 7. 下一步建议

当前建议优先推进：

- 系统化清理和统一文档。
- 继续补边界测试与故障路径测试。
- 对产品接入样例开展真实业务数据验证。
