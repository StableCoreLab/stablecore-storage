# 能力缺口评估

本文用于对照以下设计文档，评估当前 Storage 仓库的实现完成度与剩余缺口：

- Docs/Roadmap.md
- Docs/TaskBreakdown.md
- Docs/V1BaselineDecisions.md

评估日期：

- 2026-03-25

## 状态模型

本文采用三档状态：

- 已完成：仓库中已有稳定实现、测试或明确文档，可直接支撑后续开发。
- 基本完成：主路径已经跑通，但仍存在约束、覆盖不足或工程性短板。
- 未完成：设计已有目标，但仓库尚未提供足够实现或验证。

## 执行结论

总体上，仓库已经跨过“只有接口骨架”的阶段，具备了如下基础：

- 内存后端已形成可用的事务、记录、Schema、Undo/Redo 基线。
- SQLite 后端已具备启动、元数据加载、读写、恢复和诊断路径。
- 计算列、批量导入、性能冒烟和数据库编辑器等 M3 级能力已出现可运行原型。

但从“可研发”走向“可直接接产品”仍需注意两个方面：

- 工程完整性仍弱于能力完整性，尤其是持续集成、边界测试、异常输入覆盖和长期维护文档。
- 若面向真实业务接入，仍需继续补充约束说明、性能上界和运维手册。

## M1 评估

### 公共 V1 基线

状态：已完成。

说明：

- 公共头文件已经形成统一入口和基本类型体系。
- 对外接口已经明确为 Database、Table、Record、Schema、ChangeSet 等核心抽象。
- API 命名与句柄风格已趋于统一。

### SCValue 系统

状态：已完成。

说明：

- SCValue 已覆盖 Null、Int64、Double、Bool、String、RecordId、Enum 等基础类型。
- 已提供类型化访问和错误返回。
- 字符串复制语义和直接访问语义已区分。

### 事务语义

状态：已完成。

说明：

- BeginEdit、Commit、Rollback 的基本模型已经落地。
- 写操作要求进入事务，符合设计基线。
- Undo/Redo 已按事务粒度处理。

### 删除 / tombstone / undo-redo 基线

状态：基本完成。

说明：

- 记录删除、恢复和句柄有效性模型已经有实现基础。
- 仍建议增加更多对“已删除句柄继续访问”的边界测试。

### ChangeSet 基线

状态：基本完成。

说明：

- ChangeSet 与 DataChange 已能描述事务提交结果。
- 变更来源、聚合策略、观察者语义仍有继续细化空间。

### 最小查询基线

状态：基本完成。

说明：

- 已具备按表枚举、按条件查找等最小查询能力。
- 更复杂的组合条件、排序、分页和高阶索引能力仍未展开。

### M1 测试与文档

状态：已完成。

说明：

- 已存在 M1 相关单元测试。
- 示例和基线文档可支撑研发人员理解接口边界。

说明：

- SQLite 适配层已经存在并承担主要持久化职责。
- 数据库启动时的基础对象构造与资源管理链路已经建立。

### 持久化 Schema 管理

状态：基本完成。

说明：

- 表、列、元数据的持久化路径已具备。
- 复杂迁移和跨版本兼容策略仍需继续强化。

### 持久化写入与读取

状态：已完成。

说明：

- SQLite 后端已经支持创建表、创建记录、写入列和读取列。
- 常见查询路径可以工作。

### SQLite 事务对齐

状态：基本完成。

说明：

- 持久化层与内存层的事务模型已经基本对齐。
- 崩溃恢复、一致性边界和异常中断路径建议继续补测。

### Journal 持久化与 Undo/Redo

状态：基本完成。

说明：

- 已有 Journal 持久化和回放思路。
- 对复杂批量事务的回放正确性仍建议继续压测。

### SQLite 测试

状态：已完成。

说明：

- 已有 SQLite 集成测试和性能冒烟。
- 真实规模数据下的长时稳定性仍需后续补充。

### M2 剩余注意项

当前最大的 M2 风险不在于“没有能力”，而在于：

- 一些能力只有主路径验证，缺少系统性的边界输入测试。
- 数据损坏、迁移中断和部分恢复场景需要更明确的运维文档。

## 严格缺口列表

### 高优先级缺口

- 异常输入和边界条件测试仍需继续补强。
- 恢复、迁移和损坏诊断的运维文档还不够完整。
- 数据库编辑器当前更偏研发工具，仍缺少更严格的权限与保护策略。

### 中优先级缺口

- 大规模数据性能基准还不够系统。
- 对观察者、ChangeSet 聚合和复杂事务的行为说明还可继续收紧。
- 文档存在部分历史编码问题，需要持续清理。

### 低优先级缺口

- 示例数量还可以继续增加。
- 若未来支持更多后端，抽象边界还需再次审视。

## 最近审查补充

以下问题来自后续代码审查，属于当前实现的工程性缺口，应继续保留在缺口评估中：

- 导入 `finalize` 的原子收口边界还不够强，失败后可能留下恢复状态和业务数据不同步的风险。
- 数据库编辑器新增列失败时，视图与 schema 的回滚链路仍需要更强的失败收口。
- 新增的 `RemoveColumn()` 语义还偏回滚专用，不应被误读成完整删列能力。
- 当前性能基线对 SQLite 排序路径覆盖还不够完整。
- `MeasureMs` 辅助代码里的断言方式还会影响失败定位可读性。

## 最终判断

当前 Storage 仓库已经不是“设计讨论稿”，而是具备明确产品接入潜力的实现基线。

更准确的判断是：

- 作为研发底座，已经可用。
- 作为直接面向复杂业务的一线基础设施，仍需继续补强测试、恢复和工程说明。

因此，仓库当前阶段适合进入“持续集成 + 面向真实业务样例验证”的下一轮迭代。
