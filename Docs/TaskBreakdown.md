# StableCore Storage 任务拆解

本文把 Roadmap.md 中的 M1 到 M3 进一步拆成可以顺序执行的任务列表。

原则：

- 任务按依赖顺序排列。
- 每个任务尽量有明确产出。
- 尽量避免跨阶段并行推进高耦合事项。

## M1 可开发

### T1. 清理公共 API 骨架

目标：统一公共头文件入口、核心接口和命名边界。

### T2. 完善 SCValue 类型能力

目标：补齐 V1 需要的值类型与类型化读取能力。

### T3. 明确字符串与内存语义

目标：定义直接返回字符串与复制字符串的边界。

### T4. 收紧错误码和错误行为

目标：建立统一错误码集合和错误处理约定。

### T5. 修正内存实现的事务语义

目标：让 BeginEdit、Commit、Rollback 行为与设计基线一致。

### T6. 修正内存实现的删除语义

目标：明确 tombstone、恢复和已删除句柄行为。

### T7. 完善 Journal 聚合逻辑

目标：减少同事务内重复噪声并保持回放正确性。

### T8. 完善 ChangeSet 生成逻辑

目标：让提交后的变更集合可被 UI 与上层逻辑消费。

### T9. 完善 Schema 校验

目标：约束非法列定义、重名、类型不匹配等问题。

### T10. 完善最小查询能力

目标：提供表枚举、等值查找与基础查询入口。

### T11. 建立单元测试工程

目标：让核心语义具备防回归能力。

### T12. 补 M1 核心测试

目标：覆盖事务、删除、Schema、SCValue、查询主路径。

### T13. 补最小示例和开发文档

目标：让接入者能快速理解并使用仓库。

## M2 可持久化

### T14. 设计 SQLite 物理表结构

目标：定义元数据、记录、Journal、索引等基本表结构。

### T15. 编写 SQLite 适配层

目标：建立与核心接口对齐的持久化实现。

### T16. 实现数据库初始化与元数据加载

目标：数据库打开时完成必要的初始化与状态构建。

### T17. 实现 SQLite 版 Schema 管理

目标：支持持久化表与列定义管理。

### T18. 实现 SQLite 版记录创建与列写入

目标：打通基本写路径。

### T19. 实现 SQLite 版事务对象

目标：让持久化层事务与核心语义保持一致。

### T20. 实现 SQLite 版读取与查询

目标：支撑事实读取与典型查询主路径。

### T21. 实现 Journal 持久化

目标：持久化事务历史并为 Undo/Redo 提供基础。

### T22. 实现 SQLite 版 Undo/Redo

目标：让持久化后端具备可回滚能力。

### T23. 建立 SQLite 集成测试

目标：验证启动、读写、事务、恢复和查询主路径。

## M3 可接产品

### T24. 跑通楼层关系基础

目标：建立典型业务关系的最小验证样例。

### T25. 建立计算列元模型

目标：定义计算列结构、依赖与求值入口。

### T26. 实现最小表达式求值器

目标：支撑基础计算列表达式。

### T27. 实现 ruleId 注册机制

目标：让业务规则可由外部注册。

### T28. 实现计算列缓存与失效

目标：降低重复求值成本，并在依赖变化时正确失效。

### T29. 补批量写入与导入能力

目标：支撑产品导入与批量修改场景。

### T30. 补性能与索引优化

目标：建立最小性能基线并优化典型查询路径。

### T31. 建数据库升级与迁移方案

目标：让数据库格式演进可控。

### T32. 建崩溃恢复和诊断能力

目标：提高启动异常、恢复失败与数据问题的定位能力。

### T33. 建产品接入示例

目标：为上层团队提供接入样例和研发验证工具。

## 推荐执行顺序

推荐顺序：

1. T1-T13
2. T14-T23
3. T24-T33

在每个大阶段内，优先先收敛接口和测试，再继续叠加能力。

## 建议下一步

当前更适合继续做的不是新增更多大功能，而是：

- 持续补测试。
- 清理文档和术语一致性。
- 通过真实业务样例验证现有能力边界。
