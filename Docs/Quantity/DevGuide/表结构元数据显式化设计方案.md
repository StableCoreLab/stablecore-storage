# 表结构元数据显式化设计方案（最终修订版）

---

# 一、背景

当前 DatabaseEditor 在生成表结构代码时，主要依赖“当前表列定义”进行推导：

* 表名可以直接读取。
* 列定义可以直接读取。
* `indexed` 可以作为历史索引线索。
* `Id` 列可以作为主键习惯用法进行推测。

这种方式能满足简单场景，但不满足“存储层显式表达完整表结构事实”的要求。

主要问题包括：

1. 主键只能靠约定推导，不能作为一等 schema 事实。
2. 索引只能靠列级标记推导，不能表达命名索引、联合索引、列顺序、索引来源。
3. 表描述缺少正式存储位置。
4. 历史数据无法从存储层准确恢复完整表结构定义。
5. DatabaseEditor 与底层 SQLite 存储实现容易耦合。
6. 当前 schema 仍然偏“表附属信息”，缺少完整 schema graph 表达能力。

本方案目标是将表描述、主键、索引、约束从“推断结果”提升为“显式元数据”，并为后续 schema diff、migration、导出、回放校验打基础。

---

# 二、设计目标

## 2.1 完整 schema snapshot

让 DatabaseEditor 能直接读取完整 schema snapshot，并生成类似如下代码：

```cpp
SC_SCHEMA_TABLE(Element)
{
    Table("Element")
        .Description("构件主表")
        .PrimaryKey("Id")
        .Column("Id", SCType::Int64).NotNull().Description("构件唯一 ID")
        .Column("FloorId", SCType::Int64)
            .NotNull()
            .Ref("Floor", "Id")
            .Description("所属楼层")
        .Column("ElementType", SCType::Int32)
            .NotNull()
            .Description("构件类型")
        .Column("Name", SCType::String)
            .NotNull()
            .Description("构件名称")
        .Index("idx_Element_FloorId")
            .Columns("FloorId");
}
```

---

## 2.2 schema 一等事实化

让以下内容成为存储层的一等 schema 事实：

* 表描述
* 主键
* 索引
* 约束

---

## 2.3 逻辑与物理分离

明确区分：

* 逻辑 schema
* SQLite 物理实现
* 历史兼容信息
* 显式定义
* 历史 hint
* 迁移推断结果

---

## 2.4 保持现有运行时语义稳定

保持以下语义不变：

* `open()` 不写数据
* 读取不写入
* 升级必须显式执行
* rollback 必须完整恢复
* 删除记录不能复活
* `RecordId` 仍保持系统 identity 语义

### 2.4.1 `CreateFileDatabase()` 门面语义

`CreateFileDatabase()` 是对外的用户门面函数。

它的职责不是暴露版本差异，而是保证调用方拿到的是**最新文件格式**的数据库句柄。

因此：

* 调用方通过 `CreateFileDatabase()` 打开数据库后，不需要再区分旧版本或新版本。
* 若文件版本低于当前支持版本，门面层会在返回句柄前完成升级。
* 若文件已经是最新版本，门面层直接返回可用句柄。
* 这个门面行为属于 Storage 内部实现细节，对外不要求显式处理版本分支。
* 该门面允许在打开阶段发生受控升级，但升级仍然必须经过明确的版本判断与迁移流程，不能隐式跳过。

---

## 2.5 为后续能力预留基础

为以下能力建立基础：

* schema diff
* migration script 生成
* schema compare
* schema export
* replay validation
* schema graph analysis

# 三、非目标

本轮不实现：

1. SQLite 物理索引细节暴露。
2. RecordId 身份模型重构。
3. 权限、租户、协作控制。
4. ORM。
5. 自动 schema diff 改库。
6. Query Model 重构。
7. Check 约束执行器。
8. ForeignKey enforcement。
9. 自动 schema merge。

# 四、核心设计原则

## 4.1 显式优先

能显式保存的 schema 事实，不再依赖推断。

---

## 4.2 逻辑与物理分离

逻辑 schema：

* 可导出
* 可比较
* 可迁移
* 可反射

SQLite index / pragma / table：

仅属于后端实现。

---

## 4.3 schema graph 优先

schema 不应仅是“表附属信息”。

应表达为：

```text
Schema
 ├─ Tables
 │   ├─ Columns
 │   ├─ Constraints
 │   ├─ Indexes
 │   └─ Metadata
 └─ Version
```

---

## 4.4 约束统一建模

主键、唯一约束、外键、检查约束统一使用 Constraint 模型。

主键不是特殊对象，而是：

```text
ConstraintKind::PrimaryKey
```

---

## 4.5 历史兼容不伪造事实

历史推断结果：

* 可以保存
* 必须标记来源
* 不能冒充显式 schema

---

## 4.6 迁移可回滚

迁移：

* 必须在事务内完成
* 提交前不能替换 live schema snapshot
* 失败后必须保持旧 snapshot

---

## 4.7 工具层不直连 SQLite

DatabaseEditor 不直接读取 SQLite metadata。

统一通过：

```text
ISCSchema::GetSchemaSnapshot(...)
```

访问 schema。

---

# 五、目标架构

```text
Storage Runtime
    ↓
Logical Schema Layer
    ↓
Snapshot Entry (`ISCSchema::GetSchemaSnapshot(...)`)
    ↓
Migration Layer
    ↓
Tooling / DatabaseEditor
```

---

## 5.1 各层职责

| 层级                      | 职责                    |
| ----------------------- | --------------------- |
| Storage Runtime         | 数据读写、事务、记录身份、日志、回放    |
| Logical Schema Layer    | 表、列、约束、索引等逻辑 schema   |
| Snapshot Entry          | `ISCSchema::GetSchemaSnapshot(...)`、schema snapshot、导出、渲染 |
| Migration Layer         | schema 升级、回滚          |
| DatabaseEditor          | UI 编辑、结构展示、代码生成       |

# 六、目标数据模型

---

# 6.1 SCSchemaSnapshot

```cpp
struct SCSchemaSnapshot
{
    int schemaVersion{0};

    std::vector<SCTableSchemaSnapshot> tables;
};
```

语义：

* 表示完整逻辑 schema。
* immutable value object。
* 不持有 live schema。
* 不 lazy load。
* 不允许隐式变化。

---

# 6.2 SCTableSchemaSnapshot

```cpp
struct SCTableSchemaSnapshot
{
    SCTableDef table;

    std::vector<SCColumnDef> columns;

    std::vector<SCConstraintDef> constraints;

    std::vector<SCIndexDef> indexes;
};
```

---

# 6.3 SCTableDef

```cpp
struct SCTableDef
{
    std::wstring name;

    std::wstring description;
};
```

---

# 6.4 SCColumnDef

现有 `SCColumnDef` 保持。

继续承载：

* 列名
* 类型
* 可空性
* 可编辑性
* 描述
* referenceTable
* indexed_flag

但必须明确：

```text
indexed_flag
```

不再代表完整逻辑索引。

仅表示：

* 历史兼容信号
* 物理索引 hint
* migration 输入之一

---

# 6.5 SCSchemaSourceKind

```cpp
enum class SCSchemaSourceKind
{
    Explicit,
    MigratedConvention,
    LegacyHint,
    Unknown
};
```

说明：

| 类型                 | 含义        |
| ------------------ | --------- |
| Explicit           | 显式 schema |
| MigratedConvention | 历史约定迁移    |
| LegacyHint         | 历史 hint   |
| Unknown            | 未知来源      |

该枚举用于记录 schema 元数据的来源和迁移可追溯性，不能仅靠 `bool` 标志替代。
它同时支撑：

* 导出时是否显示 legacy hint
* 迁移后 provenance 标记
* 未来 schema diff / compare 中的来源判断
* 表结构快照中的显式事实与历史兼容事实区分

不再使用：

```cpp
bool explicitDefined;
bool inferred;
```

避免状态冲突。

---

# 6.6 SCConstraintKind

```cpp
enum class SCConstraintKind
{
    PrimaryKey,
    Unique,
    ForeignKey,
    Check
};
```

---

# 6.7 SCConstraintDef

```cpp
struct SCConstraintDef
{
    SCConstraintKind kind;

    std::wstring name;

    std::vector<std::wstring> columns;

    SCSchemaSourceKind sourceKind
        {SCSchemaSourceKind::Explicit};

    std::wstring referencedTable;

    std::vector<std::wstring> referencedColumns;

    std::wstring checkExpression;
};
```

---

## 6.7.1 Constraint 合法性规则

### PrimaryKey

必须：

* 有 columns

必须为空：

* referencedTable
* referencedColumns
* checkExpression

---

### Unique

必须：

* 有 columns

必须为空：

* referencedTable
* referencedColumns
* checkExpression

---

### ForeignKey

必须：

* 有 columns
* 有 referencedTable

可为空：

* referencedColumns

为空表示：

```text
引用目标表默认主键
```

必须为空：

* checkExpression

---

### Check

必须：

* checkExpression 非空

可为空：

* columns

必须为空：

* referencedTable
* referencedColumns

---

# 6.8 SCIndexColumnDef

```cpp
struct SCIndexColumnDef
{
    std::wstring columnName;

    bool descending{false};
};
```

---

# 6.9 SCIndexDef

```cpp
struct SCIndexDef
{
    std::wstring name;

    std::vector<SCIndexColumnDef> columns;

    SCSchemaSourceKind sourceKind
        {SCSchemaSourceKind::Explicit};
};
```

说明：

* Index 不表达唯一性。
* 唯一性使用：

```text

```

表达。

# 九、逻辑索引与 SQLite 物理索引

---

# 9.1 逻辑索引

逻辑索引：

* 属于 schema
* 可导出
* 可比较
* 可迁移

---

# 9.2 SQLite 物理索引

SQLite index：

* 属于后端实现
* 不作为 schema 事实
* 不直接导出

---

# 9.3 生成方向

推荐：

```text
Logical Index
    ↓
SQLite Physical Index
```

不推荐：

```text
SQLite Physical Index
    ↓
Logical Index
```

---

# 9.4 本轮物理索引策略

本轮 migration：

* 只迁移逻辑 schema metadata
* 不强制创建新的 SQLite index

是否创建物理 index：

由现有 Storage Runtime 策略决定。

---

# 十、RecordId 与业务主键

必须明确：

1. RecordId 是系统 identity。
2. PrimaryKey 是业务 schema。
3. 两者可以相同。
4. PrimaryKey 不替代 RecordId。
5. replay / undo / redo 仍基于 RecordId。

---

# 十一、历史数据升级方案

---

# 11.1 schema version

当前最新版本：

```text
v2
```

本轮升级后：

```text
v3
```

---

# 11.2 schema version 存储

如果已有统一元信息表：

继续沿用。

否则建议：

```text
storage_meta(key, value)
```

其中：

```text
key = 'schema_version'
```

---

# 11.3 升级原则

1. open() 不自动升级
2. 必须显式升级
3. 单事务
4. rollback 完整恢复
5. 提交前不替换 snapshot
6. 升级后重新加载 snapshot

---

# 11.4 升级幂等规则

```text
currentVersion == 3
    → 直接返回 SC_OK

currentVersion > 3
    → 返回 SC_E_SCHEMA_VERSION_UNSUPPORTED

currentVersion < 2
    → 按 migration chain 逐步升级
```

---

# 11.5 v2 → v3 升级

## 步骤 1

开启事务。

---

## 步骤 2

创建：

* schema_tables
* schema_constraints
* schema_constraint_columns
* schema_indexes
* schema_index_columns

---

## 步骤 3

扫描旧表。

写入：

```text
schema_tables
```

description 默认为空。

---

## 步骤 4

保留 schema_columns。

兼容扩展：

* ordinal
* description

等字段。

---

## 步骤 5

迁移主键。

### 已有明确主键

迁移为：

```text
sourceKind = Explicit
```

---

### 单列 Id 约定

迁移为：

```text
ConstraintKind::PrimaryKey
sourceKind = MigratedConvention
```

---

### 无法确认

不创建主键。

---

## 步骤 6

迁移 indexed_flag。

迁移为：

```text
sourceKind = LegacyHint
```

索引名：

```text
idx_<Table>_<Column>
```

---

## 步骤 7

最后更新 schema version。

---

## 步骤 8

提交事务。

---

## 步骤 9

重新加载 schema snapshot。

---

# 11.6 rollback 要求

失败后必须：

1. rollback
2. 保持旧 snapshot
3. schema version 不抬升
4. 不留下半成品 metadata

# 十二、schema cache 与 migration

必须明确：

1. migration 前 snapshot 不变
2. migration 使用临时 context
3. commit 成功后替换 snapshot
4. rollback 后继续使用旧 snapshot

不得出现：

```text
memory schema
≠
database schema
```

# 十三、DatabaseEditor 改造

---

# 13.1 导出流程

```text
ISCSchema::GetSchemaSnapshot(...)
    ↓
SCSchemaSnapshot
    ↓
Schema Code Generator
    ↓
SC_SCHEMA_TABLE(...)
```

---

# 13.2 导出规则

1. Description 来源于 table.description
2. Column 来源于 SCColumnDef
3. PrimaryKey 来源于 ConstraintKind::PrimaryKey
4. Index 来源于 SCIndexDef
5. Ref(...) 来源于 referenceTable
6. 不再从 Id 推断主键
7. 不再从 indexed_flag 自动生成 Index

---

# 13.3 referenceTable 与 ForeignKey 的关系

本轮：

```text
Column(...).Ref(...)
```

只读取：

```text
SCColumnDef::referenceTable
```

ForeignKey constraint：

* 仅进入 snapshot
* 不参与默认导出

---

## 13.3.1 冲突规则

如果：

```text
referenceTable
```

与：

```text
ForeignKey constraint
```

冲突：

则：

1. DatabaseEditor 仍以 referenceTable 为导出来源
2. 输出诊断信息
3. 不自动修复
4. 不自动覆盖

---

# 13.4 LegacyHint 导出策略

LegacyHint：

默认不导出。

DatabaseEditor 必须提示：

```text
发现历史 indexed 标记，可选择导出为显式索引。
```

用户确认导出后：

```text
Index(...)
```

视为 Explicit schema。

migration 成功前：
