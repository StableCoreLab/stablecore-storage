# 全量文档架构审查修改记录

> 依据文档：[`全量文档架构审查.md`](全量文档架构审查.md)
> 记录目的：把本次审查结论收敛为可执行的统一修订口径，便于后续逐份同步 Architecture / Design 文档。

## 1. 修改目标

本次修改不是单纯补充意见，而是把原先分散在各份文档里的问题，统一收敛成一套必须遵守的架构边界。

1. 工程模型唯一数据源。
2. 运行时上下文唯一主权。
3. 命令提交唯一边界。
4. 显示链路唯一分层。
5. 命名与 target 唯一口径。
6. Extern 唯一消费边界。
7. 资产生命周期唯一规则。
8. 审查、台账、待细化内容之间必须形成闭环。

## 2. 已确认的问题边界

### 2.1 状态与上下文

- `Project Model` 是工程模型唯一数据源。
- `ModelingContext` 是运行时唯一真值源。
- `ViewState / SelectionSet / DisplayFilter / CommandContext / EditingContext / DisplayContext / ExecutionContext / WorkContext` 都只能是投影、派生或临时快照，不能并列为真值源。

### 2.2 命令与回放

- `CommandSubmission / CommitId / SubmissionId` 统一命令调用、Storage 事务、Undo、Replay、Assert 的提交边界。
- 普通命令调用与正式提交必须分离。
- 后续记录都必须挂在同一个提交单元上。

### 2.3 显示与交互

- 显示链路固定为 `Presenter -> Adapter -> Backend`。
- Presenter 只负责批次装配与增量失效。
- Adapter 只负责语义到显示数据的翻译。
- Backend 只负责渲染执行，不再保存业务真值。

### 2.4 命名与构建

- 代码文件、代码标识、公开命名空间和公开 target 统一采用 `SC/ISC` 品牌前缀。
- `sc_quantity_*` 不再作为公开 target 的主口径。
- 概念术语可以保留领域原名，但不能与代码层命名规则混用。

### 2.5 外部依赖与资产

- 主仓库只消费 `..\Extern\stablecore`、`..\Extern\thirdparty-ogre-next`、`..\Extern\thirdparty-saribbon` 的安装结果。
- 不直接依赖外部库源码树。
- `Journal / Snapshot / Replay / Crash Recovery / Cleanup` 必须共享同一套资产生命周期规则。

### 2.6 治理闭环

- `文档审查结论`、`问题记录台账`、`文档待细化内容` 必须共享编号和状态。
- 处理结论必须反向回写到源文档。
- 不允许只保留审查意见而没有修订结果。

## 4. 本次修改后续顺序

1. 先改基础层文档，锁定目录、命名、target 和 Extern 边界。
2. 再改命令与回放文档，锁定提交边界和参数 schema。
3. 再改统一建模视图专题，锁定状态主权和显示链路。
4. 最后改治理类文档，锁定归档、清理、台账和测试闭环。

## 5. 使用说明

本记录用于后续同步修订时对齐口径，不替代逐份文档修改。

如果后续要把这些结论直接落到 `Docs/Architecture/` 或 `Docs/Design/` 下的具体文档，应严格按本记录的顺序推进，避免再次出现状态分叉、命名分叉或治理闭环缺失。
