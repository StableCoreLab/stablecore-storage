# 全量文档架构审查

> 审查对象：`Docs/文档索引.md` 中提到的全部文档（已吸收原 `Docs/文档索引2.md` 的内容）
> 审查目标：逐一架构审查 + 全局收敛
> 审查结论：当前文档体系方向正确，但还没有完成“唯一状态源 + 唯一上下文源 + 唯一显示语义源”的最终收敛

## 审查口径

1. 每份文档都按 `批次评估 / 关键问题 / 内部一致性问题 / 修改建议 / 结论` 五项给出结果。
2. 重点判断标准不是“文档写得是否完整”，而是“写完后能不能直接指导实现，并且不会和其他文档打架”。
3. 对统一建模视图相关文档，优先检查 `ModelingContext / ViewState / SelectionSet / DisplayFilter / CommandContext / Presenter / Backend` 是否收敛到同一主线。
4. 对命令与回放相关文档，优先检查 `CommandManager / Journal / Snapshot / Replay / Undo / Storage` 是否共用同一提交边界。
5. 原 `Docs/文档索引2.md` 已并入主索引；A-01 ~ A-24 只保留一份正式入口，不再双索引维护。
6. `Docs/文档索引.md` 第 5 节列出的非索引治理文档虽然不属于当前有效设计资产，但被索引文件明确提到，且会影响文档治理闭环，因此纳入本次统一审查。

## 一、架构基础层

### `Docs/Architecture/目录定义.md`

- 批次评估：整体可落地，目录职责清楚，已经具备主仓库骨架。
- 关键问题：目录层次虽然已经把 `..\Extern\stablecore`、`..\Extern\thirdparty-ogre-next`、`..\Extern\thirdparty-saribbon` 纳入顶层约束，但还没有把“主仓库只消费安装结果、不直接依赖源码树”的规则彻底写成不可违背的边界，后果是后续仍可能把外部库源码当成正式构建来源。
- 内部一致性问题：与 `模块职责与依赖约束定义`、`CMake target 划分方案` 的分层口径基本一致，但对 `Extern` 依赖根、安装产物目录和模块消费方式仍偏描述性。
- 修改建议：补一张“目录 - 模块 - target - Extern 安装根”的对应表，并明确资源、脚本、测试、生成物和第三方安装产物的落盘规则。
- 结论：可作为仓库布局基线，但必须补 Extern 消费边界和生成物边界。

### `Docs/Architecture/模块职责与依赖约束定义.md`

- 批次评估：方向正确，已经能约束最常见的反向依赖。
- 关键问题：当前更像职责宣言，不像可执行的依赖矩阵；如果没有明确允许/禁止列表，后果是 UI、Command、Render、Project 之间仍会通过“临时便利”互相直连。
- 内部一致性问题：与 `目录定义`、`CMake target 划分方案` 需要共享同一张模块边界图，否则目录和构建图会分别演进。
- 修改建议：按模块列出“允许依赖、禁止依赖、只允许通过接口依赖”的三列表，并把 `Storage`、`Geometry`、`Render`、`UI` 的方向写死。
- 结论：可落地，但必须从原则文档升级为约束表。

### `Docs/Architecture/命名规范.md`

- 批次评估：可作为代码规范起点，但目前偏宽松。
- 关键问题：最新规范已经要求代码文件、代码标识和公开 target 统一携带 `SC/ISC` 品牌前缀，但当前文档审查里还保留了“通用语义可以不带前缀”的旧口径，后果是实现阶段很容易把品牌约束弱化成可选项。
- 内部一致性问题：与命令、视图、回放专题中的类型命名口径需要区分“概念术语”和“代码标识”，否则会出现“文档概念无前缀、代码实现有前缀”的双轨解释。
- 修改建议：把命名规范明确拆成两层，概念名可以保持领域语义原名，代码层的类、接口、文件、公开命名空间和 target 必须统一带 `SC/ISC` 前缀；同时给出允许保留无前缀的纯概念术语白名单。
- 结论：可用，但必须把品牌前缀从建议项升级为实现硬约束。

### `Docs/Architecture/CMake target 划分方案.md`

- 批次评估：可执行性强，是当前最接近工程落地的基础文档之一。
- 关键问题：当前 target 划分仍使用 `sc_quantity_*` 这一套小写品牌表达，与最新命名规范要求的 `SC/ISC` 品牌前缀不完全一致，后果是代码文件、代码标识和构建产物会出现品牌口径分裂。
- 内部一致性问题：与 `目录定义`、`模块职责与依赖约束定义`、`命名规范` 的术语需要完全对齐，否则构建层和设计层会变成两套语言；同时，`Extern` 依赖接入规则已经进入目录定义，但 target 文档还没有把“只消费 `..\Extern\` 安装产物”写成统一入口。
- 修改建议：把 target 划分限定为“支撑层 / 核心业务层 / 表现流程层 / 应用层”，并统一采用 `SC` / `ISC` 品牌前缀的 target 命名方案，至少在公开 target 名称里保持和代码层一致的品牌识别。
- 结论：可落地，但要先收敛 target 语义和品牌前缀，再进入拆分。

**本批结论：** 架构基础层可以作为工程骨架使用，但必须先把目录、模块、target、命名、Extern 安装依赖统一成一张硬约束矩阵；否则实现阶段会出现构建边界和代码边界不一致，后续模块拆分会反复返工。

## 二、命令与主界面基础层

### `Docs/Architecture/命令系统总体设计.md`

- 批次评估：总体方向正确，命令链路已经搭起来了。
- 关键问题：文档把 `EditingContext / DisplayContext / ExecutionContext` 并列成三类上下文，但没有明确它们与统一视图专题里的 `ModelingContext` 是主从还是并列，后果是命令可见、可用、可执行的判断会分裂。
- 内部一致性问题：与 `命令提交边界设计`、`命令参数与序列化规范设计`、`A-04`、`A-10`、`A-15` 都共享上下文概念，但没有统一总主线。
- 修改建议：把 `ModelingContext` 作为运行时主上下文，把 `EditingContext / DisplayContext / ExecutionContext` 定义为投影或派生视图，而不是独立真值源。
- 结论：可落地，但必须先统一上下文主权。

### `Docs/Architecture/主界面设计方案.md`

- 批次评估：UI 信息结构完整，首版工作台可按此搭建。
- 关键问题：文档描述了布局，但没有把主界面和统一建模视图宿主、命令状态刷新、录制回放入口放到同一条运行链路里，后果是主界面会成为“壳很好看，状态却分散”的装配体。
- 内部一致性问题：与 `启动前界面与录制回放策略设计`、`命令系统总体设计`、`A-01` 的宿主职责有重叠，但没有明确谁负责状态切换。
- 修改建议：把主界面明确分成“启动前壳、工程壳、建模壳”三态，并指定每一态由谁持有状态与切换逻辑。
- 结论：可落地，但需要与宿主和运行策略文档合并视角。

### `Docs/Architecture/启动前界面与录制回放策略设计.md`

- 批次评估：入口和策略都考虑到了，覆盖面较全。
- 关键问题：文档把版本行为、启动首页、录制开关、工程打开后的录制延续揉在一起，缺少单独的运行策略对象，后果是发布版和内部版行为会越来越像“靠 if 分支堆出来的规则”。
- 内部一致性问题：与 `工程文件与日志回放关系设计`、`工程生命周期命令设计`、`回放执行策略设计` 的边界需要统一，否则启动页会替代命令层做决策。
- 修改建议：把策略收敛到 `SCAppRuntimePolicy` 之类的单一对象，启动页只负责入口呈现，不负责业务判定。
- 结论：可落地，但必须把“显示入口”和“录制判定”拆开。

### `Docs/Architecture/工程文件与日志回放关系设计.md`

- 批次评估：主从关系方向正确，尤其强调工程文件是正式状态主载体，这是必须的。
- 关键问题：弱绑定策略是对的，但目录结构、迁移矩阵、崩溃恢复优先级还不够落地，后果是回放资产、工程副本、临时恢复包会互相污染。
- 内部一致性问题：与 `JournalReplay数据模型设计`、`Replay 快照策略设计`、`日志归档与清理策略设计` 必须共享同一套资产生命周期，否则清理和恢复会冲突。
- 修改建议：明确工程目录、回放目录、快照目录、归档目录的固定关系，并给出 `Save / Save As / Copy / Package / Export / Crash Recovery` 的行为矩阵。
- 结论：可落地，但要补目录级真值表。

**本批结论：** 命令入口、主界面和启动策略已经能指导首版工作台，但“界面状态”和“运行策略”仍混在一起；如果不拆出统一运行策略对象，发布版、内部版、录制回放和工程生命周期会在 UI 层形成复杂分支。

## 三、命令 / 回放 / 上下文层

### `Docs/Architecture/命令提交边界设计.md`

- 批次评估：这是命令系统里最关键的收敛文档之一。
- 关键问题：提交边界定义了 UI 动作、业务意图、Storage 事务、Replay 记录四种口径，但尚未明确“一次提交是否必须原子一致”，后果是 Undo、Replay、ChangeSet 可能各自认为自己是一个边界。
- 内部一致性问题：与 `CommandResult`、`JournalReplay`、`工程生命周期命令设计`、`A-04` 共享边界概念，但没有统一提交 ID。
- 修改建议：定义唯一的 `CommitId / SubmissionId`，把“命令调用”和“正式提交”分开，所有后续记录都挂到同一个提交单元上。
- 结论：必须收敛后再实现，否则后面一定返工。

### `Docs/Architecture/UI命令装配与菜单配置体系设计.md`

- 批次评估：装配思路正确，已经从“代码里硬写 UI”转向“命令驱动 UI”。
- 关键问题：混合模式里缺少统一的运行时展示状态服务，后果是命令元数据、菜单位置、显隐策略、快捷键会分散在不同位置维护。
- 内部一致性问题：与 `快捷键体系设计`、`命令系统总体设计`、`A-24` 共享“展示状态”概念，但没有统一 `CommandMetadata`。
- 修改建议：把“命令定义、UI 装配、运行时显隐、动作映射”拆成四层，UI 只消费元数据和状态服务。
- 结论：可落地，但必须补统一的展示状态计算器。

### `Docs/Architecture/命令参数与序列化规范设计.md`

- 批次评估：必要且正确，是命令可回放的基础。
- 关键问题：参数独立于 UI 的方向对，但还没细到“每个命令族的 schema 和对象引用规则”，后果是旧回放、自动化测试和跨版本兼容会依赖字符串猜测。
- 内部一致性问题：与 `JournalReplay数据模型设计`、`自动化测试断言模型设计`、`回放执行策略设计` 必须共用参数版本号和对象引用规则。
- 修改建议：按命令族定义参数 schema，所有引用统一走稳定 ID，不允许把控件状态或临时指针写入序列化层。
- 结论：可落地，但要补版本迁移和引用稳定性样例。

### `Docs/Architecture/JournalReplay数据模型设计.md`

- 批次评估：层次划分清楚，已经具备落库与测试的基础。
- 关键问题：`ReplaySession / CommandJournal / AssertJournal / Snapshot` 的分层是对的，但还缺少和工程目录、Storage 事务、命令提交边界的闭环，后果是数据模型会很完整，但执行时不知道该挂在哪个生命周期上。
- 内部一致性问题：与 `工程文件与日志回放关系设计`、`Replay 快照策略设计`、`自动化测试断言模型设计` 必须共用同一套会话 ID 和资产目录。
- 修改建议：补“会话 - 日志 - 快照 - 工程 - Storage”关联图，并明确哪些数据可删除、哪些必须保留。
- 结论：骨架完整，但还不是最终落库版。

### `Docs/Architecture/NavigationCategory设计.md`

- 批次评估：有价值，但风险也很高。
- 关键问题：把 `NavigationCategory` 定位成“工作语义入口”是对的，但文档同时赋予它显示、命令、属性面板、拾取规则的驱动能力，后果是它会变成第二个上下文根。
- 内部一致性问题：与 `DisplayFilterService`、`命令系统总体设计`、`A-10`、`A-15`、`选择与拾取规则设计` 都在争夺工作语义解释权。
- 修改建议：把它降级为导航语义定义或导航到显示上下文的映射源，不要让它直接承载所有运行时决策。
- 结论：可用，但必须防止语义膨胀。

### `Docs/Architecture/事件与通知模型设计.md`

- 批次评估：必要，且方向正确。
- 关键问题：事件系统只写了“谁发谁收”，但没有把事务后通知、批量聚合、循环防护和事件作用域钉死，后果是 UI 刷新和状态同步会逐渐变成广播风暴。
- 内部一致性问题：与 `命令提交边界设计`、`A-13`、`A-15`、`DisplayFilterService`、`自动化测试断言模型设计` 都需要同一套通知粒度。
- 修改建议：明确领域事件、上下文事件、UI 通知事件三层，并要求所有通知都在提交之后或批量结束之后发出。
- 结论：可落地，但要先控住粒度和作用域。

**本批结论：** 命令、回放和上下文层是当前最高优先级收敛点。必须先定义唯一 `CommandSubmission / CommitId`，并把 Storage 事务、Undo、Replay、Assert、事件通知挂到同一个提交单元；否则回放、撤销和诊断会各自形成边界。

## 四、统一建模视图专题

### `Docs/Architecture/A-01统一建模视图系统总体架构设计.md`

- 批次评估：方向正确，已经能作为统一建模视图的骨架。
- 关键问题：`ModelingContext / ViewState / SelectionSet / DisplayFilter` 已经进入核心主线，但文档还没有把它们的主从关系钉死，后果是后续切换、回放、属性联动会重复维护状态。
- 内部一致性问题：与 `A-02 / A-03 / A-04` 对同一语义有重复定义。
- 修改建议：把 `ModelingContext` 定义为唯一运行时真值源，其他对象都只能是投影或派生。
- 结论：可落地，但必须继续收敛主状态。

### `Docs/Architecture/A-02统一场景模型与视图状态设计.md`

- 批次评估：这是统一语义层的核心文档，质量高。
- 关键问题：`SelectionSet / DisplayFilter / ViewState` 的边界定义得很接近，但还没有明确哪些字段是可恢复状态、哪些字段只是临时交互状态，后果是快照、回放和切换会把不该持久化的东西一并带走。
- 内部一致性问题：与 `A-01`、`A-03`、`A-12`、`A-15` 多处重复定义选择和状态。
- 修改建议：把视图状态拆成“可恢复视图语义”和“瞬时交互语义”，并将 `SceneElementProxy` 明确为代理而非第二真值源。

### `Docs/Architecture/A-03二三维无感切换设计.md`

- 批次评估：方向正确，是 2D/3D 统一体验的关键文档。

- 修改建议：切换只迁移投影语义，不复制选择和过滤副本；后端只负责可视表现，不负责业务状态。
- 结论：可落地，但必须先统一模式主键。

### `Docs/Architecture/A-04建模命令接入统一视图系统设计.md`

- 批次评估：命令接入路径明确，方向正确。

- 内部一致性问题：与 `命令系统总体设计`、`命令提交边界设计`、`A-08` 重复维护交互语义。
- 修改建议：收缩 `CommandContext`，只保留命令执行必需的最小集合，其余通过服务接口和投影上下文获取。
- 结论：可落地，但不能继续膨胀。

### `Docs/Architecture/A-05构件显示适配体系设计.md`

- 关键问题：`DisplayStyle` 把高亮、选择、只读、透明等语义混在同一个状态里，后果是样式会反向成为状态源，显示规则难以统一。

- 结论：可落地，但必须拆分语义层和渲染层。

### `Docs/Architecture/A-06二维后端QWidget加QPainter设计.md`

- 批次评估：技术路线稳妥，适合当前阶段。
- 关键问题：`SC2DViewportState` 和 `ViewState` 并存，会让 2D 后端慢慢拥有自己的主状态，后果是切换、回放和选择结果会在 2D 里另起一套。
- 内部一致性问题：与 `A-01`、`A-02`、`A-03`、`A-11` 的状态投影关系不一致。
- 修改建议：2D 后端只消费 `ModelingContext` 的投影，不允许保存独立视图主状态；命中结果也必须回到统一命中模型。
- 结论：可落地，但不能让 2D 后端持有独立真值。

### `Docs/Architecture/A-07三维后端OgreNext集成设计.md`

- 批次评估：集成路径合理。
- 关键问题：`SC3DViewState`、`void*` / 后端私有句柄的使用方式，会让 3D 后端很容易形成黑箱状态，后果是生命周期、失效和调试都难以定位。
- 内部一致性问题：与 `A-06`、`A-13`、`A-14` 的句柄和缓存口径要统一。
- 修改建议：用明确的后端句柄封装替代裸指针，句柄只在后端内生效，向上只暴露统一 ID 和可撤销操作。
- 结论：可落地，但必须收敛句柄模型。

### `Docs/Architecture/A-08建模交互模型设计.md`

- 批次评估：很关键，交互模型是统一视图是否可用的核心。
- 关键问题：交互上下文与命令上下文出现了大量重叠字段，后果是 hover、active、selection、replayMode 会在多个层里互相覆盖。
- 内部一致性问题：与 `A-04`、`A-12`、`A-15`、`选择与拾取规则设计` 在交互语义上重复定义。
- 修改建议：把交互层定位成事件路由和短暂状态投影层，不要承载正式选择或过滤副本。
- 结论：可落地，但要先把交互和正式语义分开。

### `Docs/Architecture/A-09捕捉与参考查询系统设计.md`

- 批次评估：方向正确，属于核心交互能力。
- 关键问题：捕捉结果、参考查询、优先级规则如果各自定义输出结构，后果是命中行为在 2D/3D/预览里无法统一。
- 内部一致性问题：与 `A-12`、`选择与拾取规则设计`、`A-16` 的预览参考对象需要共享同一套命中语义。
- 修改建议：把捕捉统一收敛为查询服务，输出同一个结果模型，再由交互层做优先级决策。
- 结论：可落地，但要统一结果出口。

### `Docs/Architecture/A-10工作平面楼层视图上下文设计.md`

- 批次评估：场景语义划分有价值。
- 关键问题：`SCWorkContext` 如果成为独立主上下文，后果是楼层、工作平面、视图和命令各自维护自己的当前语义，切换时非常容易漂移。
- 内部一致性问题：与 `命令系统总体设计`、`A-01`、`A-15` 都在争夺当前工作语义的解释权。
- 修改建议：把工作平面和楼层视为 `ModelingContext` 的投影字段，`semanticName` 只能做辅助标记，不能做真值键。
- 结论：可落地，但不能独立成主上下文。

### `Docs/Architecture/A-11构件建模数据与显示数据映射设计.md`

- 批次评估：这是连接工程模型和显示系统的关键层。
- 关键问题：`SCModelObjectRef / SceneElementProxy / SCSceneProxyData` 的关系没有完全钉死，后果是对象、代理、显示数据会形成两套近义模型。
- 内部一致性问题：与 `A-02` 的 `SceneElementProxy`、`A-13` 的句柄映射、`A-14` 的缓存键口径重叠。
- 修改建议：只保留一层代理模型，显示映射只负责转换，不负责缓存和 Presenter 装配。
- 结论：可落地，但必须避免代理概念再分叉。

### `Docs/Architecture/A-12视图选择高亮与命中语义设计.md`

- 批次评估：是选择体系的核心文档，必须高度收敛。
- 关键问题：文档如果再定义一套 `SelectionSet`，后果就是选择语义会直接分裂成“集合作为容器”与“集合带规则”两种口径。
- 内部一致性问题：与 `A-02`、`A-05`、`A-06`、`A-07`、`A-08` 重叠最严重。
- 修改建议：拆开 `HitResult / HighlightState / SelectionRule`，选择集合只保留唯一实现，不再重复定义同名集合类型。
- 结论：必须收敛后再实现。

### `Docs/Architecture/A-13ScenePresenter设计.md`

- 批次评估：结构清晰，Presenter 的职责定位已经比较正确。
- 关键问题：Presenter 仍有吸收适配、缓存、刷新、句柄桥接的倾向，后果是一旦它变胖，就会变成新一层“隐性上帝对象”。
- 内部一致性问题：与 `A-05`、`A-11`、`A-14` 的职责分界必须极清楚。
- 修改建议：只保留批次装配、增量失效、句柄映射三类职责，任何业务规则都不许进入 Presenter。
- 结论：可落地，但要守住边界。

### `Docs/Architecture/A-14建模视图性能与缓存设计.md`

- 批次评估：必要且合理，属于后续性能保障的底座。
- 关键问题：缓存键如果只看 `elementId / floorId / viewMode / revision`，而不包含显示过滤、工作平面和主题状态，后果是“命中率看似高，实际内容却错了”。
- 内部一致性问题：与 `A-03`、`A-13`、`A-24`、`DisplayFilterService` 必须共享失效维度。
- 修改建议：缓存分层管理，键必须纳入会影响最终显示结果的所有语义维度，句柄对象用弱引用或受控句柄，不要直接长期持有后端裸对象。
- 结论：可落地，但必须先定失效键。

### `Docs/Architecture/A-15属性面板与建模视图联动设计.md`

- 批次评估：联动思路正确，属性面板不应只跟窗口绑定。
- 关键问题：属性面板如果直接持有选择、导航、视图和编辑上下文副本，后果是 2D/3D 切换后面板会失去“当前对象是谁”的一致答案。
- 内部一致性问题：与 `A-01`、`A-02`、`A-10`、`选择与拾取规则设计` 共享当前对象语义。
- 修改建议：引入统一的属性面板上下文控制器，属性面板只消费上下文投影，不维护自己的语义真值。
- 结论：可落地，但不能和视图状态并行存储。

### `Docs/Architecture/A-16局部预览与参数化预览设计.md`

- 批次评估：非常必要，尤其适合交互式建模。
- 关键问题：预览对象如果和正式对象共用同一条场景链路，后果是预览会污染正式选择、命中和回放。
- 内部一致性问题：与 `A-04`、`A-08`、`A-13`、`A-23` 的状态恢复和命令边界必须严格分开。
- 修改建议：预览必须有独立场景层或预览层，提交前做校验，取消时必须一次性清理。
- 结论：可落地，但必须和正式场景隔离。

### `Docs/Architecture/A-17节点构造二维表达系统设计.md`

- 批次评估：专题方向准确，适合做成参数驱动的二维表达体系。
- 关键问题：节点定义、实例、热区、构造切换如果不分层，后果是编辑、预览和回放会混在同一对象上。
- 内部一致性问题：与 `A-16`、`A-18`、`A-21` 的二维表达资源应该共享布局和符号层。
- 修改建议：把节点定义、节点实例、二维表达模型、热区模型拆开，实例只引用定义，不复制定义。
- 结论：可落地，但不要把节点系统做成单对象大杂烩。

### `Docs/Architecture/A-18钢筋示意图系统设计.md`

- 批次评估：业务语义明确，是后续钢筋表达的主通道。
- 关键问题：钢筋语义、示意、标注、搭接、弯钩如果不分层，后果是图形表达会逐渐退化成业务逻辑的直接绘制。
- 内部一致性问题：与 `A-17`、`A-20`、`A-21`、`A-19` 的复用关系必须统一。
- 修改建议：保持“语义模型 -> 草图模型 -> 符号/布局 -> 预览/输出”四层结构，不允许业务代码直接自绘示意。
- 结论：可落地，但需要严格的表达分层。

### `Docs/Architecture/A-19表格单元格二维缩略图渲染设计.md`

- 批次评估：适合报表与列表的落地场景。
- 关键问题：如果把单元格缩略图当成完整视图缩放，后果是小尺寸下信息不可读、缓存失控、批量渲染卡顿。
- 内部一致性问题：与 `A-18`、`A-21`、`A-22` 共用二维表达资源，但不能共用完整交互视图。
- 修改建议：单元格缩略图必须使用紧凑布局引擎和专用缓存，不允许直接复用交互视图渲染路径。
- 结论：可落地，但要把“缩略图”当成独立输出形态。

### `Docs/Architecture/A-20钢筋形状图库设计.md`

- 批次评估：必要且合理，能把形状模板资产化。
- 关键问题：模板、实例、目录、选型如果不分清，后果是形状库会逐渐变成“画法和数据混存”的杂物间。
- 内部一致性问题：与 `A-18`、`A-21` 的模板复用和符号复用要共享稳定 ID。
- 修改建议：模板只定义结构与参数，实例只承载具体值，图库服务只负责检索和实例化。
- 结论：可落地，但要防模板和实例混淆。

### `Docs/Architecture/A-21统一图元与符号库设计.md`

- 批次评估：这是统一视觉资产的关键层。
- 关键问题：如果业务代码仍允许自绘箭头、弯钩、标注和热区，后果是符号风格会再次分裂。
- 内部一致性问题：与 `A-18`、`A-19`、`A-22`、`A-24` 的样式和输出复用关系需要统一。
- 修改建议：图元库只输出参数化实例，不承载业务逻辑；业务层只能引用图元，不许自画同类符号。
- 结论：可落地，但必须建立统一符号入口。

### `Docs/Architecture/A-22打印导出截图统一输出设计.md`

- 批次评估：方向正确，应该把输出当成正式链路而不是临时截屏。
- 关键问题：截图、导出、打印如果各自使用不同渲染上下文，后果是屏幕看到的和输出得到的会不一致。
- 内部一致性问题：与 `A-14`、`A-24`、`A-19`、`A-23` 的显示语义必须共享同一输出上下文。
- 修改建议：统一 `OutputContext / OutputRenderer / CaptureRegion`，2D/3D 只是输出目标不同，不是输出语义不同。
- 结论：可落地，但输出链路必须统一。

### `Docs/Architecture/A-23回放系统与建模视图联动设计.md`

- 批次评估：这是回放体系落地的关键补丁。
- 关键问题：如果回放只重放命令，不恢复视图语义，后果是“业务上成功，现场却看不见当时问题”。
- 内部一致性问题：与 `JournalReplay数据模型设计`、`回放执行策略设计`、`A-01`、`A-02`、`A-03` 必须共用视图恢复顺序。
- 修改建议：录制时采集视图基线，回放时先恢复上下文再执行命令，且回放不能依赖 UI 显示显隐。
- 结论：可落地，但必须把视图状态纳入回放基线。

### `Docs/Architecture/A-24主题样式与显示规范设计.md`

- 批次评估：非常必要，属于视觉一致性的底座。
- 关键问题：主题如果直接决定语义状态，后果是颜色、线型、透明度会反向污染业务判断。
- 内部一致性问题：与 `A-05`、`A-21`、`A-22`、`DisplayFilterService` 共享显示语义，但职责不同。
- 修改建议：主题只管表现约束，不管业务语义；高亮、选中、锁定、禁用必须由统一语义层决定，再映射成样式。
- 结论：可落地，但必须把样式与语义分开。

**本批结论：** 统一建模视图专题覆盖完整，已经形成 `ModelingContext -> Scene/Proxy -> Presenter -> Backend` 主线；但 A-01 到 A-24 仍需要统一字段级契约，尤其是 `ViewState`、`SelectionSet`、`DisplayFilter`、`SCWorkContext`、`ReplayViewState` 和后端句柄的归属。若直接按各专题并行实现，2D/3D 切换、属性联动、预览、输出和回放会出现状态漂移。

## 五、设计专题

### `Docs/Design/DisplayFilterService与显示规则设计.md`

- 批次评估：必要，且定位基本正确。
- 关键问题：显示规则的三层叠加思路对，但 `defaultVisibleTypes / forceVisibleTypes / forceHiddenTypes` 还不够表达复杂优先级，后果是后续会用更多分支去补规则。
- 内部一致性问题：与 `NavigationCategory`、`A-24`、`A-12`、`选择与拾取规则设计` 都会争夺“谁决定可见”的解释权。
- 修改建议：明确显示服务只输出显示结果，不决定命令可用性，也不决定选择结果。
- 结论：可落地，但必须限制职责边界。

### `Docs/Design/回放执行策略设计.md`

- 批次评估：质量较高，能够作为回放执行框架的直接依据。
- 关键问题：无界面回放、调试回放、批量回放三种模式都提到了，但还缺少与快照、错误模型、断言模型的统一恢复规则，后果是不同执行模式会形成三套排障路径。
- 内部一致性问题：与 `JournalReplay数据模型设计`、`Replay 快照策略设计`、`自动化测试断言模型设计` 必须共用回放上下文。
- 修改建议：明确回放上下文重建的最小数据集、失败中断策略和诊断输出格式。
- 结论：可落地，但需要补统一诊断闭环。

### `Docs/Design/命令结果与错误模型设计.md`

- 批次评估：非常必要，是命令体系可测试、可回放的基础。
- 关键问题：`SCResult<T>` 只能表达成败和值，无法表达错误分类、可重试性、可回滚性，后果是 UI、日志、回放和 Storage 都会回到字符串判断。
- 内部一致性问题：与 `命令提交边界设计`、`回放执行策略设计`、`自动化测试断言模型设计` 共享错误语义。
- 修改建议：把结果、错误码、严重度、建议动作拆成独立字段，并统一映射 UI 文案。
- 结论：可落地，但必须升级结果模型。

### `Docs/Design/快捷键体系设计.md`

- 批次评估：方向正确，已经意识到快捷键应该服务命令而不是服务按钮。
- 关键问题：如果快捷键绑定到按钮实例而不是命令 ID，后果是菜单、Ribbon、对话框和回放脚本的触发口径会不一致。
- 内部一致性问题：与 `UI命令装配与菜单配置体系设计`、`命令系统总体设计`、`命令结果与错误模型设计` 要共用统一命令元数据。
- 修改建议：统一注册、统一冲突检查、统一作用域，快捷键只做触发，不承载逻辑。
- 结论：可落地，但要以命令为主键。

### `Docs/Design/选择与拾取规则设计.md`

- 批次评估：必要且正确。
- 关键问题：文档已经指出显示与选择必须分开，但还缺少统一的命中结果模型和优先级规则，后果是不同后端对同一输入给出不同选择结果。
- 内部一致性问题：与 `A-09`、`A-12`、`DisplayFilterService`、`A-15`、`A-16` 共用命中语义。
- 修改建议：选择策略、命中结果、命中上下文分层，隐藏与锁定对象的规则必须由同一策略服务解释。
- 结论：可落地，但必须统一命中出口。

### `Docs/Design/属性面板View对话框联动设计.md`

- 批次评估：联动方向是对的。
- 关键问题：属性面板、View、对话框如果各自维护一份上下文，后果就是“面板看的是一个对象，对话框编辑的是另一个对象”。
- 内部一致性问题：与 `A-15`、`命令系统总体设计`、`A-01` 的上下文投影重叠。
- 修改建议：把三者统一接入同一个面板上下文控制器，通过绑定服务共享状态，不再直接互相拉状态。
- 结论：可落地，但必须以统一上下文为主。

### `Docs/Design/自动化测试断言模型设计.md`

- 批次评估：必要，是回放和测试闭环的一部分。
- 关键问题：断言类型已经覆盖数据、命令、显示、结果、几何，但缺少和 `JournalReplay` 的强关联，后果是断言失败难以回到具体命令和会话。
- 内部一致性问题：与 `Replay 执行策略`、`JournalReplay 数据模型`、`命令结果与错误模型` 需要同一套诊断标识。
- 修改建议：给断言结果增加会话 ID、命令 ID、步骤 ID 和实际/预期快照引用。
- 结论：可落地，但必须和回放模型同源。

### `Docs/Design/回放快照策略设计.md`

- 批次评估：思路正确，快照应当是辅助恢复手段。
- 关键问题：如果快照策略没有体积预算和失效规则，后果是回放目录会快速膨胀，维护成本很高。
- 内部一致性问题：与 `JournalReplay数据模型设计`、`工程文件与日志回放关系设计`、`日志归档与清理策略设计` 必须统一保留策略。
- 修改建议：明确 baseline / checkpoint / debug / autotest 四类快照的生成节点和清理条件。
- 结论：可落地，但要先管住快照数量。

### `Docs/Design/工程生命周期命令设计.md`

- 批次评估：必要，且边界基本正确。

### `Docs/Design/导入导出类命令设计.md`

- 批次评估：必要，且属于高风险命令族。
- 关键问题：导入导出会同时影响文件系统、工程正式状态和派生数据，后果是如果没有清晰错误处理和撤销边界，极容易产生不可恢复的数据冲突。
- 内部一致性问题：与 `工程文件与日志回放关系设计`、`命令参数与序列化规范设计`、`命令结果与错误模型设计` 直接相关。

**本批结论：** 设计专题多数具备实施价值，但仍偏“原则 + 示例”，缺少统一 schema、状态机、错误码、诊断字段和命令族模板。它们应作为 Architecture 的落地细化层，不能重新定义上下文、选择、显示、回放或提交边界。

## 全局架构审查总结

### 1. 跨批次不一致问题

1. `ModelingContext / ViewState / SelectionSet / DisplayFilter / CommandContext / EditingContext / DisplayContext / ExecutionContext / WorkContext` 在不同文档里被重复定义，后果是状态真值源不唯一。
2. `SelectionSet / Highlight / HitResult / PickResult / SnapResult` 的语义层级不完全一致，后果是选择、命中、捕捉和高亮在不同模块里会给出不同解释。
3. `ViewMode / BackendKind / ViewState.mode / SC2DViewportState / SC3DViewState / ViewSwitchSnapshot` 共存，但主键没有统一，后果是 2D/3D 切换和回放恢复会分叉。
4. `SceneElementProxy / SCSceneProxyData / SCModelObjectRef / ElementHandle / SC2DRenderItem / SC3DEntityHandle` 同时出现，后果是工程对象、代理对象、后端句柄的边界不清。
5. `NavigationCategory`、`DisplayFilterService`、`属性面板`、`拾取规则`、`命令可用性` 之间的解释权分散，后果是 UI 和业务状态会互相倒灌。
6. 索引已经合并为单一入口；如果后续再拆出第二份索引，会重新引入审查范围和实现范围不一致的问题。
7. Storage、Geometry 在多份设计文档中被引用为基础能力，但没有统一 Adapter 契约，后果是业务命令可能直接依赖 `D:\code\StableCore\Storage` 或 `D:\code\StableCore\Geometry` 的具体实现细节。

### 2. 架构断层

1. 缺少一个统一的 `ModelingContext` 主线，去承载可恢复语义并向各子系统投影。
2. 缺少一个明确的 `Context Projection` 层，去把命令、交互、捕捉、工作平面、显示过滤从主状态派生出来。
3. 缺少一个严格的 `Presenter / Adapter / Backend` 三段式边界，当前 Presenter 仍有吸收适配和缓存职责的趋势。
4. 缺少统一的句柄注册和失效管理层，`void*` 和后端私有句柄在多份文档里仍然是高风险接口。
5. 缺少命令提交单元与 Storage 事务、Replay 记录的一一对应层。
6. 缺少 `StorageAdapter / GeometryAdapter` 的产品侧契约层，当前只说明“使用自研库”，还没有说明事务、错误、对象 ID、几何容差、版本兼容如何进入 Quantity 架构。

### 3. 高风险设计点

1. `SelectionSet`、`DisplayFilter`、`ViewState` 在多个文档里被重复持有，一旦实现，回放和 Undo 迟早不一致。
2. `NavigationCategory` 如果继续承接显示、命令、属性、拾取四类职责，会演变成第二个主上下文。
3. `DisplayStyle`、`ThemeDefinition`、`HighlightState` 如果混在一起，样式会反向污染业务语义。
4. `void*` 和后端私有句柄如果直接泄露到上层，生命周期失控和悬挂引用几乎必然出现。
5. 快照策略如果没有预算和失效键，回放资产会指数式膨胀，后果是长期维护成本失控。
6. 生命周期命令如果混入普通 Undo 链，工程状态历史会被污染。
7. 如果工程模型、Storage 事务、Replay Journal 三者没有共享提交 ID，崩溃恢复时会出现“数据库已写入但 Journal 未记录”或“Journal 已记录但工程状态未落库”的不一致。
8. 如果 Geometry 的容差、坐标系和楼层工作平面没有统一上下文，捕捉、建模、计算、显示会各自得到不同几何结果。

### 4. 架构优化建议

1. 统一定义 `ModelingContext` 为唯一运行时主状态，所有其他上下文都改成投影或派生。
2. 统一定义 `SelectionSet / DisplayFilter / ViewState` 的归属，不允许在 2D/3D/Presenter/命令层重复存副本。
3. 统一定义 `CommandSubmission`，让命令提交、Storage 事务、Undo、Replay、Assert 挂在同一个提交单元上。
4. 统一定义 `Presenter -> Adapter -> Backend` 的三段式链路，Presenter 不做业务，Adapter 不做状态，Backend 不做语义。
5. 统一定义回放资产目录和清理规则，让工程文件、Journal、Snapshot、Test Asset 分层存放。
6. 统一定义命名主键和对象引用主键，禁止临时指针成为跨模块主键。
7. 统一定义 `StorageAdapter`：只暴露 Quantity 需要的工程事务、对象仓储、变更集、错误码和版本迁移，不让上层直接依赖 SQLite 表细节。
8. 统一定义 `GeometryAdapter`：只暴露 Quantity 需要的几何构造、布尔/相交、投影、捕捉、容差策略和坐标转换，不让 UI、命令、显示后端直接调用算法库内部模型。

### 5. 最终建议结构

推荐最终架构模型如下：

```text
Project Model
  ├─ StorageAdapter
  │   ├─ Project Repository
  │   ├─ Transaction / ChangeSet
  │   └─ Migration / Recovery
  ├─ GeometryAdapter
  │   ├─ Tolerance / Coordinate Policy
  │   ├─ Construct / Intersect / Project
  │   └─ Snap Geometry Query
  ├─ ModelingContext
  │   ├─ ViewState
  │   ├─ SelectionSet
  │   ├─ DisplayFilter
  │   └─ Context Projections
  ├─ Command Layer
  │   ├─ CommandManager
  │   ├─ CommandContext Projection
  │   └─ Submission / Replay / Undo Boundary
  ├─ Coordination Layer
  │   ├─ ModelingViewHost
  │   ├─ ScenePresenter
  │   ├─ InteractionRouter
  │   └─ SnapService
  ├─ Display Layer
  │   ├─ DisplayAdapterRegistry
  │   ├─ ElementDisplayAdapter
  │   ├─ 2D Backend
  │   └─ 3D Backend
  └─ Asset / Replay Layer
      ├─ Journal
      ├─ Snapshot
      ├─ AssertJournal
      └─ Archive / Cleanup Policy
```

### 6. 最终判断

当前文档体系不是“方向错了”，而是“骨架已经具备，但主状态和主上下文还没有完全收敛”。

如果现在直接开工，短期能出功能，长期一定会在以下地方返工：

1. 2D / 3D 切换恢复不稳定。
2. 命令回放和 Undo 不一致。
3. 选择、显示、命中、属性联动各自为政。
4. 缓存、句柄和预览的生命周期失控。

因此，最终建议是：

1. 先统一状态主权。
2. 再统一上下文投影。
3. 然后统一命令提交边界。
4. 再把 `SC/ISC` 品牌前缀和 `Extern` 安装依赖规则固定到代码、文件和构建层。
5. 最后再扩展 2D / 3D、预览、回放、输出和专题表达。

## 七、修订后的统一结论

本次审查的核心结论已经从“分散指出问题”收敛为“统一约束边界”，后续所有架构设计文档都应按同一套主线修订：

### 1. 唯一状态主权

`Project Model` 是工程模型唯一数据源，`ModelingContext` 是运行时唯一真值源。`ViewState / SelectionSet / DisplayFilter / CommandContext / EditingContext / DisplayContext / ExecutionContext / WorkContext` 只能作为投影、派生或临时快照，不得再作为并列真值源。

### 2. 唯一上下文投影

统一建模视图、命令、捕捉、属性面板、导航、回放都必须从同一运行时主上下文投影而来，禁止各模块各自保存一份“当前对象 / 当前选择 / 当前显示 / 当前模式”的副本。

### 3. 唯一提交边界

`CommandSubmission / CommitId / SubmissionId` 统一命令调用、Storage 事务、Undo / Replay / Assert 记录边界。正式提交与普通命令调用必须分离，任何后续记录都挂在同一提交单元上。

### 4. 统一显示链路

显示体系固定为 `Presenter -> Adapter -> Backend` 三段式链路，Presenter 只做批次装配与增量失效，Adapter 只做语义到显示数据翻译，Backend 只做渲染执行。2D / 3D 后端都不得再持有独立业务真值。

### 5. 统一命名与 target 口径

概念术语可以保持领域原名，但代码文件、代码标识、公开命名空间和公开 target 必须统一采用 `SC/ISC` 品牌前缀。`sc_quantity_*` 这种小写品牌表达不再作为公开 target 主口径。

### 6. 统一 Extern 消费边界

主仓库只消费 `..\Extern\stablecore`、`..\Extern\thirdparty-ogre-next`、`..\Extern\thirdparty-saribbon` 的安装结果，不直接依赖外部库源码树。目录定义、模块依赖和 CMake target 都必须围绕这一边界展开。

### 7. 统一资产生命周期

`Journal / Snapshot / Replay / Crash Recovery / Cleanup` 必须共享同一套资产生命周期规则，工程文件、回放资产、快照资产和测试资产要分层存放、分层保留、分层清理，禁止相互污染。

### 8. 后续落地顺序

后续若继续修订 Architecture / Design 文档，建议按以下顺序同步：

1. 先改 `目录定义`、`模块职责与依赖约束定义`、`CMake target 划分方案`、`命名规范`。
2. 再改 `命令系统总体设计`、`命令提交边界设计`、`命令参数与序列化规范设计`、`JournalReplay数据模型设计`。
3. 然后改统一建模视图专题 `A-01 ~ A-16`。
4. 最后补 `日志归档与清理策略设计`、`回放执行策略设计`、`自动化测试断言模型设计` 等治理和闭环文档。
