## 二、实现问题

### 1. SQLite 查询仍是内存全扫，索引没有真正吃到

- 问题描述：`SqliteTable::FindRecords` 和 `EnumerateRecords` 先把结果完整塞进 `std::vector<SCRecordPtr>`，再返回游标；查询并没有走 SQLite 索引和 SQL 过滤。
- 代码层体现：`D:\code\stablecore\Storage\Src\Sqlite\SCSqliteAdapter.cpp:983`，`D:\code\stablecore\Storage\Src\Sqlite\SCSqliteAdapter.cpp:997`，以及 `FindRecords` 中的 `for (const auto& [_, data] : records_)`。
- 风险等级：高
- 建议改法：把 SQLite 查询改成真正的后端查询游标，至少对 `recordId`、单列等值条件走 SQL 过滤；`EnumerateRecords` 也应按需迭代，不要一次性物化整表。

### 2. 删除引用检查是跨表、跨记录的全量扫描

- 问题描述：删除前的引用判断在内存与 SQLite 两个后端里都要遍历全部表、全部候选记录和全部关系列。
- 代码层体现：`D:\code\stablecore\Storage\Src\Memory\SCMemoryDatabase.cpp:803`，`D:\code\stablecore\Storage\Src\Sqlite\SCSqliteAdapter.cpp:1862`。
- 风险等级：高
- 建议改法：把关系引用做成显式索引或反向引用表，删除检查只查受影响的引用集合，不要全库扫描。

### 3. 计算列缓存的失效逻辑没有清理旧版本条目

- 问题描述：缓存 key 带 `version`，但失效只删除命中的旧 key，没有做按记录/列维度的旧版本回收，旧版本缓存会持续累积。
- 代码层体现：`D:\code\stablecore\Storage\Include\ISCComputed.h:43`，`D:\code\stablecore\Storage\Src\Computed\SCTableView.cpp:353`，`D:\code\stablecore\Storage\Src\Computed\SCComputedRuntime.cpp:520`。
- 风险等级：高
- 建议改法：缓存 key 不要只依赖版本号做唯一定位；至少补一层按 `recordId + columnName` 的覆盖式缓存，并在版本推进时清理过期条目。

### 4. 计算列失效没有覆盖结构性变更

- 问题描述：`DoesDependencySetIntersect` 只比对字段名，`change.structuralChange` 没有参与；而创建/删除记录的 `fieldName` 为空，导致依赖于结构变化的计算列很容易失效不完整。
- 代码层体现：`D:\code\stablecore\Storage\Src\Computed\SCComputedRuntime.cpp:488`，`D:\code\stablecore\Storage\Src\Computed\SCComputedRuntime.cpp:615`，`D:\code\stablecore\Storage\Src\Memory\SCMemoryDatabase.cpp:1001`，`D:\code\stablecore\Storage\Src\Sqlite\SCSqliteAdapter.cpp:2112`。
- 风险等级：高
- 建议改法：把结构性变更、记录新增/删除、关系字段变更纳入依赖匹配；对聚合列至少增加表级和记录级依赖标记。

### 5. 批处理错误路径可能把数据库留在活跃事务里

- 问题描述：`ExecuteBatchEdit` 在失败时只有 `rollbackOnError == true` 才回滚；如果是 `false`，函数会直接返回错误，活跃编辑可能不会被关闭。
- 代码层体现：`D:\code\stablecore\Storage\Src\Batch\SCBatchOperations.cpp:61`，`D:\code\stablecore\Storage\Src\Batch\SCBatchOperations.cpp:134`，`D:\code\stablecore\Storage\Src\Batch\SCBatchOperations.cpp:141`。
- 风险等级：高
- 建议改法：批处理必须用 RAII 守卫保证函数退出时一定关闭编辑事务；`rollbackOnError` 只决定是否回滚，不应该决定是否泄漏活跃会话。

### 6. 观察者机制使用原始指针，生命周期和重复注册都没有防护

- 问题描述：观察者以 `ISCDatabaseObserver*` 直接入栈，没有去重、没有所有权约定、没有失效检查，通知时还是同步回调。
- 代码层体现：`D:\code\stablecore\Storage\Src\Memory\SCMemoryDatabase.cpp:406`，`D:\code\stablecore\Storage\Src\Memory\SCMemoryDatabase.cpp:589`，`D:\code\stablecore\Storage\Src\Sqlite\SCSqliteAdapter.cpp:737`，`D:\code\stablecore\Storage\Src\Sqlite\SCSqliteAdapter.cpp:1660`，`D:\code\stablecore\Storage\Src\Memory\SCMemoryDatabase.cpp:1027`，`D:\code\stablecore\Storage\Src\Sqlite\SCSqliteAdapter.cpp:2137`。
- 风险等级：高
- 建议改法：把观察者注册改成带去重的订阅句柄或弱引用语义；通知阶段要隔离异常并避免悬挂指针。

### 7. SQLite 提交路径忽略了部分持久化调用的返回值

- 问题描述：`PersistTouchedRecords`、`SaveMetadata`、`DeleteRedoJournalRows` 等辅助函数里有多处 `Step()`/`Reset()` 结果未检查，提交成功与否不完全受返回值约束。
- 代码层体现：`D:\code\stablecore\Storage\Src\Sqlite\SCSqliteAdapter.cpp:2159`，`D:\code\stablecore\Storage\Src\Sqlite\SCSqliteAdapter.cpp:2188`，`D:\code\stablecore\Storage\Src\Sqlite\SCSqliteAdapter.cpp:2244`，`D:\code\stablecore\Storage\Src\Sqlite\SCSqliteAdapter.cpp:2264`。
- 风险等级：高
- 建议改法：把持久化步骤改成逐步失败即中止的事务函数，任何一步失败都应导致整个提交返回失败，并确保事务回滚。

### 8. 启动元数据读取对损坏输入过于脆弱

- 问题描述：`LoadMetadata` 直接对字符串做 `stoull / stoi / stoll`，没有容错；一旦元数据损坏，打开过程会直接落到泛化失败。
- 代码层体现：`D:\code\stablecore\Storage\Src\Sqlite\SCSqliteAdapter.cpp:1091`，`D:\code\stablecore\Storage\Src\Sqlite\SCSqliteAdapter.cpp:1106`。
- 风险等级：中
- 建议改法：元数据解析要显式捕获格式错误，使用默认值或进入诊断模式，而不是把问题统一压成 `SC_E_FAIL`。

### 9. 升级路径只有单线性计划，没有完整版本图谱

- 问题描述：迁移接口存在，但 SQLite 只实现了固定的 `1 -> 2` 迁移步骤，没有降级、没有分支、没有回滚窗口。
- 代码层体现：`D:\code\stablecore\Storage\Src\Migration\SCMigration.cpp:8`，`D:\code\stablecore\Storage\Src\Sqlite\SCSqliteAdapter.cpp:1284`。
- 风险等级：中
- 建议改法：把迁移定义提升为版本图谱，至少明确 forward-only 的版本链、失败补偿和启动时的恢复策略。

### 10. Memory / SQLite 两套后端逻辑高度重复，后端耦合偏重

- 问题描述：内存和 SQLite 两份实现复制了大量相同的事务、删除、Journal、ChangeSet、引用检查逻辑，未来修 bug 很容易只修一半。
- 代码层体现：`D:\code\stablecore\Storage\Src\Memory\SCMemoryDatabase.cpp:729`，`D:\code\stablecore\Storage\Src\Memory\SCMemoryDatabase.cpp:986`，`D:\code\stablecore\Storage\Src\Sqlite\SCSqliteAdapter.cpp:1793`，`D:\code\stablecore\Storage\Src\Sqlite\SCSqliteAdapter.cpp:2097`。
- 风险等级：中
- 建议改法：把事务语义、Journal 组装、ChangeSet 生成、约束校验抽到共享核心层，SQLite 只保留存储与 SQL 映射。

## 三、隐患分析

### 1. 数据量扩大 10x / 100x 后的隐患

- 全库扫描会先爆：`FindRecords`、`EnumerateRecords`、引用检查都在内存里枚举集合，不是流式查询。
- 计算列缓存会膨胀：缓存 key 带版本号，但旧版本条目没有系统性回收。
- 观察者通知会放大成本：每次提交、Undo、Redo 都要同步通知全部订阅者。
- 具体影响：查询延迟上升、内存占用增长、删除/关系校验变慢、长会话下缓存越来越大。

### 2. 引入 Replay / 更强 Undo 时的隐患

- 现在的回放更像“事务栈恢复”，不是完整的 Replay 契约。
- `JournalEntry` 里缺少更强的结构性标记，计算列失效也没把结构变化纳入。
- SQLite 提交路径对持久化步骤的错误处理不够硬，一旦出现部分写入失败，Replay 的可重建性会变差。
- 具体影响：跨版本回放、损坏恢复、审计回放都可能出现“能恢复大部分，但不能证明完全一致”的问题。

### 3. 多模块接入时的隐患

- 观察者是原始指针，模块退出顺序只要稍有不一致，就可能产生悬挂回调。
- 同一观察者可以重复注册，通知次数不可控。
- 通知是同步的，任何观察者阻塞都会拖慢存储提交链路。
- 具体影响：模块越多，越容易出现回调风暴、时序耦合和难定位崩溃。

### 4. 批量编辑和导入场景的隐患

- 失败路径依赖调用方是否愿意回滚，事务收口不够强。
- 导入逻辑本身复用批处理，但没有单独的部分失败隔离策略。
- 具体影响：批量导入一旦中途失败，数据库可能停留在活跃编辑状态，后续操作被写冲突拦住。

## 四、结论

代码已经把设计中的主干能力大体落地了：

- Schema、值模型、事务、Journal、Undo/Redo、SQLite 持久化、计算列、诊断、迁移都已经有实现。

但和“平台级完整实现”相比，当前偏差主要集中在四类：

1. 查询和引用检查仍是全量扫描，不适合数据继续放大。
2. Replay / 计算列失效 / 版本演进这些历史链路还不够硬。
3. 观察者和批处理的边界不够稳，模块接入后容易出时序问题。
4. 内存与 SQLite 的重复实现太多，后续维护风险高。

如果下一步继续收敛，优先级建议是：

1. 先把查询、引用检查、批处理事务收口改成可扩展且可失败收敛的形式。
2. 再补完整 replay / cache invalidation / migration 契约。
3. 最后再做后端代码去重和工程化收束。
