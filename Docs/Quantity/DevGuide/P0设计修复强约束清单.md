# P0 设计修复强约束清单

本清单用于约束 Phase 1 的 P0 设计修复，目标是把工程数据、配置数据、Replay 资产和问题包边界彻底收紧，避免任何隐式写入和语义混用。

## 强制 / 禁止规则

1. 工程正式状态必须只进入 Project / ChangeSet / Journal 主链路。
2. 用户偏好配置不得进入工程 Undo 链、Redo 链或 Replay 主链路。
3. 系统配置必须独立于工程数据存储，不得默认随工程保存或回放。
4. open 必须只承担加载、校验与兼容性判断，不得产生持久化写入。
5. open 不得自动升级工程格式，不得自动标记工程为可录制状态。
6. upgrade 必须作为独立显式动作，不得由 open、read、preview 等动作隐式触发。
7. record 必须依赖显式开始条件，不得由工程打开或加载结果自动开启。
8. replay continue editing 必须绑定明确的 branch anchor，不得仅依赖当前界面状态。
9. Replay 继续编辑必须记录来源 CommitId / SnapshotId / SessionId，不得只记录“当前状态”。
10. Replay 状态必须显式区分 Available、Partial、Broken、VersionMismatch 等状态，不得仅用布尔值判断。
11. Replay 回放入口必须先判断锚点，再判断版本契约，最后判断资产完整性，不得反向推断。
12. 问题包导出必须显式选择资产范围，不得默认包含全部 Replay 资产。
13. 问题包不得作为工程正式状态的替代副本，不得承载可继续编辑的主链路数据。
14. 问题包导出必须具备大小上限、脱敏规则和资产筛选规则，不得无边界打包。
15. 主题样式必须归入用户偏好或系统展示配置，不得默认进入工程正式状态。
16. 只有当主题参数会改变业务语义时，才必须单独定义为工程元数据，不得混入普通设置项。
17. 自动化测试中的只读打开必须保持只读语义，不得因为测试初始化而触发工程写入。
18. 只读打开必须能够完整完成加载、校验与判定，不得通过自动升级绕过只读边界。
19. 任何会改变工程正式状态的动作必须显式进入事务或命令边界，不得靠隐式副作用完成。
20. 任何跨越工程、配置、Replay、问题包的行为都必须先明确归属，再定义写入边界，不得混用同一条持久化链路。

## 说明

- 以上规则优先级高于便利性设计。
- 任何例外都必须单独形成设计豁免说明，不能在实现中临时放宽。
- 若后续文档与本清单冲突，必须先修正文档，再推进实现。
