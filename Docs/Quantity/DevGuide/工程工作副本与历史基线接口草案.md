# 工程工作副本与历史基线接口草案

> 草案目的：为算量产品定义一套可实现的 Storage 接口，覆盖“打开工程 -> 工作副本 -> 编辑 -> 保存/另存 -> 历史重置 -> 关闭丢弃”流程。
>
> 边界说明：
> - 临时目录管理由产品层负责。
> - Storage 不负责替产品层做目录策略，也不直接暴露 `CopyFile`。
> - Storage 只负责一致性备份、历史基线重置、历史状态枚举、工作副本状态查询。
> - 现有事务、导入恢复、诊断、观察者接口继续保留。

## 1. 设计目标

1. 保存时，将当前工作副本固化为新的工程文件。
2. 保存成功后，Undo/Redo 从新的保存基线重新开始。
3. 配置库可显式关闭历史，避免无意义的历史增长。
4. 产品层能展示 Undo/Redo 列表项，而不只是计数。
5. 不把“保存语义”混进“单次事务提交语义”。

## 2. 范围与约束

### 2.1 适用范围

这份草案要求以下后端都支持同一套语义：

- Memory backend
- SQLite backend

### 2.2 非目标

- 不在 Storage 内部管理临时目录。
- 不要求产品层直接调用 `CopyFile`。
- 不要求把“保存后清空历史”并入 `Commit()`。
- 不要求把完整审计日志暴露给用户界面。

### 2.3 核心约束

- `Normal` 模式下，数据库维护正常 Undo/Redo 和必要恢复日志。
- `NoHistory` 模式下，数据库不维护用户可见历史，`GetHistoryState()` 返回空列表。
- `ReadOnly` 模式下，不允许写入和任何历史改变。
- `Commit()` 仍然只表示一次编辑事务的提交，不表示“保存点归零”。

## 3. 打开模式

```cpp
enum class SCDatabaseHistoryMode
{
    Normal,
    NoHistory,
    ReadOnly,
};

struct SCOpenDatabaseOptions
{
    SCDatabaseHistoryMode historyMode{SCDatabaseHistoryMode::Normal};
    bool useWorkingCopy{true};
};
```

语义：

- `Normal`：工程库，保留 Undo/Redo、保存基线和恢复日志。
- `NoHistory`：配置库，不维护用户可见 Undo/Redo，也不保留增长型历史日志。
- `ReadOnly`：只读打开，不允许写入和历史变更。
- `useWorkingCopy = true`：产品层以工作副本方式使用该数据库实例。

### 4.2 备份结果


### 4.3 历史项

```cpp
enum class SCHistoryActionKind
{
    Unknown,
    Commit,
    Undo,
    Redo,
    Import,
    RuleWriteback,
    SaveBaseline,
};

struct SCHistoryEntry
{
    CommitId commitId{0};
    VersionId version{0};
    SCHistoryActionKind kind{SCHistoryActionKind::Unknown};
    std::wstring displayText;
    std::wstring detailText;
    std::uint64_t timestampUtcMs{0};
};

struct SCHistoryState
{
    VersionId baselineVersion{0};
    std::vector<SCHistoryEntry> undoItems;
    std::vector<SCHistoryEntry> redoItems;
};
```

约束：

- `undoItems` 是当前可撤消项。
- `redoItems` 是当前可重做项。
- `NoHistory` 模式下，两者都必须为空。

### 4.4 工作副本状态

```cpp
struct SCProjectWorkState
{
    bool open{false};
    bool dirty{false};
    SCDatabaseHistoryMode historyMode{SCDatabaseHistoryMode::Normal};
    std::wstring projectPath;
    std::wstring workingCopyPath;
    VersionId currentVersion{0};
    VersionId baselineVersion{0};
    std::size_t undoCount{0};
    std::size_t redoCount{0};
};
```

约束：

- `projectPath` 是用户工程文件路径。
- `workingCopyPath` 是当前工作副本路径，由产品层管理目录。
- `dirty = true` 表示相对 `baselineVersion` 已发生可保存变更。

## 5. `ISCDatabase` 建议扩展

下面这些接口建议直接挂到 `ISCDatabase` 上。

```cpp
class ISCDatabase : public virtual ISCRefObject
{
public:
    // Creates a consistent on-disk copy of the current database.
    // The target path is owned by the product layer.
    virtual ErrorCode CreateBackupCopy(
        const wchar_t* targetPath,
        const SCBackupOptions& options,
        SCBackupResult* outResult) = 0;

    // Resets the current working copy to a new save baseline.
    // History trimming and recovery-log compaction happen inside this call.
    virtual ErrorCode ResetHistoryBaseline(SCBackupResult* outResult = nullptr) = 0;

    // Returns the current visible undo/redo history items.
    virtual ErrorCode GetHistoryState(SCHistoryState* outState) const = 0;

    // Returns the current working-copy state.
    virtual ErrorCode GetWorkState(SCProjectWorkState* outState) const = 0;

    // Discards transient working-copy state.
    virtual ErrorCode DiscardWorkingCopy() = 0;
};
```

### 5.1 方法语义

#### `CreateBackupCopy`

- 从当前工作副本生成一致性文件副本。
- 后端必须能处理数据库文件处于打开状态的场景。
- Memory backend 也必须实现，语义为“把当前内存状态导出为一个可重打开的 SQLite 文件”。
- SQLite backend 语义为“生成当前数据库的原子一致性副本”。

#### `ResetHistoryBaseline`

- 将当前状态设为新的保存基线。
- `Normal` 模式下，清空当前保存点之前的用户可见历史。
- `NoHistory` 模式下，清空残留历史摘要并保持空历史状态。
- 内部可以顺带做恢复日志裁剪或封存。
- 对用户可见的结果是：保存后，旧保存点之前的 Undo/Redo 列表不再显示。

#### `GetHistoryState`

- 返回当前工作副本的可见历史，不是完整审计日志。
- `undoItems` 和 `redoItems` 都按 UI 可直接展示的顺序返回。
- `NoHistory` 模式下必须返回空列表。

#### `GetWorkState`

- 返回当前工作副本的脏状态、模式、路径和历史摘要。
- 供产品层决定是否启用保存、撤销、重做、关闭提示。

#### `DiscardWorkingCopy`

- 丢弃当前工作副本的 transient 状态。
- 文件删除、临时目录清理仍由产品层负责。
- 这个接口只负责让数据库对象回到“未继续编辑”的状态。

## 6. 打开接口关系

建议保留现有底层打开原语，同时新增带选项的打开入口。

```cpp
ErrorCode CreateInMemoryDatabase(
    const SCOpenDatabaseOptions& options,
    SCDbPtr& outDatabase);

ErrorCode CreateFileDatabase(
    const wchar_t* path,
    const SCOpenDatabaseOptions& options,
    SCDbPtr& outDatabase);
```

兼容说明：

- 现有无 options 的重载可以保留，默认等价于 `Normal + useWorkingCopy = false` 或当前仓库既有行为。
- 产品层打开工程时应显式传入 `SCOpenDatabaseOptions`。
- 配置库建议使用 `SCDatabaseHistoryMode::NoHistory`。

## 7. 推荐保存流程

1. 产品层打开用户工程文件。
2. 产品层创建临时工作目录并复制工程文件。
3. 产品层用带选项的 `CreateFileDatabase()` 打开临时工作副本。
4. 用户编辑作用在工作副本。
5. 用户点击保存时：
   - 调用 `CreateBackupCopy(targetPath, options, outResult)` 生成一致性工程文件。
   - 调用 `ResetHistoryBaseline(&outResult)` 重置 Undo/Redo 基线。
   - 必要时由产品层原子替换用户工程文件。
6. 用户关闭且未保存时，产品层删除临时目录并释放数据库句柄。

## 8. 约束建议

- `Commit()` 仍然只表示单次编辑事务提交。
- `Undo()` / `Redo()` 在 `Normal` 模式下按现有语义工作。
- `Undo()` / `Redo()` 在 `NoHistory` 模式下应返回空栈错误，且 `GetHistoryState()` 必须为空。
- `ReadOnly` 模式下任何写操作都应返回只读错误。
- `TrimRecoveryLog()` 不建议单独暴露给产品层；如有需要，放在 `ResetHistoryBaseline()` 内部实现。

## 9. 结论

这版草案的可实现边界是：

- Memory 和 SQLite 都实现同一套接口。
- 配置库通过 `NoHistory` 模式关闭用户可见历史。
- `GetHistoryState()` 用列表项表达 Undo/Redo，而不是只给计数。
- 工作副本生命周期由产品层管理，Storage 只提供一致性备份和历史基线能力。
