# Phase 2：P1 平台能力修复方案

本阶段修复的是平台能力问题，不直接进入代码实现。目标是把版本升级、导入导出、主体隔离、查询与引用检查从“可用”提升到“可扩展、可维护、可恢复”的平台级能力。

## 一、P1 各问题的目标能力定义

### 1. 存储版本升级没有形成完整版本图谱、降级 / 回滚 / 兼容窗口不够硬

- 当前缺失能力：
  - 缺少版本图谱。
  - 缺少明确的迁移方向约束。
  - 缺少启动升级、失败恢复和兼容窗口策略。
- 修复后平台必须提供的能力：
  - 明确的 Version Graph。
  - 可判断的 forward-only 迁移链路。
  - 有界兼容窗口。
  - 启动升级流程和失败恢复流程。
  - 历史工程可识别、可导入、可提示。
- 所属层：
  - Storage Core
  - Adapter
  - Product
  - Tooling

### 2. 大型模型 / 钢筋导入被默认当作单次可整体回滚事务

- 当前缺失能力：
  - 缺少分块提交。
  - 缺少 checkpoint。
  - 缺少幂等重试。
  - 缺少局部失败恢复。
- 修复后平台必须提供的能力：
  - 分块导入与分块提交。
  - 可恢复 checkpoint。
  - 导入会话级幂等控制。
  - 最终确认机制。
  - 局部失败可继续或可回退的策略。
- 所属层：
  - Storage Core
  - Adapter
  - Product
  - Tooling

### 3. 问题包导出把工程、Replay、Snapshot、Log 默认打包在一起

- 当前缺失能力：
  - 缺少普通导出与问题包导出的明确边界。
  - 缺少资产选择能力。
  - 缺少大小上限与脱敏能力。
  - 缺少压缩与流式写出能力。
- 修复后平台必须提供的能力：
  - 普通导出和问题包导出分离。
  - 受控资产选择。
  - 大小上限。
  - 脱敏规则。
  - 压缩与流式写出。
  - 默认不带全量 Replay 资产。
- 所属层：
  - Product
  - Tooling
  - Adapter

### 4. 多模块接入时没有正式的用户 / 租户 / 权限隔离模型

- 当前缺失能力：
  - 缺少主体模型。
  - 缺少权限与作用域边界。
  - 缺少未来多模块扩展口。
- 修复后平台必须提供的能力：
  - 最小可用的 User / Role / Scope 模型。
  - 可扩展的访问控制边界。
  - 单机桌面场景的轻量接入方式。
  - 未来多角色、多模块的扩展口。
- 所属层：
  - Product
  - Storage Core
  - Tooling

### 5. 实现层查询、引用检查仍是全量扫描，10x / 100x 数据量下不可扩展

- 当前缺失能力：
  - 缺少后端下推查询。
  - 缺少必要索引。
  - 缺少引用检查的局部化机制。
  - 缺少与现有接口兼容的渐进演进路径。
- 修复后平台必须提供的能力：
  - 查询下推到存储后端。
  - 索引先行。
  - 引用检查避免全库扫描。
  - 兼容现有接口的渐进改造。
- 所属层：
  - Storage Core
  - Adapter
  - Product

## 二、版本升级体系修复方案

### 2.1 Version Graph

版本升级体系必须从单线升级改为 Version Graph。Version Graph 至少应表达以下关系：

1. 当前工程版本。
2. 可直接升级的目标版本。
3. 必须经过中间版本的跳转关系。
4. 当前版本是否仍受兼容窗口支持。
5. 哪些版本只能读不能写。

Version Graph 的作用不是记录历史口号，而是明确平台可执行的迁移边界。

### 2.2 forward-only 还是双向迁移策略

平台主策略应采用 forward-only 迁移。

理由如下：

1. 迁移实现和验证成本可控。
2. 版本图谱更容易稳定。
3. 降级逻辑容易破坏数据一致性。
4. 工程正式状态只能向前演进，更符合平台维护模型。

双向迁移不作为默认策略。若业务确实需要回退，只能通过显式备份恢复或显式导出再导入方式完成，不得把“降级”当作常规迁移能力。

### 2.3 启动升级流程

启动升级流程必须分成以下阶段：

1. 只读读取工程版本元数据。
2. 根据 Version Graph 判断是否需要升级。
3. 判断当前版本是否落在兼容窗口内。
4. 若需要升级，则进入显式升级前检查。
5. 生成升级计划。
6. 执行升级。
7. 写入升级结果与新版本标识。

启动升级流程必须满足：

1. 不得在 open 阶段隐式写入。
2. 不得跳过兼容性判断。
3. 不得在未确认前修改工程正式状态。

### 2.4 升级失败恢复

升级失败恢复必须提供两条路径：

1. 保留原工程，不提交升级结果。
2. 回到升级前可识别状态，并输出失败原因。

升级失败后，工程不得停留在不明版本状态。升级过程必须具备原子性语义，至少在平台层表现为“要么升级成功，要么保留原状态”。

### 2.5 兼容窗口

兼容窗口必须明确为有界区间，而不是永久兼容。

兼容窗口至少应定义：

1. 可读版本范围。
2. 可写版本范围。
3. 仅可导入版本范围。
4. 仅可诊断版本范围。

兼容窗口到期后，平台必须明确拒绝写入，必要时只保留读取或导入诊断能力。

### 2.6 历史工程支持策略

历史工程支持策略必须分层：

1. 在线可升级工程，允许启动时升级。
2. 只读历史工程，仅允许打开和诊断。
3. 归档历史工程，仅允许通过工具导入或转换。

历史工程支持不得依赖模糊口径，必须在启动前就能判断工程属于哪一类。

## 三、大数据导入修复方案

### 3.1 分块提交

大型模型、钢筋和批量数据导入必须采用分块提交。

分块提交要求：

1. 每一块有明确边界。
2. 每一块有独立提交结果。
3. 每一块可以单独恢复。
4. 每一块必须可记录 checkpoint。

导入过程不得默认把全部数据塞进单个大事务。

### 3.2 checkpoint

导入过程必须提供 checkpoint 机制。

checkpoint 至少应记录：

1. 已完成块编号。
2. 当前导入会话。
3. 上一次成功提交位置。
4. 待恢复的偏移位置。

checkpoint 的作用是让导入过程支持中断恢复，而不是让失败后整包重来。

### 3.3 幂等重试

导入必须支持幂等重试。

幂等重试要求：

1. 同一导入会话重复执行同一块时，不应产生重复数据。
2. 导入块必须有稳定块标识。
3. 重试必须能够识别已提交块与未提交块。

### 3.4 最终确认

导入过程必须有最终确认步骤。

最终确认的作用是：

1. 将分块导入的中间状态收敛为正式结果。
2. 把会话级导入标记为完成。
3. 让未确认导入保持可恢复而非正式生效。

最终确认必须显式，不得由最后一个分块自动替代。

### 3.5 局部失败恢复

导入局部失败时，平台必须支持以下策略之一：

1. 仅回退失败块。
2. 保留已成功块，标记会话待恢复。
3. 按策略重试失败块。

不能把局部失败默认升级成全量回滚，也不能让失败状态停留在不可判断区间。

### 3.6 与 Undo / Redo / Replay 的关系

导入过程与 Undo / Redo / Replay 的关系必须明确：

1. 导入不是普通单命令编辑，不应默认进入逐命令 Undo 链。
2. 导入应以导入会话为边界，而不是以每个子行输入为边界。
3. Replay 只负责记录和重放导入过程中的显式提交点，不负责还原所有中间内部状态。
4. Undo / Redo 只能作用于导入会话的正式提交边界，不得作用于每一条内部扫描行。

## 四、问题包导出修复方案

### 4.1 普通导出 vs 问题包导出边界

普通导出只面向工程交付，默认只包含 Project 正式状态。

问题包导出只面向诊断和问题复现，必须显式选择附带资产，不能默认把 Project、Replay、Snapshot、Log 全部打包。

### 4.2 资产选择

问题包导出必须显式支持以下资产选择：

1. Project。
2. 指定 Replay Session。
3. 指定 Snapshot。
4. Runtime Log。
5. 额外诊断附件。

资产选择必须支持按范围、按会话、按时间和按大小条件筛选。

### 4.3 大小上限

问题包导出必须支持大小上限。

若超过上限，必须提供：

1. 裁剪策略。
2. 仅导出摘要策略。
3. 分卷导出策略。

不得无限制打包。

### 4.4 脱敏规则

问题包导出必须支持脱敏。

至少应支持：

1. 路径脱敏。
2. 用户标识脱敏。
3. 敏感文本脱敏。
4. 诊断日志关键字过滤。

脱敏规则必须在导出前显式配置，不得靠默认猜测。

### 4.5 压缩与流式写出

问题包导出必须支持压缩与流式写出。

流式写出的目的不是性能优化口号，而是保证大包可导出、可中断、可恢复。

### 4.6 默认策略

默认策略必须是：

1. 普通导出只导出 Project。
2. 问题包导出不默认包含全部 Replay Assets。
3. 任何附带 Replay、Snapshot、Log 的行为都必须显式勾选。

## 五、主体隔离模型

### 5.1 是否需要 User / Role / Scope

需要，但要轻量化。

当前单机桌面产品不需要完整企业级 IAM，但必须预留主体隔离模型：

1. User：当前操作主体。
2. Role：主体能力集合。
3. Scope：主体可作用对象范围。

### 5.2 哪些能力必须留扩展口

必须预留的扩展口包括：

1. 单用户到多用户。
2. 单角色到多角色。
3. 本地工程到共享工程。
4. 本机权限到租户权限。
5. 工程级权限到对象级权限。

### 5.3 当前单机桌面产品如何最小化接入

当前阶段建议采用“轻主体 + 轻作用域”模型：

1. 默认 User 固定为当前登录主体或本机主体。
2. Role 先只区分管理员、普通用户、诊断用户。
3. Scope 先只区分工程、项目集、诊断包和 Replay 资产。

这样可以先满足隔离边界，又不把系统锁死在企业 IAM 的复杂实现里。

### 5.4 不破坏未来扩展的原则

1. 访问控制必须挂在边界上，不挂在 UI 上。
2. 权限判断必须可查询、可记录、可扩展。
3. 不得用硬编码角色名替代主体模型。
4. 不得把权限逻辑散落在各个命令里。

## 六、查询与引用检查扩展方案

### 6.1 哪些查询必须后端下推

以下查询必须后端下推：

1. 按 `recordId`、`projectId`、`sessionId`、`commitId` 查询。
2. 按单列等值、范围和前缀查询。
3. 按引用关系查询。
4. 按状态、版本、时间窗口查询。
5. 按分页、排序和过滤查询。

### 6.2 哪些索引必须先建

至少必须优先建立：

1. 主键索引。
2. `recordId` 索引。
3. `projectId` 索引。
4. `sessionId` 索引。
5. `commitId` 索引。
6. 引用列索引。
7. 常用过滤列索引。

### 6.3 引用检查如何避免全库扫描

引用检查必须从“全库扫描”改为“索引命中 + 受影响集合扫描”。

路线如下：

1. 为引用列建立正向索引。
2. 为常用被引用对象建立反向索引。
3. 删除和变更时只扫描受影响集合。
4. 在必要时才做降级全扫，并把降级路径显式标记为慢路径。

### 6.4 如何兼容现有接口

兼容现有接口的原则是：

1. 保留现有查询入口。
2. 内部先替换查询执行器。
3. 外部接口保持返回语义不变。
4. 逐步增加可选条件和分页参数，不强制一次性改全。

## 七、实施优先顺序

### 7.1 先设计

先完成以下设计文档收口：

1. Version Graph。
2. 导入会话与 checkpoint 模型。
3. 问题包导出边界。
4. 主体隔离模型。
5. 查询与引用检查目标模型。

### 7.2 再接口

再补齐平台接口：

1. 升级计划接口。
2. 导入会话接口。
3. 问题包导出策略接口。
4. 主体与作用域接口。
5. 查询与索引接口。

### 7.3 再适配层

再让 Adapter 接入：

1. 版本读取与迁移执行。
2. 分块导入落库。
3. 流式导出写出。
4. 索引查询和引用查询下推。

### 7.4 再工具 / 迁移

最后补工具和迁移能力：

1. 历史工程转换工具。
2. 问题包导出工具。
3. 导入恢复工具。
4. 迁移诊断工具。

## 八、路线图

### Phase P1-1

冻结 Version Graph 设计，明确 forward-only 迁移和兼容窗口。

### Phase P1-2

定义导入会话、checkpoint、幂等重试和最终确认模型。

### Phase P1-3

拆分普通导出与问题包导出，定义资产选择、上限和脱敏策略。

### Phase P1-4

引入轻主体模型和作用域模型，预留多用户、多角色扩展口。

### Phase P1-5

推动查询下推、索引建设和引用检查去全库扫描化。

## 九、最终结论

P1 修复的目标不是把功能做多，而是把平台能力补齐：升级要可判定、导入要可恢复、导出要可控、隔离要可扩展、查询要可伸缩。只要这五条能力成立，后续实现与产品接入才有稳定的演进基础。
