# Step1-3 优先级结论

基于 Step1（存储模块设计）、Step2（实现偏差）、Step3（产品设计文档使用审查），按“是否影响架构成立”划分优先级如下。

## P0

### 1. 设置项、主题样式、基础配置被纳入工程正式属性编辑链路

- 类型：设计问题 + 使用层问题
- 影响范围：工程属性编辑、保存、Undo / Redo、Replay、配置迁移
- 不修的后果：工程正式状态与偏好设置会互相污染，回放和恢复无法区分“业务变更”和“偏好变更”，最终导致状态不可复现。

### 2. 打开工程时自动升级、自动标记可录制，把“读工程”变成隐式写工程

- 类型：设计问题 + 使用层问题
- 影响范围：工程打开、自动恢复、批量打开、自动化测试、只读场景
- 不修的后果：任何加载动作都可能修改工程，破坏只读边界和幂等性，用户无法确认什么时候发生了写入。

### 3. Replay 只有粗粒度布尔判断，没有形成分支锚点、版本契约和可验证基线

- 类型：设计问题
- 影响范围：Replay、继续编辑、损坏恢复、跨版本审计
- 不修的后果：回放会和当前态混淆，无法可靠证明从哪个提交点分叉，Replay 失去作为可验证历史链路的意义。

## P1

### 1. 存储版本升级没有形成完整版本图谱

- 类型：存储能力问题
- 影响范围：启动迁移、版本升级、跨版本部署、历史工程兼容
- 不修的后果：一旦版本进入生产，升级只能靠人工兜底，后续每次格式演进都会放大维护成本和故障风险。

### 2. 大型模型 / 钢筋导入被默认当作单次可整体回滚事务

- 类型：设计问题 + 使用层问题
- 影响范围：钢筋导入、模型批量导入、工程迁移、长时间恢复
- 不修的后果：数据量上来后，单事务会带来内存膨胀、长锁、恢复时间过长，失败一次就可能拖垮整个导入流程。

### 3. 问题包导出把工程、Replay、Snapshot、Log 默认打包在一起

- 类型：设计问题 + 使用层问题
- 影响范围：问题包导出、客户交付、内部诊断、长工程共享
- 不修的后果：导出会变成无边界的数据复制任务，包体无限膨胀，后续还会把回放资产当成普通导出内容继续累积。

### 4. 多模块接入时没有正式的用户 / 租户 / 权限隔离模型

- 类型：存储能力问题
- 影响范围：多模块接入、角色访问、未来协作场景、共享工程
- 不修的后果：一旦进入多模块或多角色环境，就没有“谁能看、谁能改、谁能回放”的基础约束，跨边界污染会很快出现。

### 5. 实现层查询、引用检查仍是全量扫描

- 类型：实现问题
- 影响范围：大工程查询、删除引用检查、计算列联动、批量筛选
- 不修的后果：数据一大就会卡顿、内存暴涨、删除 / 校验变慢，平台级存储会退化成“能用但撑不住规模”。

## P2

### 1. 查询 API 仍偏最小可用集

- 类型：存储能力问题
- 影响范围：中层业务查询、复杂报表、诊断工具、二次开发
- 不修的后果：短期不影响主流程，但后续做分析工具、批量操作和诊断能力时会不断补洞。

### 2. 计算列失效和缓存清理不够精细

- 类型：实现问题
- 影响范围：计算列、增量重算、长工程运行
- 不修的后果：不一定立刻出错，但会逐步积累错误缓存和额外内存，占用越来越明显。

### 3. 内存后端与 SQLite 后端重复实现较多

- 类型：设计问题 + 实现问题
- 影响范围：后续维护、双后端一致性、回归修复
- 不修的后果：功能短期能跑，但 bug 很容易修一半漏一半，后续维护成本持续上升。

### 4. 观察者 / 通知链路缺少更强的生命周期防护

- 类型：实现问题
- 影响范围：多模块接入、长会话、Undo / Redo 通知
- 不修的后果：早期可能只是偶发崩溃或悬挂回调，规模一大就会变成稳定性问题。

## 结论

当前优先级判断是：

1. 先修 P0，确保工程状态、加载流程和 Replay 链路的边界成立。
2. 再规划 P1，补齐版本升级、隔离、导入导出和规模化能力。
3. 最后处理 P2，优化查询、缓存、重复实现和通知防护。
