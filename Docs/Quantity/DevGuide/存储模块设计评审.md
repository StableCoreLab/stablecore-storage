## 一、存储模块能力模型

### 1. 总体设计抽象

当前设计已经形成一条比较清晰的存储主链路：

- 以 `Database / Table / Record / Schema / EditSession / ChangeSet / JournalEntry` 作为核心抽象。
- 以“受控动态 Schema”作为数据建模前提，而不是完全自由的键值存储。
- 以事务为核心边界，围绕 `BeginEdit / Commit / Rollback` 组织写入语义。
- 以内存后端和 SQLite 后端并行演进，强调同一套核心语义可落到不同持久化载体。

这说明该模块不是单纯的存储容器，而是一个带有事务、历史、恢复、增量通知和可扩展计算能力的存储子系统。

### 2. 数据建模

已包含的能力可以概括为：

- **Schema 建模**
  - 表有正式 Schema。
  - 列必须注册后才能可靠访问。
  - 允许新增事实列，但仍纳入 Schema 管理。

- **列元信息**
  - 已明确列级元数据至少包含 `name / displayName / valueKind / columnKind / nullable / editable / indexed / referenceTable`。
  - 这意味着设计已经覆盖了最基本的列描述、可编辑性、索引提示和关系引用。

- **值模型**
  - `SCValue` 作为统一值容器，至少覆盖 `Null / Int64 / Double / Bool / String / RecordId / Enum`。
  - 设计上明确了类型访问与类型错误反馈，不把字符串伪装成布尔或枚举。

- **记录生命周期**
  - Record 不是删除即消失，而是通过 `Alive / Deleted` 的状态建模 tombstone。
  - 删除后的 `recordId` 不复用，旧句柄仍可识别到删除状态。

- **关系建模**
  - 关系列被单独看待，不简单退化成字符串字段。
  - 通过 `RecordId` 承载引用语义，避免关系表达失真。

- **计算列建模**
  - 计算列被显式区分为 `Expression / Rule / Aggregate` 等类型。
  - 计算列定义中包含依赖信息，支持后续缓存失效和增量重算。
  - 还区分了会话级计算列与更长期的持久化数据，避免把临时视图直接混入事实模型。

### 3. 事务模型

当前事务设计已经不是“简单批量写入”，而是一个有明确边界的编辑事务系统：

- **事务边界**
  - 一个用户动作通常对应一个事务。
  - 写操作必须处于活动事务内。
  - `EditSession` 是事务上下文的承载者。

- **事务状态**
  - 已定义至少 `Active / Committed / RolledBack` 三态。
  - `Database` 负责统一管理 `BeginEdit / Commit / Rollback`。

- **事务粒度的历史**
  - `ChangeSet` 以事务为单位表达提交后的变化集合。
  - `JournalEntry` 记录事务中的基础操作，用于持久化回放与 Undo/Redo。
  - 同一事务内允许合理聚合，但不能破坏回放正确性。

- **写入聚合**
  - 设计允许把同事务内重复写入做合并，减少噪声。
  - 但聚合的边界被明确约束在“不能牺牲 Undo/Redo 正确性”。

这套模型已经具备平台级事务最关键的能力：写入有边界、提交有摘要、回放有依据、撤销有锚点。

### 4. API 设计

当前 API 设计已经比较接近“稳定核心接口”：

- **核心对象 API**
  - `Database / Table / Record / Schema / ChangeSet / EditSession` 已被明确为对外核心抽象。
  - 这使上层调用不必直接依赖底层存储细节。

- **访问风格**
  - 推荐通过 `GetKind / AsXxx / FromXxx` 建立统一访问模式。
  - 错误以统一错误码返回，而不是依赖模糊布尔值。

- **查询 API**
  - V1 只提供最小查询能力：枚举记录、按单列等值条件查找、按 `recordId` 直接读取。
  - 推荐使用 `RecordCursor / QueryCondition` 这类轻量对象表达查询。

- **变更摘要 API**
  - `ChangeSet` 面向 UI 和上层逻辑作为提交结果摘要。
  - `JournalEntry` 面向持久化回放和正确性。

从设计角度看，API 风格已经从“功能堆叠”转向“语义分层”。

### 5. 数据隔离

从设计文档里可以拆出三个层次：

- **工程隔离**
  - 以单数据库实例 / 单文件作为基本承载边界。
  - 启动、恢复、迁移都围绕数据库实例展开。
  - 这意味着工程级隔离的主边界是“一个数据库会话或一个数据库文件”。

- **系统隔离**
  - 事实数据、Schema、Journal、元数据在职责上已经分层。
  - 计算列默认是会话级视图，不直接混入事实存储。
  - SQLite 持久化层被限定为承接持久化，不承担上层业务规则。

- **用户隔离**
  - 目前设计里没有看到正式的多用户、租户、权限隔离模型。
  - 现有更多是记录级别的生命周期隔离，而不是主体级隔离。
  - 也就是说，用户隔离仍然偏弱，更多是隐含能力，不是显式设计能力。

### 6. 扩展能力

设计中已经明确了多条可扩展路径：

- **计算能力扩展**
  - 规则注册通过 `ruleId` 机制外置。
  - 计算列支持表达式、规则、聚合三类场景。
  - 依赖信息显式化，便于缓存失效与重算。

- **查询能力扩展**
  - 当前只保留最小查询集，但设计中已经预留更复杂过滤、排序、分页、关系反查的方向。

- **后端扩展**
  - Storage 核心与 SQLite 持久化层分离，说明后端适配不是写死在核心模型里。

- **工具扩展**
  - 数据库编辑器被设计成研发验证工具，说明它可以持续承接模型演进与验证。

### 7. 版本升级能力

版本升级能力已经被纳入路线图，但还处于“有方向、未完全定型”的阶段：

- 已明确需要：
  - 迁移步骤。
  - 迁移计划接口。
  - 启动阶段依据版本信息决定是否执行升级。
  - 迁移逻辑与运行时业务逻辑解耦。

- 说明当前设计已经承认：
  - Schema、Journal、SQLite 物理结构都可能跨版本演进。
  - 不能只依赖人工清库或一次性脚本。

### 8. Undo / Redo / Replay 支持能力

这部分是当前设计最有平台味道的能力之一：

- **Undo / Redo**
  - 已明确按事务粒度处理。
  - 删除恢复要求保留 `recordId` 稳定性。
  - tombstone 让“已删除但可识别”成为一等状态。

- **Replay**
  - `JournalEntry` 面向回放正确性。
  - 启动时会加载持久化 Journal 和恢复 Undo/Redo 所需状态。
  - 这意味着系统已经具备“从历史重建状态”的基础路径。

- **ChangeSet 与 Replay 的分工**
  - `ChangeSet` 面向通知与摘要。
  - `Journal` 面向回放与持久化。
  - 两者职责不同，但可相互转换。

整体看，Undo/Redo/Replay 不是附属功能，而是当前存储设计中的主轴之一。

## 二、设计层面的缺口

下面只从平台架构完整性角度指出缺口，不讨论实现细节。

| 问题 | 风险等级 | 为什么是问题 |
|---|---|---|
| 没有正式的用户 / 租户 / 权限隔离模型 | 高 | 当前设计只看到记录级生命周期和会话级上下文，没有主体隔离边界。未来一旦进入多人协作、分角色访问或多租户场景，存储层会缺少“谁能看、谁能改、谁能回放”的基础约束。 |
| 系统级命名空间边界不够明确 | 中 | 事实数据、Schema、Journal、元数据已经分层，但缺少更正式的系统/用户对象命名空间约束。若后续扩展更多对象类型，容易出现元数据与业务对象相互污染。 |
| 版本升级方案还不完整 | 高 | 文档已经提到迁移接口和启动时决定是否升级，但没有看到版本图谱、升级顺序、可回滚策略、失败补偿和兼容窗口。版本一旦进入生产，这会直接影响可维护性。 |
| Replay 只有思路，没有完整契约 | 高 | 目前有 Journal 和启动恢复思路，但没有看到 Replay 的明确输入、输出、幂等性、检查点和部分回放规则。缺少这些契约时，回放在跨版本、损坏恢复或审计场景中容易变得不可验证。 |
| Undo / Redo 与计算列的边界不够统一 | 中 | 文档明确计算列不进入数据库级 Undo/Redo，但没有把“事实数据变更”和“会话视图变更”的一致性策略完全收紧。若上层同时依赖事实和计算结果，容易出现语义断层。 |
| ChangeSet 语义还不够形式化 | 中 | 目前知道它是提交后的摘要，但缺少更完整的版本字段、稳定排序、去重规则、来源追踪和兼容说明。ChangeSet 一旦要给 UI、审计、同步或外部订阅使用，这些字段会变成必要条件。 |
| Journal / ChangeSet 之间的映射规则还不够明确 | 中 | 文档说两者可相互转换，但没有把转换损失、聚合边界和不可逆信息说清楚。若后续把 ChangeSet 当成审计或同步载体，容易误把摘要当成完整历史。 |
| 查询能力仍然偏“最小集” | 中 | 目前只覆盖枚举、单列等值、按 `recordId` 读取。对于平台级存储来说，这足够起步，但还不足以支撑复杂分析、稳定分页、组合过滤和关系反查等中层平台需求。 |
| 约束模型不够完整 | 中 | 列元信息里有 `nullable / editable / indexed / referenceTable`，但缺少更系统的唯一性、默认值、级联、引用完整性、值域和跨列约束模型。没有这些约束，Schema 只能算“描述型”，还没到“强约束型”。 |
| Deleted / tombstone 的保留与清理策略未形成完整设计 | 中 | 已知删除后保留可识别状态，但没有看到保留周期、压缩/归档、回收条件和历史查询边界。时间一长，存储膨胀和语义混乱都会变成问题。 |
| 错误码统一了，但诊断契约还偏弱 | 低 | 目前强调统一错误码与清晰返回，但对错误上下文、关联对象、链路追踪和恢复建议没有形成更强的结构化契约。对功能正确性影响不大，但会影响大规模排障效率。 |
| SQLite 适配边界还需要更正式的职责声明 | 低 | 文档已说清 SQLite 只负责持久化，但没有把升级、迁移、校验、恢复、索引管理、错误映射的边界进一步形式化。短期可用，长期容易出现职责争议。 |
| 大规模与长期运行场景的工程性保障仍不足 | 中 | 路线图多次强调还需要补测试、边界场景和工程化能力，这说明设计已有主路径，但缺少面向长期维护的稳定性约束。作为平台底座，这会限制后续直接接入高强度业务。 |

## 三、结论

从设计完整性看，这个存储模块已经不是“只有存储”的状态，而是形成了以下几个比较完整的能力骨架：

- 以 Schema 和统一值模型支撑数据建模。
- 以事务、ChangeSet、Journal 支撑历史和回放。
- 以 SQLite 适配层支撑持久化后端。
- 以计算列和规则注册支撑可扩展表达能力。
- 以启动恢复和迁移思路支撑版本演进。

但从平台架构角度，它仍然有几个关键短板没有收口：

- 没有正式的用户/租户隔离。
- 版本升级和 Replay 契约还不够硬。
- 查询、约束、ChangeSet 语义还偏“基础可用”，没有到“平台级完备”。

如果后续目标是把它作为稳定平台底座，优先补的不是更多功能，而是：

1. 把升级、回放、恢复写成正式契约。
2. 把隔离、约束、命名空间做成显式模型。
3. 把 ChangeSet 和 Journal 的边界固定下来。
4. 把查询与诊断能力补到可长期维护的程度。
