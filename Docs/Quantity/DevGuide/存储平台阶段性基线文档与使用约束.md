# 存储平台阶段性基线文档与使用约束

## 1. 文档定位
本文档用于固化 `stablecore-storage` 当前阶段性能力边界，并对后续使用方式形成强约束。

本文档适用于：
- 平台能力基线确认
- 产品侧调用约束
- 业务侧使用边界约束
- 后续优化前的稳定参考基线

本文档不用于：
- 引入新架构
- 重新设计存储体系
- 展望未来实现细节
- 推导未落地能力

---

## 2. 当前平台能力总览

### 2.1 打开与版本体系
当前平台必须支持以下能力：
- `Version Graph`
- `CompatibilityWindow`
- `OpenDecision`
  - `ReadWrite`
  - `ReadOnly`
  - `UpgradeRequired`
  - `Unsupported`
- 禁止隐式升级
- 必须通过显式 `ExecuteUpgradePlan(...)` 执行升级

当前平台规则：
- 打开工程时不得自动升级。
- 只读打开不得产生持久化写入。
- 升级判断与升级执行必须分离。
- 异常关闭场景不得进入升级执行。

### 2.2 Import 体系
当前平台必须支持以下能力：
- `ImportSession`
- `ImportChunk`
- `ImportCheckpoint`
- `ImportFinalizeCommit`
- `ImportRecoveryState`

当前平台规则：
- `checkpoint` 不是正式状态。
- `finalize` 是唯一正式提交边界。
- `Undo / Redo / Replay` 只能绑定 `finalize`。
- `append chunk` 不得直接进入正式数据模型。

### 2.3 Export 体系
当前平台必须支持以下能力：
- 普通导出与问题包导出分离
- `AssetSelection`
- `RedactionPolicy`
- `PackageSizePolicy`
- `StreamingExportContext`
- 流式写出采用 `QSaveFile` 语义

当前平台规则：
- 普通导出默认仅包含 `Project`。
- 问题包导出默认不包含全部 `Replay Assets`。
- 问题包必须支持脱敏。
- 导出不得输出敏感明文配置。

### 2.4 Query 执行体系
当前平台必须支持以下能力：
- `QueryPlan`
  - `DirectIndex`
  - `PartialIndex`
  - `ScanFallback`
  - `Unsupported`
- `SCQueryBridge`
  - 仅负责归一化与结果适配
- `IQueryPlanner`
- `IQueryExecutor`
- 唯一执行入口 `ExecuteQueryPlan(...)`
- `QueryExecutionContext`
  - 必须包含 `backendKind`

当前平台规则：
- 旧查询接口必须通过 QueryPlan 进入执行链路。
- 不得绕过 `ExecuteQueryPlan(...)`。
- 不得新增第二套查询执行入口。
- 不得在产品侧直接拼接 SQL。

#### 执行路径收口机制
- `QueryPlan` 是唯一合法的查询描述。
- `ExecuteQueryPlan(...)` 是唯一合法的执行入口。
- 不允许存在第二套执行路径，包括优化路径、特判路径、临时路径。
- 任何绕过 `QueryPlan` 的执行路径必须视为架构违规。
- 所有 backend 执行能力必须通过 `registry / dispatch` 接入。
- 不允许直接调用 backend-specific executor。

### 2.5 Executor 分层
当前平台执行层必须分为：
- `Core executor`
  - 负责调度、归一化、分发
- `SQLite executor`
- `Memory executor`
- `registry / dispatch`

当前平台规则：
- Core executor 不得写死 backend 分支。
- Core executor 不得直接承载具体 backend 执行逻辑。
- backend 选择必须通过 registry / dispatch 机制完成。

### 2.6 Fallback 扫描体系
当前平台必须支持 fallback 扫描作为正式执行路径。

当前平台规则：
- fallback 仅能在 backend executor 内部实现。
- fallback 必须支持分页。
- fallback 必须支持统计。
- fallback 必须支持可观测结果回填。
- Core executor 不得实现 fallback。

### 2.7 ReferenceIndex（只读）
当前平台必须支持 `IReferenceIndexProvider` 的只读接入。

当前平台规则：
- 必须支持按 source / target 查询。
- 删除前引用检查必须先查 `ReferenceIndex`，缺失时再 fallback 扫描。
- 不得在 Query 路径写入 `ReferenceIndex`。
- 不得实现 `ReferenceIndex` 重建或增量维护。

### 2.8 可观测性体系
当前平台必须提供 `QueryExecutionResult` 的完整可观测字段：
- `mode`
- `unsupportedSource`
- `fallbackSource`
- `fallbackReason`
- `scannedRows`
- `matchedRows`
- `returnedRows`
- `usedIndexIds`
- `fallbackTriggered`
- `executionNote`

引用检查结果必须包含以下可观测信息：
- 是否命中 `ReferenceIndex`
- 是否触发 fallback
- fallback 原因（`fallbackReason`）

上述信息必须用于诊断、性能分析与索引优化。

当前平台规则：
- 执行结果必须能够反映实际路径。
- 执行失败时结果结构不得缺失。

## 3. 执行链路说明

### 3.1 查询链路
```text
旧查询入口
  -> SCQueryBridge 归一化
  -> IQueryPlanner 生成 QueryPlan
  -> ExecuteQueryPlan(...)
  -> Core executor 调度
  -> SQLite / Memory executor 执行
  -> QueryExecutionResult 回填
```

说明：
- 旧接口仍可使用。
- 旧接口不得绕过 QueryPlan。
- 执行器由 registry / dispatch 选择。

### 3.2 Import 链路
```text
ImportSession Begin
  -> Append Chunk
  -> Checkpoint
  -> Finalize
  -> 正式数据生效
```

说明：
- checkpoint 仅表示恢复点。
- finalize 才是正式提交边界。
- `Undo / Redo / Replay` 只绑定 finalize。

### 3.3 Export 链路
```text
导出请求
  -> 判断 ExportMode
  -> 选择 AssetSelection
  -> 应用 RedactionPolicy
  -> 按 PackageSizePolicy 控制
  -> StreamingExportContext 写出
```

说明：
- 普通导出与问题包导出必须分离。
- 默认问题包不得包含全部 Replay Assets。

### 3.4 引用检查链路
```text
删除前检查
  -> ReferenceIndexProvider 按 source / target 查询
  -> 若缺失，再走 fallback 扫描
  -> 返回诊断结果
```

说明：
- 只读引用检查不得写索引。
- fallback 仅是补救路径。

## 4. 平台强约束

### 4.1 查询体系约束
- 不允许绕过 `ExecuteQueryPlan(...)`。
- 不允许直接调用 backend 执行逻辑。
- 不允许新增第二套执行入口。
- 不允许 Core executor 写死 backend 分支。
- 不允许在 Core 层实现 fallback。

### 4.2 Fallback 约束
- fallback 必须分页。
- fallback 必须可统计。
- fallback 不允许无边界扫描。
- fallback 不允许写入任何数据。
- fallback 必须支持最大扫描行数限制（hard limit）。
- fallback 必须支持可中止（cancel / abort）。
- fallback 超过阈值必须中断或返回受限结果。
- 不允许无限扫描。
- fallback 行为必须可诊断。
- fallback 是受控执行路径，不是无限兜底机制。
- fallback 不得成为默认高频路径。
- fallback 成本必须可观测，并应配合 `QueryExecutionResult` 进行记录。

### 4.3 ReferenceIndex 约束
- 不允许在 Query 路径写入 `ReferenceIndex`。
- 不允许在 fallback 中更新 `ReferenceIndex`。
- 不允许在 checkpoint 阶段更新引用关系。
- `ReferenceIndex` 只反映正式 `Commit` 状态。

### 4.4 Import / Commit 约束
- `append chunk` 不属于正式数据。
- `finalize` 是唯一写入边界。
- 不允许在非 `finalize` 状态更新索引或引用关系。

### 4.5 Export 约束
- 默认问题包不包含 Replay Assets。
- 必须支持脱敏。
- 不允许导出敏感明文配置。

### 4.6 查询执行入口强制约束
- 所有查询执行必须通过 `ExecuteQueryPlan(...)`。
- 不允许直接调用 SQLite executor 或 Memory executor。
- backend executor 不得对外暴露执行入口。
- Storage 内部不得存在绕过 `QueryPlan` 的执行路径。
- 违反本约束属于架构违规。
- 本约束为强制约束（Hard Constraint），不属于建议项。
- 本约束适用于 `Storage Core`、`Adapter`、`Tooling`、`Product` 调用层。

## 5. 产品使用约束

### 5.1 查询使用方式
- 产品侧必须优先使用现有接口，如 `FindRecords`。
- 产品侧不得拼接 SQL。
- 产品侧不得依赖 fallback 性能作为主路径保障。
- 查询行为必须由平台内部自动转换为 QueryPlan。

### 5.2 删除操作
- 删除前必须走引用检查接口。
- 不得自行扫描判断引用。
- 不得绕过 `ReferenceIndexProvider` 语义。
- 调用方不得假设引用检查一定基于索引。
- 调用方必须允许 fallback 路径存在。

### 5.3 导入流程
- 大数据导入必须走 `ImportSession`。
- 不得使用单次大事务直接导入大批量数据。
- checkpoint 只能用于恢复，不得当作正式完成。

### 5.4 导出流程
- 普通导出默认仅包含 `Project`。
- 问题包默认仅包含 `Project + Diagnostics`。
- Replay Assets 若需导出，必须显式勾选，不得默认带出。

### 5.5 升级流程
- 打开工程不得自动升级。
- 必须显式执行升级计划。
- 升级失败不得留下隐式写入副作用。

---

## 6. 当前不支持能力
当前平台明确不支持以下能力：
- 多表查询 / join
- `group by` / `having`
- 子查询
- 分布式查询
- `ReferenceIndex` 写入 / 重建
- SQL 深度下推优化
- 自动索引选择与优化
- fallback 并行执行

---

## 7. 后续演进方向
以下仅为演进方向，不构成当前实现承诺：
- SQLite 查询下推优化
- `ReferenceIndex` 写入与增量维护
- 索引体系增强
- 更复杂查询能力，如多表与聚合
- 权限模型扩展

---

## 8. 使用结论
- 本平台当前已经形成稳定基线，后续开发必须在现有边界内演进。
- 任何新增能力必须先确认是否破坏以下边界：
  - `Project / System Config / User Config / Replay Assets / Debug Package`
  - `open / save / upgrade / Import / Export / Query / Reference` 的动作语义
  - `Undo / Redo / Replay` 的正式提交边界
- 任何超出本文档的不明确调用方式，必须视为不允许。
