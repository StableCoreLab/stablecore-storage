# Phase 1：P0 设计修复方案

本阶段只修复 P0 设计问题。目标不是补功能，而是先把架构边界立住，避免后续所有实现都建立在错误的状态模型上。

## 一、要修复的问题

### 1. 设置项、主题样式、基础配置被纳入工程正式属性编辑链路

- 问题类型：设计修复
- 现状问题：工程数据、用户配置、系统配置的边界不清，设置项容易被当成工程正式状态参与保存、Undo / Redo 和 Replay。
- 为什么必须先修：这会直接污染工程历史，使“正式业务状态”和“偏好配置”混入同一条变更链路。

### 2. 打开工程时自动升级、自动标记可录制，把“读工程”变成隐式写工程

- 问题类型：设计修复
- 现状问题：open 流程里混入升级和录制动作，导致只读加载和写入升级没有明确分离。
- 为什么必须先修：只要加载动作可能写入，工程边界和幂等性就不成立。

### 3. Replay 只有粗粒度布尔判断，没有形成分支锚点、版本契约和状态模型

- 问题类型：设计修复
- 现状问题：Replay 能否使用只靠几个布尔值判断，缺少可验证的锚点、版本边界和状态分层。
- 为什么必须先修：没有锚点和版本契约，Replay 就无法作为可审计、可分叉、可继续编辑的历史链路。

## 二、修复后的目标状态

### 1. 边界清晰

- 工程数据（Project）：只承载工程正式状态。
- 系统配置（System Config）：只承载系统级偏好和运行配置。
- 用户配置（User Config）：只承载用户级偏好和个性化设置。
- Replay 资产（Journal / Snapshot / Session）：只承载历史链路、恢复材料和审计数据。
- 问题包（Debug Package）：只承载诊断所需的受控导出资产。

### 2. 写入行为显式化

- 所有会改变工程正式状态的动作都必须显式进入事务或命令边界。
- open 流程只负责加载和校验，不得隐式升级、不待读即写。
- 任何升级动作必须由独立命令或明确确认后的写事务触发。

### 3. Replay 模型可验证

- Replay 必须有明确锚点：`CommitId` / `SnapshotId` / `SessionId` 至少一种主锚点。
- Replay 必须有版本契约：工程格式版本、Replay 格式版本、Journal 格式版本要可判断兼容性。
- Replay 必须有状态模型：可用、缺失、部分可用、损坏、过期、不可跨版本使用等状态要显式建模。

## 四、具体修复方案

### 1. 重新定义三类配置与工程状态的归属

- 将设置项拆成两条线：
  - 工程正式状态：只进入 Project / ChangeSet / Journal。
  - 配置状态：进入 System Config 或 User Config，不进入工程 Undo / Replay 主链路。
- 明确主题样式只属于表现层配置，不默认属于工程正式状态。
- 如果某个主题项确实影响业务语义，必须单独定义为工程元数据，而不是混入普通设置项。

### 2. 拆分 open 与 upgrade

- `Open Project` 只做加载、校验、兼容性判断、只读分析。
- `Upgrade Project` 作为独立动作，必须满足：
  - 明确的用户确认或显式调用。
  - 明确的前置检查。
  - 明确的回滚或失败恢复策略。
- 打开工程后自动进入可录制状态的行为取消，改为由显式动作决定是否创建新 Session 或开始录制。

### 3. 重构 Replay 状态模型

- 统一定义 Replay 资产状态：
  - `Available`
  - `Partial`
  - `Broken`
  - `VersionMismatch`
  - `Expired`
  - `NotRecordable`
- 对每个状态都要定义：
  - 触发条件
  - 可执行动作
  - 是否允许继续编辑
  - 是否允许创建新分支
  - 是否允许导出问题包
- 仅靠布尔值判断的策略全部替换为状态机式判断。

### 4. 统一 Replay 锚点与分支规则

- 继续编辑必须显式绑定分支锚点，禁止仅凭“当前屏幕状态”开启新会话。
- 每次从回放状态继续编辑，都必须记录：
  - 来源 `CommitId`
  - 来源 `SnapshotId`
  - 新建 `SessionId`
  - 分支原因
- Replay 回放入口必须先判断锚点是否存在，再判断版本是否兼容，最后判断资产是否完整。

### 5. 收紧问题包边界

- 问题包导出改成受控导出模型：
  - 可选项明确列出：工程、Journal、Snapshot、Log、额外诊断附件。
  - 不允许默认把所有 Replay 资产全带走。
  - 必须有大小上限、脱敏规则和导出策略。
- 问题包只作为诊断资产，不作为工程正式状态的替代品。

## 五、需要同步调整的上下游

- 上游设计：
  - 工程生命周期设计
  - 属性编辑类命令设计
  - 主题样式与显示规范设计
  - Replay 录制 / 启动策略设计
  - 问题包导出设计
- 下游能力：
  - 存储层的事务边界
  - Journal / Replay 数据模型
  - 版本迁移判定
  - 自动化测试中的工程打开流程
  - 诊断与恢复工具

## 六、风险与注意事项

- 这一步是设计收口，不是功能堆叠，不能为了兼容把所有东西继续塞回工程状态。
- 任何“自动修复”“自动升级”“自动开始录制”都要特别谨慎，优先保证显式性。
- Replay 的锚点、版本契约和状态模型必须一起改，不能只改一个字段或一个布尔值。
- 主题样式和系统配置如果继续混入工程，将直接破坏后续 P1 的隔离能力。
- 本阶段先确保边界成立，再进入存储能力和实现修复。
