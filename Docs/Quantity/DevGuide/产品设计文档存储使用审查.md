## 一、误解

### 1. 把工程信息和基础配置一起纳入正式属性编辑，容易把设置项误当成工程正式状态

问题描述：`属性编辑类命令设计`把“工程信息编辑”与“基础配置”放在同一类正式属性编辑里，并且又明确属性编辑会进入回放和 Storage，这会让读者默认所有设置项都应和工程正式数据一样落库、可撤销、可回放。

所在设计位置：`D:\code\stablecore\Quantity\Docs\Design\属性编辑类命令设计.md:10`，`D:\code\stablecore\Quantity\Docs\Design\属性编辑类命令设计.md:47`，`D:\code\stablecore\Quantity\Docs\Design\属性编辑类命令设计.md:53`，`D:\code\stablecore\Quantity\Docs\Design\属性编辑类命令设计.md:129`

为什么是问题（结合 Step1/2）：Step1/2 的 Storage 设计已经把正式工程状态、回放过程资产、运行配置分成不同职责域。这里如果把“基础配置”也按正式业务变更处理，就会把本应属于配置层的内容塞进工程历史和 ChangeSet，导致正式状态与偏好设置混在一起。

影响范围：工程级元数据、属性面板、设置项编辑、回放记录、Undo / Redo。

严重程度：高

修正建议：把设置项拆成两类接口和两类持久化目标。工程正式数据继续走 Storage 的事务 / ChangeSet；用户偏好和系统设置走独立 Config / Profile 体系，不进入工程 Undo 链和回放主链路。

### 2. 主题样式的持久化和配置管理没有明确落点，容易被误放进工程文件或回放资产

问题描述：`A-24` 明确说主题样式只决定表现参数，但又提到“主题持久化和配置管理”以及“主题参数配置化”。如果不进一步区分配置归属，使用层很容易把主题当成工程状态的一部分。

所在设计位置：`D:\code\stablecore\Quantity\Docs\Architecture\A-24主题样式与显示规范设计.md:9`，`D:\code\stablecore\Quantity\Docs\Architecture\A-24主题样式与显示规范设计.md:185`，`D:\code\stablecore\Quantity\Docs\Architecture\A-24主题样式与显示规范设计.md:193`，`D:\code\stablecore\Quantity\Docs\Architecture\目录定义.md:507`

为什么是问题（结合 Step1/2）：Step1/2 里的 Storage 适合正式工程数据和回放过程数据，不适合承载纯展示偏好。主题如果跟工程文件或 Replay 绑定，会造成不同工程之间的视觉偏好串味，也会让保存、另存为和导出多出无意义的差异。

影响范围：主题切换、默认样式、工程复制、问题包导出、回放展示。

严重程度：中

修正建议：明确主题配置只属于应用级或用户级配置，不进入工程正式文件和 Replay 资产。只有确实影响工程语义的显示模式才允许单独作为工程元数据保存。

## 二、误用

### 3. 打开工程时自动升级为可录制工程，把“读工程”变成了隐式写工程

问题描述：`启动前界面与录制回放策略设计`在打开工程的分支里，允许根据策略自动创建 baseline snapshot、标记工程可录制并启动录制。这实际上把打开动作变成了隐式写入动作。

所在设计位置：`D:\code\stablecore\Quantity\Docs\Architecture\启动前界面与录制回放策略设计.md:674`，`D:\code\stablecore\Quantity\Docs\Architecture\启动前界面与录制回放策略设计.md:684`，`D:\code\stablecore\Quantity\Docs\Architecture\启动前界面与录制回放策略设计.md:690`，`D:\code\stablecore\Quantity\Docs\Architecture\启动前界面与录制回放策略设计.md:865`，`D:\code\stablecore\Quantity\Docs\Architecture\启动前界面与录制回放策略设计.md:878`

为什么是问题（结合 Step1/2）：Step1/2 里的 Storage 事务语义要求明确的写边界。打开工程按理应先完成只读加载和兼容性判断，再由显式命令决定是否写入。把升级和录制塞进 open 流程，会让“只想看一眼”的动作也触发落盘。

影响范围：工程打开、文件 dirty 状态、自动化测试、批量打开、回放目录写入。

严重程度：高

修正建议：把“打开工程”与“升级为可录制工程”拆成两个显式动作。open 只负责加载和校验；升级必须是独立命令或明确确认后的写事务，不能由默认打开路径偷偷完成。

## 三、错误假设

### 4. 把回放可用性简化成几个布尔值，忽略了资产完整性和版本兼容性

问题描述：启动策略里用 `isRecordable`、`hasReplayData`、`replayDataBroken` 这几个布尔值来决定是否继续录制、是否自动升级、是否进入回放。但回放资产实际是多件套，不是“有 / 没有 / 坏了”三种状态就能覆盖。

所在设计位置：`D:\code\stablecore\Quantity\Docs\Architecture\启动前界面与录制回放策略设计.md:395`，`D:\code\stablecore\Quantity\Docs\Architecture\启动前界面与录制回放策略设计.md:397`，`D:\code\stablecore\Quantity\Docs\Architecture\启动前界面与录制回放策略设计.md:399`，`D:\code\stablecore\Quantity\Docs\Architecture\启动前界面与录制回放策略设计.md:684`，`D:\code\stablecore\Quantity\Docs\Architecture\启动前界面与录制回放策略设计.md:687`

为什么是问题（结合 Step1/2）：Step1/2 中已经把 Journal、Snapshot、日志、工程正式状态拆开了。只靠几个布尔值无法表达“Journal 存在但 Snapshot 缺失”“回放格式版本不匹配”“部分会话损坏但还能恢复”等中间态，策略判断会过于粗糙。

影响范围：回放入口显示、打开工程后的录制决策、损坏工程恢复、内部版诊断流程。

严重程度：高

修正建议：把工程元数据改成分项状态对象，至少区分工程格式版本、Replay 格式版本、Journal 完整性、Snapshot 完整性、日志可用性和可恢复性。策略层不要直接依赖单个布尔值做最终判断。

### 5. 大型模型 / 钢筋导入被默认当成一次可整体回滚的事务，低估了数据量和恢复成本

问题描述：导入导出设计把导入视为正式事务，通常还要求失败回滚；但对大工程、钢筋模型、批量模板导入来说，这个前提过于乐观，容易把超大数据一次性塞进单事务边界。

所在设计位置：`D:\code\stablecore\Quantity\Docs\Design\导入导出类命令设计.md:36`，`D:\code\stablecore\Quantity\Docs\Design\导入导出类命令设计.md:39`，`D:\code\stablecore\Quantity\Docs\Design\导入导出类命令设计.md:132`，`D:\code\stablecore\Quantity\Docs\Design\计算类命令设计.md:30`，`D:\code\stablecore\Quantity\Docs\Design\计算类命令设计.md:32`

为什么是问题（结合 Step1/2）：Step1/2 的 Storage 事务和 ChangeSet 适合明确的提交边界，但大模型导入、钢筋导入和批量修正往往需要分段落盘、断点恢复和增量回滚。把它们硬塞成单个大事务，会让内存、时间和失败恢复成本同时放大。

影响范围：钢筋数据导入、模型批量导入、模板导入、工程迁移、长时间恢复。

严重程度：高

修正建议：导入必须支持分段处理、可恢复检查点和幂等重试。大数据导入不要只设计成单次全量事务，至少要明确“分块提交 + 最终汇总确认”的方案。

### 6. 把问题包导出当成普通导出看待，容易无上限打包回放、快照和日志

问题描述：导入导出设计把工程打包、问题包导出和普通导出放在同一命名空间里，并允许附带回放会话、快照和日志。如果没有单独的大小、脱敏和保留策略，使用层会默认“打包就全带上”。

所在设计位置：`D:\code\stablecore\Quantity\Docs\Design\导入导出类命令设计.md:55`，`D:\code\stablecore\Quantity\Docs\Design\导入导出类命令设计.md:132`，`D:\code\stablecore\Quantity\Docs\Design\导入导出类命令设计.md:136`，`D:\code\stablecore\Quantity\Docs\Design\导入导出类命令设计.md:185`，`D:\code\stablecore\Quantity\Docs\Design\导入导出类命令设计.md:190`

为什么是问题（结合 Step1/2）：Step1/2 里 Replay、Snapshot、日志本来就是附加资产，不是正式工程本体。大工程下这些资产增长很快，如果导出语义不区分普通导出和问题包导出，就会让导出操作变成一个巨大的存储拷贝任务。

影响范围：问题复现包、客户交付包、内部诊断包、长工程的导出和分享。

严重程度：中

修正建议：把问题包导出独立成受控模式，支持资产选择、大小上限、脱敏规则、压缩和流式写出。普通导出默认不要把 Replay / Snapshot / Log 全部带走。

### 7. 回放继续编辑时缺少明确的分支锚点，容易把“当前状态”误当成可复现基线

问题描述：`启动前界面与录制回放策略设计`允许回放结束后继续编辑，并提出可以“创建新 session，从当前状态开始录制”。如果没有明确的 CommitId / SnapshotId 作为锚点，这个“当前状态”很容易和可复现状态混淆。

所在设计位置：`D:\code\stablecore\Quantity\Docs\Architecture\启动前界面与录制回放策略设计.md:581`，`D:\code\stablecore\Quantity\Docs\Architecture\启动前界面与录制回放策略设计.md:595`，`D:\code\stablecore\Quantity\Docs\Architecture\JournalReplay数据模型设计.md:62`，`D:\code\stablecore\Quantity\Docs\Architecture\JournalReplay数据模型设计.md:69`

为什么是问题（结合 Step1/2）：Step1/2 的 Storage 事务是有提交单元和版本号的，回放又依赖这些提交单元重建状态。若继续编辑没有明确记录从哪个提交点分叉，后续的回放和审计就很难把新会话和旧会话准确关联起来。

影响范围：回放分支、继续录制、问题复现、自动化测试断点、会话比较。

严重程度：中

修正建议：继续编辑必须显式落在一个分支锚点上，至少保存 `CommitId`、`SnapshotId` 或等价的基线标识，再创建新的 `SessionId`。不要只凭“当前屏幕状态”直接开新会话。

## 四、结论

这批产品设计文档对 Storage 的总体方向是对的，但在使用层上有几个明显风险：

1. 把设置项和工程正式状态混在了一起。
2. 把打开工程和写入升级混在了一起。
3. 把回放资产状态简化得过于粗糙。
4. 对大工程、钢筋、模型和问题包的体积与恢复成本估计偏乐观。

如果后续要把这些文档直接交给实现团队，建议先把“工程数据 / 用户设置 / 系统设置 / Replay 资产 / 问题包”这五类边界重新收紧，再进入开发。
