# 编辑数据库 Undo / Redo 与历史基线接口开发文档

> 本文仅描述 Storage 层能力，不包含产品层工程目录、工作副本调度、文件替换策略等内容。
> 本文的目标是把数据库对象在以下五类能力上的语义收敛清楚：
> - 获取当前编辑数据库的 Undo / Redo 列表
> - 以 `Normal / NoHistory / ReadOnly` 三种方式打开
> - 生成一致性备份副本
> - 查询当前编辑数据库状态
> - 重设置保存基线

## 1. 文档目标

本文只面向 Storage 内部实现与接口对齐，不描述产品层如何组织工程文件，也不描述 UI 如何呈现。

本文要解决的是数据库对象本身的语义问题：

- 当前编辑数据库如何暴露可见历史
- 三种打开方式各自允许什么、禁止什么
- 备份副本如何生成
- 如何查询当前状态
- 如何把当前状态收敛为新的保存基线

本文不关注：

- 工程目录如何创建
- 备份文件如何替换到最终路径
- UI 如何提示保存、另存为、关闭
- 产品层如何管理临时目录

---

## 2. 设计边界

### 2.1 Storage 层职责

Storage 层只负责数据库语义，不负责产品层文件组织。

Storage 层必须提供：

- Undo / Redo 可见历史
- 三种打开方式
- 一致性备份副本
- 当前状态查询
- 历史基线重置

### 2.2 明确禁止

- 不得把工程目录策略写进接口契约
- 不得把工作副本调度写成 Storage 的业务语义
- 不得让读操作触发写入
- 不得让一个 backend 暴露比另一个 backend 更多的可见语义

### 2.3 本文适用对象

本文只覆盖数据库对象的公开能力，不覆盖：

- Query
- Computed
- Migration 的版本图细节
- 产品层目录管理
- GUI 行为

---

## 3. 核心抽象

### 3.1 当前编辑数据库

本文中的“当前编辑数据库”是指一个已经打开、可以接收编辑操作的 Storage 数据库对象。

建议统一以下概念：

- `OpenTarget`：当前打开的数据库对象
- `EditingState`：当前编辑中的数据库状态
- `SaveBaseline`：当前保存点对应的历史基线
- `VisibleHistory`：当前对外可见的 Undo / Redo 列表

这里不引入产品层“工程”“工作副本”“目录替换”等命名。

### 3.2 历史与日志

必须区分：

- `SCEditLogState`：给调用方展示的可见编辑日志
- 内部恢复日志：给恢复、诊断、回放使用

禁止把完整内部日志直接暴露成可见历史。

### 3.3 三种打开方式

#### `Normal`

- 支持写入
- 保留可见历史
- 保留恢复所需日志

#### `NoHistory`

- 支持写入
- 不保留用户可见历史
- `GetEditLogState()` 必须返回空

#### `ReadOnly`

- 不允许任何写入
- 不允许历史变更
- 不允许保存基线重置

## 4. 公共接口草案

本节定义 Storage 公共接口的草案形态，目的是先把语义和边界定死，便于后续落实现有 Memory / SQLite 两端。

### 4.1 打开选项草案

```cpp
enum class SCDatabaseOpenMode
{
    Normal,
    NoHistory,
    ReadOnly,
};

struct SCOpenDatabaseOptions
{
    SCDatabaseOpenMode openMode{SCDatabaseOpenMode::Normal};
};
```

#### 语义说明

- `openMode` 决定是否保留可见日志、是否允许写入、是否允许基线重置
- `Normal` 是默认的完整语义模式
- `NoHistory` 适合不需要对外展示 Undo / Redo 的场景
- `ReadOnly` 适合只读检查、诊断、浏览

#### 打开入口草案

```cpp
ErrorCode CreateInMemoryDatabase(
    const SCOpenDatabaseOptions& options,
    SCDbPtr& outDatabase);

ErrorCode CreateFileDatabase(
    const wchar_t* path,
    const SCOpenDatabaseOptions& options,
    SCDbPtr& outDatabase);
```

#### 打开入口约束

- 所有新打开入口都应显式接收 `SCOpenDatabaseOptions`
- 默认值应保持与当前实现兼容
- `ReadOnly` 只表示打开后禁止写入，不表示自动进入任何产品层工作流
- `NoHistory` 只表示不对外暴露可见日志，不表示禁用编辑能力
- 对外不暴露底层持久化实现方式，`CreateFileDatabase` 只表达“文件型数据库对象”的公开语义

### 4.2 备份副本草案

```cpp
struct SCBackupOptions
{
    bool preserveJournalHistory{false};
    bool vacuumTarget{true};
    bool validateTarget{true};
    bool overwriteExisting{false};
};

struct SCBackupResult
{
    std::uint64_t removedJournalTransactionCount{0};
    std::uint64_t removedJournalEntryCount{0};
    std::uint64_t outputFileSizeBytes{0};
};
```

#### 语义说明

- `CreateBackupCopy` 仅作为文件后端能力，目标是生成一份一致性副本
- `preserveJournalHistory = true` 表示尽量保留 `journal_transactions` / `journal_entries`
- `vacuumTarget = true` 表示允许在副本上执行 `VACUUM`
- `validateTarget = true` 表示允许对副本执行完整性校验
- `overwriteExisting = true` 表示允许覆盖目标文件

### 4.3 编辑日志视图草案

```cpp
enum class SCEditLogActionKind
{
    Unknown,
    Commit,
    Undo,
    Redo,
    Import,
    RuleWriteback,
    SaveBaseline,
};

struct SCEditLogEntry
{
    CommitId commitId{0};
    VersionId version{0};
    SCEditLogActionKind kind{SCEditLogActionKind::Unknown};
    std::wstring displayText;
    std::wstring detailText;
    std::uint64_t timestampUtcMs{0};
};

struct SCEditLogState
{
    VersionId baselineVersion{0};
    std::vector<SCEditLogEntry> undoItems;
    std::vector<SCEditLogEntry> redoItems;
};
```

#### 约定

- `undoItems` 和 `redoItems` 都按调用方可直接展示的顺序返回
- `NoHistory` 模式下两个列表必须为空
- `ReadOnly` 模式下可以返回只读视图，但不得引入任何写入副作用
- 本节重点是“编辑日志视图”，不是内部审计日志，也不是后台持久化实现细节

### 4.4 当前编辑状态草案

```cpp
struct SCEditingDatabaseState
{
    bool open{false};
    bool dirty{false};
    SCDatabaseOpenMode openMode{SCDatabaseOpenMode::Normal};
    VersionId currentVersion{0};
    VersionId baselineVersion{0};
    std::size_t undoCount{0};
    std::size_t redoCount{0};
};
```

#### 说明

- `open` 表示当前数据库对象处于可用状态
- `dirty` 表示相对当前保存基线已经存在可保存变更
- `currentVersion` 表示当前编辑中的版本标识
- `baselineVersion` 表示当前保存基线版本
- `undoCount` / `redoCount` 只是摘要，不替代完整历史列表
- `openMode` 与打开选项保持一致

### 4.5 对外接口草案

```cpp
class ISCDatabase : public virtual ISCRefObject
{
public:
    virtual ErrorCode CreateBackupCopy(
        const wchar_t* targetPath,
        const SCBackupOptions& options,
        SCBackupResult* outResult) = 0;

    virtual ErrorCode ResetHistoryBaseline(SCBackupResult* outResult = nullptr) = 0;

    virtual ErrorCode GetEditLogState(SCEditLogState* outState) const = 0;

    virtual ErrorCode GetEditingState(SCEditingDatabaseState* outState) const = 0;
};
```

#### 接口说明

- `CreateBackupCopy` 负责一致性副本生成，仅文件后端实现
- `ResetHistoryBaseline` 负责保存基线收敛
- `GetEditLogState` 负责返回可见 Undo / Redo 列表
- `GetEditingState` 负责返回当前编辑数据库摘要状态

> 说明：这里将“当前状态查询”表述为 `GetEditingState`，避免引入产品层工作副本语义。

## 5. 行为契约

### 5.1 `CreateBackupCopy`

#### 目标

生成一份可重新打开的、一致性的数据库副本。

#### 必须满足

- 备份过程中不得破坏当前数据库对象
- 备份结果必须是一份完整数据
- 失败时不得留下半成功结果供调用方误用
- 不得通过备份修改当前可见历史的语义

#### 允许的实现

- Memory backend：不支持 `CreateBackupCopy`，必须返回 `SC_E_NOTIMPL`
- SQLite backend：从当前数据库生成一致性副本

#### 失败场景

- `targetPath` 无效
- 目标位置不可写
- 当前处于 `ReadOnly`
- 底层 I/O 失败
- 数据一致性检查失败

### 5.2 `ResetHistoryBaseline`

#### 目标

把当前状态收敛为新的保存基线。

#### 必须满足

- 成功后新的 `baselineVersion` 必须与当前保存点一致
- 旧基线之前的可见历史必须被清理或折叠
- 不得修改用户数据内容本身，只能修改历史边界和基线边界
- `ReadOnly` 模式下必须失败

#### 约束

- `Commit()` 仍然只表示单次编辑事务提交
- `ResetHistoryBaseline()` 只负责保存基线收敛
- 不能把两个语义混成一个动作

### 5.3 `GetEditLogState`

#### 目标

返回当前编辑数据库可见的 Undo / Redo 列表。

#### 必须满足

- 只返回可见历史，不返回内部审计日志
- 不得因为读取而触发写入
- 不得因为读取而改变历史顺序
- `NoHistory` 必须返回空列表

### 5.4 `GetEditingState`

#### 目标

返回当前编辑数据库的状态摘要。

#### 必须满足

- `open`、`dirty`、`openMode`、`baselineVersion` 必须自洽，且 `baselineVersion` 必须能跨重开保持一致
- `undoCount` / `redoCount` 必须与 `GetEditLogState()` 的结果一致
- 不得因为查询状态而引入写入

### 5.5 只读约束

`ReadOnly` 模式下，以下行为都必须拒绝：

- 写入
- 历史变更
- 基线重置
- 任何会改变正式数据的副作用

## 6. 语义约束

### 6.1 三种模式的一致性

Memory backend 与 SQLite backend 必须一致遵守：

- `Normal` 允许写入并保留可见历史
- `NoHistory` 允许写入但不保留用户可见历史
- `ReadOnly` 不允许写入

### 6.2 历史可见性

- `GetEditLogState()` 返回的是“可见编辑日志”，不是内部完整日志
- 可见历史的顺序必须稳定
- 历史列表不能由读操作重建出额外数据
- `commitId` 必须是稳定事务标识，`version` 必须是该事务第一次提交成功时的版本

### 6.3 基线收敛

- 基线收敛不能改变正式数据内容
- 基线收敛只能改变历史边界
- 基线收敛后，旧保存点之前的可见历史不得继续对外显示

---

## 7. 错误与诊断

### 7.1 必须包含的信息

错误返回至少应包含：

- 原因
- 所处阶段
- 相关版本或路径
- backend 类型

### 7.2 建议的诊断阶段

- `Open`
- `CreateBackupCopy`
- `ResetHistoryBaseline`
- `ReadHistory`
- `ReadState`

---

## 8. 测试要求

必须使用 GoogleTest 覆盖以下场景。

### 8.1 正常路径

- `CreateBackupCopy` 成功
- `ResetHistoryBaseline` 成功
- `GetEditLogState` 返回正常日志视图
- `GetEditingState` 返回正确状态

### 8.2 边界路径

- `Normal` 模式下历史列表正确
- `NoHistory` 模式下历史列表为空
- `ReadOnly` 模式下写入失败
- `ReadOnly` 模式下基线重置失败

### 8.3 故障路径

- 备份中断
- 目标路径不可写
- 保存基线失败
- 历史裁剪失败

### 8.4 Backend 一致性

同一组测试必须在：

- Memory backend
- SQLite backend

两端都执行，确认语义一致。
