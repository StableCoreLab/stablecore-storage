# 本轮审查

审查范围：`Docs/task.md` 对应的当前待提交代码。

结论：仍然存在 2 个必须修改的问题，都会导致编辑器可见状态和底层持久状态不一致，不能直接提交。

## 1. 编辑器新增列失败后，schema 可能已经回滚失败，但视图已经恢复，状态仍可能不一致

- 严重级别：高
- 位置：
  - `Tools/DatabaseEditor/SCDatabaseSession.cpp:408-418`
  - `Src/Sqlite/SCSqliteAdapter.cpp:3328-3371`

### 问题描述

`SCDatabaseSession::AddColumn()` 在 `RebuildCurrentTableView()` 失败后，会尝试调用 `schema->RemoveColumn(column.name.c_str())` 做补偿回滚。

但当前实现没有处理“补偿回滚也失败”的情况，只是把 `currentTableView_` 恢复成旧对象并返回 `false`。如果 `PersistRemovedColumn()` 因 SQLite 侧错误失败，底层 schema 仍然保留新列，而 UI 已经回到旧视图。

### 影响

- 失败路径仍可能留下“UI 看起来回退了，但 schema 其实已经改过”的半成功状态。
- 后续继续编辑时，用户会基于旧视图操作，和底层 schema 进一步分叉。

### 必改建议

- `AddColumn()` 失败路径必须保证 schema/view 二者同步回滚，不能只恢复视图。
- 如果补偿删除失败，应该保留明确的错误状态，并避免继续把旧视图当成可用状态。
- 更稳妥的方案是把“视图重建可用性检查”前移到 schema 写入之前，避免先写后补偿。

## 2. 新增的 `RemoveColumn()` 不是完整的删除语义，SQLite 侧仍会残留列相关存储

- 严重级别：中高
- 位置：
  - `Include/ISCInterfaces.h:56-68`
  - `Src/Sqlite/SCSqliteAdapter.cpp:3290-3371`

### 问题描述

现在 `ISCSchema` 新增了 `RemoveColumn()`，但 SQLite 实现只删除了 `schema_columns` 元数据并卸载内存缓存列定义，没有处理列对应的持久化 footprint。

具体来说：
- `field_values` 中对应字段的历史数据没有被清理
- `EnsureColumnIndex()` 创建出来的列索引也没有被回收

对于当前的“新增列失败回滚”场景，这个接口只是在局部上够用；
但从接口语义上看，它已经对外暴露成“删除列”，实现却不是完整删除。

### 影响

- 一旦未来把这个接口用于真实删列，SQLite 库里会遗留孤儿数据和孤儿索引。
- 重复增删同名列时，底层存储 footprint 会越来越不干净，和 schema 元数据不一致。

### 必改建议

- 要么把 `RemoveColumn()` 语义收紧成“仅用于回滚新增列”，并避免对外暴露为通用删除能力。
- 要么补齐 SQLite 侧完整删除语义，包括 `field_values` 和相关索引的清理。
- 至少要补一个“删列后相关 footprint 一并清理”的回归测试。

## 建议顺序

1. 先修第 1 个问题，避免编辑器失败路径仍然留下不一致状态。
2. 再修第 2 个问题，把新增的删除接口收口成真实、完整的语义，或者明确降级成内部回滚专用能力。
