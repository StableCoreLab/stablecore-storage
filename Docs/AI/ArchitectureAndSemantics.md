# 架构与语义

## 仓库定位

- 仓库：`stablecore-storage`
- 角色：工程量/算量系统的核心 SQLite 存储基础设施

## 主要职责

- SQLite 项目持久化
- 撤销 / 重做
- 事务语义
- 变更集 / 日志 / 快照 / 回放
- 可恢复编辑
- 升级 / 迁移基础设施

## 核心架构（P0）

```text
Core（语义规则）
    -> Adapter（公共接口桥接）
    -> Backend（SQLite）
```

强制约束：

- 上层不得直接访问 SQLite
- Query / Computed 模块不得绕过公共接口
- Backend 模块不得包含业务语义
- 公共存储语义必须独立于 SQLite 实现细节

## 存储语义不变量（P0）

以下语义为不变量，绝不可违反：

- `CreateFileDatabase()` 可以在返回前自动升级，但数据库构造函数不得修改数据
- 读操作不得写入
- 升级只能通过显式的工厂或数据库升级 API 执行
- 回滚必须完全恢复状态
- 已删除记录绝不可复活
- 撤销 / 重做必须保证身份一致性
- 事务不得留下部分成功状态
- SQLite 实现细节不得泄漏到公共 API

正确性始终优先于性能。

## 核心设计原则

### 显式状态转换

所有状态变更必须通过显式 API 进行：

- BeginEdit
- Commit
- Rollback
- Open
- Upgrade
- Finalize
- Abort

禁止隐式状态变更。

### 单一真实来源

持久化真实来源仅限于：

- Project
- ChangeSet
- Journal
- Snapshot

UI 状态、缓存和会话对象不是真实来源。

### 可恢复性优先

所有写入路径必须支持：

- 回滚
- 恢复
- 中断安全

必须处理：

- 导入中断
- 迁移失败
- 回滚失败
- 访问已删除记录
- SQLite 事务中断
- 日志损坏或不完整

### Backend 隔离

SQLite 是实现细节。

规则：

- 公共 API 必须保持存储语义
- SQL 细节不得泄漏到上层
- Schema 变更必须通过显式迁移逻辑
- 存储行为不得依赖 UI 状态
- SQLite Schema 不得成为公共契约

### 最小意外原则

系统必须保持直观语义：

- 读不写
- 已删除数据不复活
- 只读模式不修改数据
- 失败操作不部分提交
- 打开项目可能在返回前将其升级到当前支持的 Schema
- 关闭项目不会静默提交未完成的编辑

## 工程哲学

优先级顺序：

```text
正确性
    > 可恢复性
        > 存储语义稳定性
            > 可维护性
                > 性能
```

对 AI 的期望行为定位为：

> 一个受约束的工程执行者，
> 而非一个自主重构代理。
