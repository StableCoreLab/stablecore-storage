# 计算列表视图与 SQLite 启动能力说明

本文记录当前仓库中与“计算列表视图”和“SQLite 启动、恢复”相关的已完成能力，以及仍保留的术语边界。

## 计算列表视图

已实现内容：

- 为表提供统一的计算列表视图抽象。
- 同时暴露事实列与计算列，供 UI 和上层调用统一读取。
- 支持按记录读取指定列的当前值。
- 支持在视图层挂载会话级计算列定义。

设计价值：

- 数据表编辑工具不必区分“事实列读取”和“计算列读取”两套通道。
- 计算列的缓存、失效和重建都可以封装在视图层。
- 上层接入时可以把表视图当作更稳定的展示模型。

当前约束：

- 计算列仍以 V1 范围为主，不追求完整表达式语言。
- 复杂业务规则仍建议通过 `ruleId` 注册机制扩展。

## SQLite 启动流程

当前启动链路覆盖：

- 打开 SQLite 数据库文件。
- 初始化基础元数据表。
- 读取表定义与列定义。
- 加载持久化 `Journal`、恢复、Undo/Redo 所需状态。
- 输出启动阶段诊断信息。

设计目标：

- 即使数据库存在旧版本或历史事务，也能给出可预期的启动行为。
- 启动失败时尽量返回清晰错误，而不是静默崩溃。
- 启动成功后，上层能够获得已对齐的数据库对象图。

## 查询索引与引用索引

当前策略：

- 对常见等值查询和关系查询形态预留查询索引支持，但不把它表述为完整的物化优化。
- 引用关系使用 `ReferenceIndex` / `ReverseReferenceIndex` 统一描述，诊断接口只用于观测与重建。
- 用性能冒烟测试验证主路径是否可用，而不是证明完整索引优化已经完成。

现阶段结论：

- 仓库已经具备“最小可研发”的查询与引用索引能力。
- 性能调优仍需在真实业务数据规模下继续迭代。
