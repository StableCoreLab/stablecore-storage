# 代码审查问题 2

审查范围：`Docs/task.md` 对应的当前待提交改动。

结论：存在 3 个需要优先修复的问题，分别影响编辑器一致性、性能基线覆盖和测试可读性。

## 1. 编辑器新增列失败时会留下半成功状态

- 严重级别：高
- 位置：[Tools/DatabaseEditor/SCDatabaseSession.cpp](/d:/code/StableCore/Storage/Tools/DatabaseEditor/SCDatabaseSession.cpp#L336)

问题描述：
- `AddColumn()` 先调用 `schema->AddColumn(column)`，随后才尝试 `RebuildCurrentTableView()`。
- 如果重建视图失败，函数直接返回 `false`，但 Schema 已经被修改，当前会话会进入“底层已变更、视图未同步”的半成功状态。
- 这和同文件里的会话级计算列 `AddSessionComputedColumn()` / `UpdateSessionComputedColumn()` 不一致，后两者在重建失败时有回滚恢复。

影响：
- 编辑器会出现“用户看到失败，但底层已经新增列”的不一致状态。
- 后续如果用户继续操作或提交，会把一个不完整的状态带入数据库。

建议修复：
- 在重建失败时回滚刚刚追加的 Schema 变更，或者先在临时视图/临时 Schema 上验证重建成功，再提交实际修改。

## 2. 性能基线已经不覆盖 SQLite 排序路径

- 严重级别：中高
- 位置：[Tests/PerformanceSmokeTests.cpp](/d:/code/StableCore/Storage/Tests/PerformanceSmokeTests.cpp#L201) [Tests/PerformanceSmokeTests.cpp](/d:/code/StableCore/Storage/Tests/PerformanceSmokeTests.cpp#L278)

问题描述：
- 新的 `SqliteQueryBaseline` 只验证了 SQLite 的查询和索引命中，没有带 `orderBy`。
- 真正验证排序语义的是 `MemoryQuerySortBaseline`，而它跑的是 Memory 后端，不是 SQLite 后端。
- 结果是 SQLite 查询执行器的 `ORDER BY` 生成和排序回归不会被 smoke baseline 发现。

影响：
- SQLite 排序逻辑如果退化或出错，当前性能基线仍可能全部通过。
- `T10` 的“查询、排序、恢复”回归门槛实际上少了一条关键后端链路。

建议修复：
- 保留一个 SQLite 版本的排序基线，或者新增独立的 SQLite `ORDER BY` smoke test。
- 如果担心执行时间，可把排序样本规模调小，但不能把 SQLite 排序路径移出覆盖范围。

## 3. `MeasureMs` 内部使用 fatal 断言会导致失败后继续执行

- 严重级别：中
- 位置：[Tests/PerformanceSmokeTests.cpp](/d:/code/StableCore/Storage/Tests/PerformanceSmokeTests.cpp#L32) [Tests/PerformanceSmokeTests.cpp](/d:/code/StableCore/Storage/Tests/PerformanceSmokeTests.cpp#L183) [Tests/PerformanceSmokeTests.cpp](/d:/code/StableCore/Storage/Tests/PerformanceSmokeTests.cpp#L261) [Tests/PerformanceSmokeTests.cpp](/d:/code/StableCore/Storage/Tests/PerformanceSmokeTests.cpp#L328)

问题描述：
- `MeasureMs()` 接收的 lambda 内部使用了 `ASSERT_EQ`。
- 在 GoogleTest 里，`ASSERT_*` 只会从当前 lambda / 函数返回，不会终止外层测试函数。
- 如果 lambda 内某个断言失败，`MeasureMs()` 仍然会返回耗时，而外层测试还会继续使用可能未初始化或部分初始化的结果对象。

影响：
- 一次失败可能引发一串后续错误，污染真实失败原因。
- 性能基线失败时，日志会更难定位根因。

建议修复：
- 不要在 `MeasureMs()` 传入的 lambda 里使用 fatal `ASSERT_*`。
- 改成先做前置断言，再计时执行纯逻辑；或者让 helper 返回 `bool`，外层用 `ASSERT_TRUE` 收口。
