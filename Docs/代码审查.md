# 代码审查问题

审查范围：`Docs/task.md` 对应的当前待提交代码。

结论：存在 2 个必须修改的问题，均会导致数据状态与可见状态不一致，不能直接提交。

## 1. 导入 finalize 不是原子收口，可能留下“业务数据未提交但恢复状态已清理”的不一致

- 严重级别：高
- 位置：
  - `Src/Batch/SCBatchOperations.cpp:319-330`
  - `Src/Sqlite/SCSqliteAdapter.cpp:2260-2301`

### 问题描述

`FinalizeImportSession()` 的当前顺序是：

1. `ApplyBatchRequests(...)`
2. `database->FinalizeImportSession(commit, &recoveryState)`
3. `database->Commit(edit.Get())`

其中 `SqliteDatabase::FinalizeImportSession()` 直接执行了 `DELETE FROM import_sessions ...`，并没有和外层 `Commit(edit.Get())` 处在同一个事务边界内。

这意味着：
- 如果 `FinalizeImportSession()` 先成功删除了恢复状态，
- 但后面的 `Commit(edit.Get())` 失败，
- 回滚只能撤销当前 edit 的业务数据，不能把已经删除的恢复记录自动恢复回来。

### 影响

- 会出现“业务数据没最终提交，但恢复状态已经消失”的脏状态。
- 进程中断或提交失败后，导入会话的可恢复性会丢失，破坏 `T6/T7` 想要保证的原子性目标。

### 必改建议

- 要么把 `FinalizeImportSession()` 的恢复状态清理纳入同一个 SQLite 事务；
- 要么把恢复状态删除延后到 `Commit(edit.Get())` 真正成功之后，再通过独立但可补偿的收尾步骤处理。
- 需要补一个“`Commit` 失败后恢复状态仍可继续加载”的回归测试。

## 2. 编辑器新增列失败后，schema 已变更但视图可能仍停留在旧状态

- 严重级别：高
- 位置：
  - `Tools/DatabaseEditor/SCDatabaseSession.cpp:398-415`

### 问题描述

`SCDatabaseSession::AddColumn()` 目前是先执行 `schema->AddColumn(column)`，再尝试 `RebuildCurrentTableView(outError)`。

如果 `RebuildCurrentTableView()` 失败，函数会直接返回 `false`，但前面的 schema 变更已经发生，当前实现没有把刚追加的列回滚掉。

也就是说，这条失败路径仍然可能留下：
- 底层 schema 已新增列
- 编辑器当前表视图未同步
- UI 收到刷新信号，但实际状态并未恢复一致

### 影响

- 编辑器层会再次出现“底层已变更、界面没跟上”的半成功状态。
- 后续继续编辑时，用户可能基于旧视图继续操作，进一步放大不一致。

### 必改建议

- 新增列前先做视图重建预检，确保重建可用后再提交 schema 变更；
- 或者在 `RebuildCurrentTableView()` 失败时补一个补偿回滚，把刚加入的列从 schema 中撤销；
- 需要增加一个“视图重建失败后 schema 不应发生最终可见变化”的回归测试。

## 建议顺序

1. 先修导入 finalize 的原子性问题。
2. 再修编辑器新增列的半成功回退问题。
3. 两项都修完后再提交，避免把不一致状态带入后续任务。
